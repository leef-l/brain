# test/ — BrainKernel 测试目录

## 组织原则

**每个版本/阶段使用独立的测试子目录。**

不同版本的代码具有不同的能力边界，测试必须与之匹配。将测试按版本隔离可以：

1. 清晰标识当前可运行的测试范围
2. 避免未实现功能的测试干扰 CI
3. 为每个版本留下可回溯的测试快照

## 目录结构

```
test/
├── 测试目录说明.md          ← 本文件
├── skeleton/               ← v0.1.0-skeleton 骨架测试（当前）
│   ├── agent_test.go       — Agent 包：Kind 常量、LLMAccessMode、Descriptor
│   ├── cli_test.go         — CLI 包：15 个退出码常量、OutputFormat
│   ├── decide_test.go      — 错误决策矩阵：Transient/Permanent/UserFault/Quota/Safety
│   ├── errors_test.go      — 错误模型：New/Wrap/Error/Unwrap/Is/JSON/Fingerprint
│   ├── kernel_test.go      — MemKernel 组装：12 字段注入、内置工具、PlanStore/Artifact 往返
│   ├── llm_test.go         — MockProvider：Complete/Stream/FIFO/Reset、Usage/Budget 字段
│   ├── loop_test.go        — Run 状态机（7 合法 + 4 非法转换）、Budget 5 维耗尽检测
│   ├── observability_test.go — Counter/Histogram/Gauge、Trace span 父子链、Log 级别过滤/环形缓冲
│   ├── persistence_test.go — SHA-256 已知向量、CAS 往返/去重、PlanStore/UsageLedger
│   ├── protocol_test.go    — 方法名常量、SidecarState FSM、Role 前缀、帧大小限制
│   ├── security_test.go    — MemVault CRUD/TTL/并发、审计无泄露、Zone 常量、哈希链完整性
│   └── tool_test.go        — MemRegistry CRUD/并发、EchoTool/RejectTaskTool 往返
│
├── compliance/             ← v2 合规测试（待实现，150 项）
│   └── （v2 核心代码完成后创建）
│
└── enterprise/             ← v3 Atlas Enterprise 测试（远期）
    └── （v3 RPC Transport 完成后创建）
```

## 运行测试

```bash
# 运行骨架测试
go test github.com/leef-l/brain/test/skeleton -v

# 运行全部测试（当有多个目录时）
go test github.com/leef-l/brain/test/... -v
```

## 注意事项

- `testing/` 包（项目根目录下）是**合规测试框架库**（ComplianceRunner、Cassette、FakeSidecar），不是测试目录
- 每个测试子目录是独立的 Go 包，package 名与目录名一致
- 新版本的测试不应破坏旧版本的测试——骨架测试永远应该能通过
