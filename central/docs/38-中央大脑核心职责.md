## 一、定位重申

中央大脑（kind=`central`）在 MACCS 中是 **L3 战略层**，唯一身份：

```
中央大脑 ≠ "主对话 LLM"
中央大脑 = 监督者 + 编排者 + 反馈整合者 + 元认知反思者
```

它**不直接动手干活**。所有"写代码 / 跑命令 / 操作浏览器 / 查市场数据 / 跑测试"都委派给专精大脑。
中央大脑唯一的"动手"权限是**只读**（read_file / list_files / search）—— 用于"理解任务"和"审查产出"。

**这是与 Kimi / Hermes / OpenClaw 的根本区别**：它们是单 agent 自我改进，
我们是 L3 中央大脑 + N 个 L2 专精大脑的**真并行协作**。

### 2.3 反馈接收（运行中持续）

**这是 MACCS 与传统 agent 的关键差别**：中央大脑不是"派完就等结果"，而是**全程接收多源反馈，随时调整任务计划**。

#### 反馈源 1：专精大脑实时进度（反向 RPC）

- 协议：`progress/report` 反向 RPC（专精 sidecar → 中央 host）
- 入口：`Orchestrator.registerReverseHandlers`（`orchestrator.go:1225`）
- 处理：`ProgressHandler.HandleReport`（`progress_rpc.go:47`）
- 持久化：`ProgressStore.SaveProgress`
- 代码引用：每个专精大脑在 turn loop 中调 `host.progress.report({turn, summary, artifacts})`

#### 反馈源 2：EventBus 全局事件总线

- 7 类事件：`brain.delegate.started/completed/failed`、`brain.progress.*`、`brain.feedback.requested`、`workflow.layer.*`、`chaos.experiment.*`
- 中央大脑订阅入口：通过 SSE `/v1/events` 或 WebSocket `/v1/dashboard/ws` 接收
- 关键代码：`Orchestrator.Publish`（`orchestrator.go:1335`）+ `assessActiveLearning`（`orchestrator.go:1740`）

#### 反馈源 3：用户中断信号

- 协议：`POST /v1/runs/{id}/interrupt` 或 chat REPL 的 `Esc`
- 数据结构：`InterruptSignal{Type, Action, Reason}`（`sdk/kernel/interrupt.go`）
- 行为：`Runner` 主循环每轮检查，命中即按 `Action` 执行（pause / replan / abort / amend）
- 中央大脑响应：从 EventBus 收到 `interrupt` 事件 → 重新走 §2.4 调整流程

#### 反馈源 4：学习引擎四层信号（L0-L3）

| 层 | 信号 | 中央大脑用法 |
|----|------|-------------|
| L0 | 5 脑各自的 `BrainLearner.Adapt()` 自适应 | 中央大脑在 ModelRouter 选模型时参考 |
| L1 | 工具序列 `RecordSequence` | 中央大脑下次拆 DAG 时优先选高成功率序列 |
| L2 | 协作模式（哪两个 brain 配对成功率高） | 中央大脑做 brain 选择时加权 |
| L3 | 用户偏好（feedback 衰减） | 中央大脑在生成 prompt / 决策时遵循 |

接入入口：`learning.go:709/718` + `orchestrator.go:1488/1509/1668`

#### 反馈源 5：项目记忆 + 模式库（跨会话）

- 项目记忆：`ProjectMemory.Store` 写入 reflection 教训 + pattern 摘要
- 经验库：`ExperienceStore` 持久化每次 ExecuteProject 结果
- 模式库：`PatternExtractor` 异步抽取 4 类模式（architecture / workflow / tool_usage / brain_selection）
- 用法：下次任务进来时通过 `MemoryRetriever` 检索 top-N 历史经验，注入 reflection.Recommendations

#### 反馈源 6：可观测 + 性能指标

- `HealthManager` → 组件健康（GET /v1/health）
- `PerfCollector` → brain_delegate 耗时分布（GET /v1/metrics/perf）
- `ObservabilityHub` → 每次 delegate 的 TraceSpan（GET /v1/observability）
- 中央大脑用法：发现某专精大脑 P95 异常或健康下降时，**重排任务避开该 brain**（接入 SmartScheduler 重排逻辑）

### 2.4 动态调整（核心能力）

中央大脑接到任何反馈源信号都可触发**重新规划**。当前已实现的调整路径：

| 触发 | 动作 | 代码位置 |
|------|------|---------|
| 子任务失败 + 可重试 | `ExecutionScheduler.MarkRetry` 同任务再派 | `execution_scheduler.go` |
| 资源冲突（同层多任务抢同一文件） | `SmartScheduler.Reschedule` 把冲突任务挤到下一层 | `smart_scheduler.go:Reschedule` |
| 测试不通过 | `ReviewLoop` 判定 → 反馈给 code 大脑改进 | `review_loop.go` |
| 子任务进度异常慢 | 通过 EventBus 发现 → `ActiveLearning.AssessUncertainty` 触发 feedback request | `orchestrator.go:1740` |
| 用户主动中断 + amend | `InterruptSignal{Action: amend}` → 中央大脑重新拆 DAG | `interrupt.go` |
| 7 阶段闭环验收失败 | `ClosedLoopController` 回退到 review 阶段重做 | `closed_loop_controller.go` |
| 元认知反思发现系统性问题 | `MetaCognitiveEngine.Reflect` 写入 ProjectMemory + 反馈 LearningEngine | `meta_cognitive.go:587` |

### 2.5 审查与交付（任务结束时）

| 工作 | 工具 / 组件 |
|------|------------|
| 验收测试 | `AcceptanceTester.RunTests`（artifacts fallback / 自定义 verify_xxx 命令）|
| 交付包生成 | `DeliveryGenerator`（README + CHANGELOG + 产物清单）|

| 模式抽取 | `PatternExtractor.Extract`（异步，5.4） |

## 三、工具白名单（中央大脑能调什么 / 不能调什么）

中央大脑在 chat / run / serve 模式下**统一身份**为 L3 编排者。工具集分三类：

### 3.1 编排工具（central 专属，必须有）

- `central.delegate` — 单步派发到专精大脑
- `central.submit_workflow` — DAG 编排（多专精并行 / 串行）
- `central.brain_manage` — 启动 / 停止 / 列出专精大脑
- `central.start_human_demo` — 触发人类接管录制
- `central.task_complete` — 标记任务结束（必须调用，否则 turn loop 不会自然终止）
- `central.reject_task` — 拒绝无法处理的任务
- `central.review_trade` / `central.daily_review` / `central.data_alert` — 量化场景特化

### 3.2 只读工具（可有，用于"理解任务" + "审查产出"）

- `code.read_file` — 看现有代码
- `code.list_files` — 看项目结构
- `code.search` — grep 关键字
- `code.note` — 中央大脑做笔记（不算"动手"，是元认知）
- `verifier.read_file` — 看产出
- `verifier.check_output` — 轻量校验产出（不跑代码，仅文本/格式）
- `human.request_takeover` — 任何 brain 都可请人接管

### 3.3 禁用工具（绝不直接调，必须委派）

- ❌ `code.write_file` / `edit_file` / `delete_file` → 委派 code 大脑
- ❌ `code.shell_exec` → 委派 code（包括 ls / pwd 等）
- ❌ `verifier.run_tests` → 委派 verifier
- ❌ `verifier.browser_action` → 委派 browser
- ❌ 任何 `quant.*` 写操作（pause / close 等） → 委派 quant
- ❌ 任何 `data.*` replay 类 → 委派 data

> **铁律**：中央大脑的 prompt 中必须包含"DELEGATE FIRST"硬约束。
> 即使任务"看起来简单"，也要委派 —— 因为：
> 1. 写文件后必须有 verifier 验收，跳过等于断闭环
> 2. 多大脑协作是产品差异化，自己干 = 退化为单 agent
> 3. 学习引擎只在派发路径上累积数据，自己干 = 系统永远学不到

## 四、中央大脑的"动嘴 vs 动手"边界

```
┌─────────────────────────────────────────────────┐
│  中央大脑（动嘴）                               │
│  ─────────                                       │
│  ✅ 读代码/文件（理解任务）                     │
│  ✅ 拆分 DAG                                     │
│  ✅ 派发工具调用                                │
│  ✅ 接收反馈，动态调整                          │
│  ✅ 元认知反思 + 写记忆                         │
│  ✅ 验收 / 交付 / 复盘                          │
└──────────────┬──────────────────────────────────┘
               │ delegate / submit_workflow
               ▼
┌─────────────────────────────────────────────────┐
│  专精大脑（动手）                               │
│  ─────────                                       │
│  code:     write/edit/delete/shell                │
│  verifier: run_tests/check_output                 │
│  browser:  浏览器交互（点击/截图/导航）         │
│  data:     行情订阅 / 历史回填                  │
│  quant:    交易决策 / 仓位控制                  │
│  fault:    混沌注入 / 故障演练                  │
│  desktop:  桌面操作（实验中）                   │
└─────────────────────────────────────────────────┘
```
