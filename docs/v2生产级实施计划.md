## 实施阶段

### 阶段 A：真实工具实现（code brain 4 个核心工具）

**新建文件：** `tool/builtin_read_file.go` / `builtin_write_file.go` / `builtin_search.go` / `builtin_shell_exec.go`

#### A1. `code.read_file` — 读文件

```
输入: {"path": "/src/main.go", "offset": 0, "limit": 200}
输出: {"content": "package main\n...", "lines": 42, "truncated": false}
```

- Risk: `RiskSafe`（纯读操作）
- 安全：禁止读取 `~/.ssh/`、`/etc/shadow` 等敏感路径
- 大文件保护：默认限制 2000 行，超出则截断 + 标记 `truncated: true`
- 二进制文件检测：前 512 字节含 NUL → 返回 `"binary file, N bytes"`

#### A2. `code.write_file` — 写文件

```
输入: {"path": "/src/utils.go", "content": "package main\n..."}
输出: {"bytes_written": 128, "path": "/src/utils.go"}
```

- Risk: `RiskMedium`（修改文件系统）
- 安全：禁止写入 `/etc/`、`~/.ssh/` 等系统目录
- 自动创建不存在的父目录
- 写入前备份原文件到 `.brain-backup/` (可选)

#### A3. `code.search` — 搜索代码

```
输入: {"pattern": "func main", "path": ".", "glob": "*.go", "max_results": 50}
输出: {"matches": [{"file": "cmd/main.go", "line": 18, "text": "func main() {"}], "total": 3}
```

- Risk: `RiskSafe`
- 实现：递归遍历目录 + `strings.Contains` 或 `regexp.MatchString`
- 忽略 `.git/`、`vendor/`、`node_modules/` 等目录
- 限制最大结果数防止输出爆炸

#### A4. `code.shell_exec` — 执行命令

```
输入: {"command": "go test ./...", "timeout_seconds": 60, "working_dir": "."}
输出: {"stdout": "ok ...", "stderr": "", "exit_code": 0, "timed_out": false}
```

- Risk: `RiskHigh`（执行任意命令）
- 安全：
  - 默认超时 60 秒，最大 300 秒
  - 输出截断：stdout/stderr 各最大 100KB
  - 禁止危险命令前缀白名单检查（可配置）
- 实现：`exec.CommandContext` + 超时 + 输出捕获

**测试：** 每个工具 3-5 个测试（正常/异常/边界/安全拦截）

**验证：**
```bash
go test ./tool/... -v -run TestReadFile
go test ./tool/... -v -run TestWriteFile
go test ./tool/... -v -run TestSearch
go test ./tool/... -v -run TestShellExec
```

func (h *codeHandler) HandleMethod(ctx context.Context, method string, params json.RawMessage) (interface{}, error) {

## 风险与约束

1. **零外部依赖**：所有工具必须只用标准库（`os`, `os/exec`, `regexp`, `path/filepath`）
2. **安全边界**：文件读写必须有路径校验，shell_exec 必须有超时
3. **向后兼容**：`--provider mock` 和无 API Key 场景必须继续工作
4. **测试覆盖**：每个阶段完成后必须 `go test ./...` 全绿
5. **AnthropicProvider 已测试**：16 个 cassette 测试确认其正确性，接入时无需修改 Provider 本身
