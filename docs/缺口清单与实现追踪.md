### 🔴 P0-1: Quant L0 `RecordOutcome` 死代码

**原文位置**：`brains/quant/brain_learner.go`（全文件，第 1-888 行）
**对应文档**：`sdk/docs/32-v3-Brain架构.md` §7.6 "L0 专精大脑学习能力设计"、第 518 行 Quant 行

**现状**：
- `brains/quant/brain_learner.go` 中 `RecordOutcome`/`Adapt`/`ExportMetrics` 三个方法均存在且有真实领域逻辑
- 但 `quant/sidecar/handler.go` 中：
  - ❌ 无 `brain/learn` RPC handler
  - ❌ `handleExecute` 内部从不调用 `RecordOutcome`
- 结果：`QuantBrainLearner` 是**死代码**——编译通过，但永远不会被执行

**实现方案**：
1. 在 `quant/sidecar/handler.go` 中添加 `brain/learn` 方法处理
2. 在 `handleExecute` 完成后调用 `learner.RecordOutcome`
3. 或通过 Orchestrator 的 `sendBrainLearn` 异步通知路径走通

**修复详情**：
- `quant/sidecar/handler.go` 新增 `case "brain/learn"` 和 `case "brain/metrics"`
- `handleLearn` 解析 `kernel.TaskOutcome`，调用 `h.learner.RecordOutcome`
- `handleExecute` 每个 instruction 执行后调用 `h.learner.RecordOutcome`
- `metricsReportLoop` 每 5 分钟上报 `brain/metrics`

### 🔴 P0-6: Streaming Edge 跨 brain 无真正流式效果 ✅（方案 B）

**原文位置**：`sdk/kernel/workflow.go:349-582`（`executeStreamingProducer`）
**对应文档**：`docs/并行拆分任务工作流.md` §3.2 "streaming edge"、`sdk/docs/35-Flow-Edge存储与注册发现设计.md` §3

**现状**：
- `WorkflowEngine` 代码中有完整的 streaming 分支（`executeStreamingProducer`/`executeNormalNode`）
- 但 `Orchestrator.ExecuteWorkflow` 创建 engine 后**从未调用 `SetStreamingExecutor`**
- `Delegate` 使用同步 JSON-RPC，sidecar 执行完毕后一次性返回完整结果
- `PipeRegistry` 是进程内 Go channel，无任何跨进程能力
- **实际效果**：streaming 边 = materialized 边 + 一层多余的 goroutine/channel 中转

**实现方案**（两种选择）：

**方案 A：真正实现跨 brain 流式**（工作量大）
1. 定义跨进程 Streaming 协议（shared memory / Unix socket / 双向 RPC 流）
2. 在 sidecar 中实现流式输出（`brain/execute` 支持流式返回）
3. `Orchestrator` 注入 `StreamingNodeExecutor`，将 sidecar 流式输出实时写入 `PipeRegistry`
4. Consumer brain 通过 `PipeRegistry` 实时读取

**方案 B：承认当前限制，退化为 batch**（工作量小）✅ 已执行
1. ~~简化 `executeStreamingProducer`，去掉多余的 pipe/channel 复杂度~~ — 保留现有实现，不破坏同进程并发能力
2. 更新文档，明确说明当前 streaming edge 是同进程内的 goroutine 并发，跨 brain 时退化为 batch
3. 保留接口，未来再实现真正的跨进程流式

**修复详情**：
- `docs/并行拆分任务工作流.md` §3.2 添加当前限制说明
- `sdk/kernel/workflow.go:349` `executeStreamingProducer` 添加详细注释
