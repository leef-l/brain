# 并行拆分任务工作流（Workflow DAG）

> 适用版本：Brain ≥ v3.2

## 概述

Brain 的 **WorkflowEngine** 支持将复杂任务拆分为有向无环图（DAG）并行执行。

- **节点（Node）**：一次 Brain 任务执行，绑定到具体的 specialist brain（如 `code`、`browser`、`data`）
- **边（Edge）**：节点间的数据传递关系
  - `materialized`：等前驱节点完全结束后，将完整输出传给后继节点
  - `streaming`：前驱节点逐帧输出，后继节点实时消费，两者并行执行
- **并行规则**：同层节点（无相互依赖）自动并行执行

## 三种启动模式的使用方式

### 1. Chat 模式（自动 + 手动）

#### 自动触发
当用户的请求可拆分为多个独立子任务时，Central Brain 会自动调用 `central.submit_workflow` 工具提交 DAG，而不是多次串行 `central.delegate`。

示例对话：
```
用户：帮我写单元测试，同时打开文档页面看看效果
AI：→ 自动构建 Workflow DAG
    Node A (code): "为当前模块编写单元测试"
    Node B (browser): "打开文档页面并截图"
    → 并行执行 → 汇总结果
```

#### 手动触发（Slash 命令）
```bash
/workflow dag.json              # 从文件加载
/workflow '{"nodes":[...]}'    # 内联 JSON
/workflow help                 # 查看帮助
```

### 2. Run 模式（CLI 单次执行）

```bash
# 执行一个 Workflow JSON 文件
brain run --workflow my_dag.json

# 可选：指定工作目录、权限模式
brain run --workflow my_dag.json --workdir ./project --mode auto
```

输出：
- 默认模式：简洁节点状态表
- `--json` 模式：完整 `WorkflowResult` JSON

### 3. Serve 模式（HTTP API）

```bash
# 提交 Workflow
POST /v1/workflows
Content-Type: application/json

{
  "workflow": {
    "id": "wf-001",
    "nodes": [
      {"id": "A", "brain_id": "code", "prompt": "write tests"},
      {"id": "B", "brain_id": "browser", "prompt": "open example.com"}
    ],
    "edges": []
  },
  "mode": "auto",
  "workdir": "/project"
}
```

响应：
```json
{"workflow_id": "wf-xxx", "status": "running"}
```

其他端点：
```bash
GET    /v1/workflows              # 列出所有 workflow
GET    /v1/workflows/:id          # 查询状态
DELETE /v1/workflows/:id          # 取消
GET    /v1/workflows/:id/events   # SSE 实时事件流
```

SSE 事件类型：
- `workflow.node.started` — 节点开始
- `workflow.node.completed` — 节点完成
- `workflow.node.failed` — 节点失败
- `workflow.completed` / `workflow.failed` — 整体完成

## Workflow JSON 格式

```json
{
  "id": "wf-example",
  "nodes": [
    {
      "id": "A",
      "brain_id": "code",
      "prompt": "Implement a ping handler",
      "depends_on": [],
      "required_caps": [],
      "preferred_caps": [],
      "task_type": "coding"
    },
    {
      "id": "B",
      "brain_id": "browser",
      "prompt": "Open http://localhost:8080/ping and verify 200 OK",
      "depends_on": ["A"]
    },
    {
      "id": "C",
      "brain_id": "verifier",
      "prompt": "Run go test ./...",
      "depends_on": ["A"]
    }
  ],
  "edges": [
    {"from": "A", "to": "B", "mode": "materialized"},
    {"from": "A", "to": "C", "mode": "materialized"}
  ]
}
```

### 字段说明

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `id` | string | ✅ | Workflow 唯一标识 |
| `nodes` | array | ✅ | 节点列表 |
| `nodes[].id` | string | ✅ | 节点 ID |
| `nodes[].brain_id` | string | ✅ | 目标 brain：`code`、`browser`、`verifier`、`data`、`quant`、`fault` |
| `nodes[].prompt` | string | ✅ | 任务指令 |
| `nodes[].depends_on` | string[] | ❌ | 依赖的节点 ID |
| `nodes[].required_caps` | string[] | ❌ | 必需能力标签（空时通过 capMatcher 自动选择） |
| `nodes[].preferred_caps` | string[] | ❌ | 偏好能力标签 |
| `nodes[].task_type` | string | ❌ | 任务类型，用于学习引擎记录 |
| `edges` | array | ❌ | 边列表 |
| `edges[].from` | string | ✅ | 源节点 ID |
| `edges[].to` | string | ✅ | 目标节点 ID |
| `edges[].mode` | string | ✅ | `materialized` 或 `streaming` |

## 执行示例

### 示例 1：并行数据采集 + 分析

```json
{
  "id": "data-analysis",
  "nodes": [
    {"id": "fetch", "brain_id": "data", "prompt": "Get BTC/USDT snapshot"},
    {"id": "analyze", "brain_id": "quant", "prompt": "Analyze risk based on the market data", "depends_on": ["fetch"]}
  ],
  "edges": [
    {"from": "fetch", "to": "analyze", "mode": "materialized"}
  ]
}
```

### 示例 2：菱形 DAG（并行编码 + 验证）

```json
{
  "id": "dev-verify",
  "nodes": [
    {"id": "impl", "brain_id": "code", "prompt": "Implement feature X"},
    {"id": "test", "brain_id": "code", "prompt": "Write tests for feature X", "depends_on": ["impl"]},
    {"id": "verify", "brain_id": "verifier", "prompt": "Run tests and check coverage", "depends_on": ["impl"]},
    {"id": "report", "brain_id": "code", "prompt": "Summarize test results", "depends_on": ["test", "verify"]}
  ],
  "edges": [
    {"from": "impl", "to": "test", "mode": "materialized"},
    {"from": "impl", "to": "verify", "mode": "materialized"},
    {"from": "test", "to": "report", "mode": "materialized"},
    {"from": "verify", "to": "report", "mode": "materialized"}
  ]
}
```

执行顺序：
1. Layer 1: `impl` 执行
2. Layer 2: `test` + `verify` **并行**执行
3. Layer 3: `report` 等 test 和 verify 都完成后执行

### 示例 3：Streaming Edge（实时传递）

```json
{
  "id": "stream-demo",
  "nodes": [
    {"id": "producer", "brain_id": "code", "prompt": "Generate logs continuously"},
    {"id": "consumer", "brain_id": "data", "prompt": "Parse and aggregate log data", "depends_on": ["producer"]}
  ],
  "edges": [
    {"from": "producer", "to": "consumer", "mode": "streaming"}
  ]
}
```

`consumer` 在 `producer` 还没完成时就开始接收数据，两者并行运行。

> ✅ **跨进程流式（v3+）**：Streaming Edge 现已支持跨 brain 流式传输。
> - 同进程内：通过 Go channel（PipeRegistry）实现 goroutine 级并发。
> - 跨 brain：前驱节点的 `node.PipeID` 被设置为出边 edge ID（如 `"producer->consumer"`），
>   Orchestrator 在 `brain/execute` payload 中携带 `pipe_id`。sidecar（ThinBrain）收到后
>   注入 `StreamingToolObserver`，在每个 tool 执行成功后通过反向 RPC `brain/stream/write`
>   将输出实时写入 host 的 `PipeRegistry`。consumer 节点同时从共享 pipe 读取，实现
>   跨进程的流式中转。
> - LLM 流式：`kernelLLMProvider.Stream` 已接入 `llm.stream` 反向 RPC，host 侧调用真正的
>   `provider.Stream` 并聚合后返回（伪流式）。未来可进一步优化为逐 token 通知。

## 注意事项

1. **brain_id 为空**：如果 `brain_id` 为空且有 `required_caps`/`preferred_caps`，Orchestrator 会通过 `CapabilityMatcher` 自动选择最佳 brain
2. **环路检测**：WorkflowEngine 会自动检测 DAG 中的环路，发现时返回错误 `ErrWorkflowCycle`
3. **超时控制**：每个节点的执行继承外层 context 的 deadline。serve 模式下可在请求中设置超时
4. **输出截断**：单个节点输出超过 100KB 时会被截断，截断信息会附在输出末尾
5. **持久化**：serve 模式的 workflow 状态保存在内存中（`sync.Map`），服务重启后丢失。需要持久化请通过 SSE 事件流客户端自行保存

## 与 BatchPlanner 的区别

| 特性 | WorkflowEngine DAG | BatchPlanner |
|---|---|---|
| 调度粒度 | 跨脑任务（节点级） | 单 Turn 内的 tool_call 级 |
| 依赖关系 | 显式 `depends_on` + `edges` | 通过 `ToolConcurrencySpec` 冲突推导 |
| 数据传递 | 节点间通过 materialized/streaming edge | 无（同一 turn 内共享上下文） |
| 使用场景 | 多步骤、跨 specialist 的复杂任务 | 单 turn 内多个独立 tool_call 的并行 |
| 接入点 | chat（自动/手动）、run（--workflow）、serve（/v1/workflows） | loop.Runner 内部 |
