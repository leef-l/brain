## 1. 背景与动机

钱学森《工程控制论》提出复杂工程系统的五大控制论主线：

1. **系统建模** — 用状态空间描述系统动态行为
2. **反馈伺服** — 通过偏差检测与修正维持系统稳定
3. **多变量耦合** — 分析子系统间的相互作用与耦合效应
4. **性能指标驱动** — 以明确的性能边界指导控制决策
5. **长期可靠** — 自稳定机制保障系统在扰动下持续运行

Brain 系统作为多 agent 分布式架构（Central + Code + Browser + Data + Quant + Verifier + Fault），天然具备"多输入-多输出-多扰动"的控制论特征。本总纲将五大主线映射到 Brain SDK 的具体代码实现，形成可观测、可控制、可自稳的闭环系统。

## 3. 组件详设

### 3.1 SystemState / StateObserver（主线1：系统建模）

**文件**: `sdk/kernel/system_state.go`

| 类型 | 职责 |
|------|------|
| `SystemState` | 全局状态向量：时间戳、各 brain 状态、全局资源、输入/输出向量 |
| `BrainUnitState` | 单个 brain 的局部状态：健康度、负载、队列深度、成功率、延迟、错误率 |
| `StateObserver` | 接口：定期从基础设施聚合状态 |
| `MemStateObserver` | 实现：30s 周期，从 BrainPool.Status()、LearningEngine.Profiles()、LeaseManager 聚合 |

**状态向量定义**:
- **状态 (State)**: `Brains[kind]` 健康度/负载/队列深度、`LeaseOccupancy`、`Concurrency`、`QueueDepth`
- **输出 (Output)**: `GlobalSuccessRate`、`GlobalAvgLatencyMs`、`GlobalErrorRate`、`Throughput`
- **输入/扰动 (Input/Disturbance)**: `RequestRate`、`TaskComplexity`、`NoiseLevel`

**接入点**:
```go
// cmd/brain/cmd_serve.go
stateObserver := kernel.NewMemStateObserver(pool, leaseManager, learner)
go stateObserver.Start(serveCtx)
```

---

### 3.2 PerformanceSpec（主线4：性能指标驱动）

**文件**: `sdk/kernel/performance_spec.go`

`PerformanceSpec` 定义系统的期望性能边界。所有阈值均可按 brain kind 覆盖。

**默认阈值**:
| 指标 | 全局默认值 | 说明 |
|------|-----------|------|
| TargetSuccessRate | 0.90 | 目标成功率 |
| MaxLatencyMs | 5000 | 最大平均延迟(ms) |
| MaxErrorRate | 0.10 | 最大错误率 |
| MaxLoad | 0.80 | 最大负载 |
| MinThroughput | 0.1 | 最小吞吐(任务/秒) |
| MaxQueueDepth | 100 | 最大队列深度 |
| MaxLeaseOccupancy | 0.85 | 最大租约占用率 |
| Cooldown | 30s | 同一 brain 控制动作间隔 |
| Consecutive | 2 | 连续越界次数（防抖） |

**BrainOverrides** 示例:
```go
spec.BrainOverrides["quant"] = kernel.BrainPerformanceSpec{
    MaxLatencyMs: 2000,  // 量化 brain 对延迟更敏感
    MaxLoad:      0.70,
}
```

### 3.3 FeedbackController / ControlLawSynthesizer（主线2：反馈伺服）

**文件**: `sdk/kernel/feedback_controller.go`、`sdk/kernel/control_law.go`

**控制循环**:
1. `StateObserver.Snapshot()` → 获取当前状态
2. `PerformanceSpec.Check(state)` → 计算 PerformanceGap 列表
3. `ControlLawSynthesizer.Synthesize(gap, state, integralErr, lastErr)` → 生成 ControlAction
4. 执行/记录动作 → 状态变化 → 下一轮循环

**PI 控制律**:
```
control = Kp * error + Ki * integral_error
```
- `Kp = 2.0`（比例增益，快速响应）
- `Ki = 0.5`（积分增益，消除稳态误差）
- 含 anti-windup 简化版：积分项不允许为负

**动作类型**:
| 动作 | 含义 | 触发条件 |
|------|------|----------|
| `throttle` | 限流 | 延迟/负载过高 |
| `boost` | 增压 | 成功率远高于目标，可分配更多任务 |
| `restart` | 重启 sidecar | 成功率/错误率突破重启阈值 |
| `reroute` | 重路由任务 | 成功率/错误率 degraded |
| `alert` | 仅告警 | 全局指标异常 |
| `noop` | 无动作 | 系统在期望范围内 |

**FeedbackAwareBrainPool**（扩展点）:
在 `BrainPool` 之上叠加反馈控制，`GetBrain` 时检查是否被限流，自动拒绝或重路由。

### 3.4 CouplingMatrix（主线3：多变量耦合）

**文件**: `sdk/kernel/coupling_matrix.go`

`CouplingMatrix` 量化 brain 间的耦合系数，供 FeedbackController 预判副作用。

**默认耦合系数**（先验知识）:
| Source → Target | 系数 | 说明 |
|----------------|------|------|
| central → code | 0.40 | 协调者任务压力向 code 传递 |
| central → browser | 0.30 | 协调者任务压力向 browser 传递 |
| code → central | 0.35 | code 失败导致 central 重试（正反馈环） |
| quant → data | 0.50 | 量化高负载拉取大量数据 |
| data → quant | 0.30 | 数据响应慢阻塞量化 |
| fault → * | -0.10~-0.15 | 故障注入降低各 brain 可用性 |

**API**:
- `EstimateImpact(source, delta)` — 估算 source 变化 delta 对所有 brain 的级联影响
- `Propagate(actions)` — 传播控制动作的副作用
- `Update(state, actions)` — 在线学习调整系数（简化梯度下降，基于 Load 变化代理实际影响）

---

### 3.5 SelfStabilizer（主线5：长期可靠）

**文件**: `sdk/kernel/self_stabilizer.go`

`MemSelfStabilizer` 基于滑动窗口历史检测四种不稳定模式：

| 检测类型 | 算法 | 恢复动作 |
|---------|------|----------|
| **oscillation** | 负载/错误率反复方向变化 ≥ 阈值 | alert（人工介入调参） |
| **deadlock** | 队列持续增长 + 吞吐量 ≈ 0 | alert（建议 flush 队列） |
| **feedback_loop** | CouplingMatrix 检测双向正耦合 + 双方高负载 | throttle（限流破环） |
| **resource_leak** | 租约占用率持续上升但任务数不增 | alert（检查 lease 释放） |

## 4. 服务接入

所有控制论组件在 `cmd/brain/cmd_serve.go` 的 `runServe` 中初始化：

```go
// Phase A: 状态观测器
stateObserver := kernel.NewMemStateObserver(pool, leaseManager, learner)
mgr.observer = stateObserver
go stateObserver.Start(serveCtx)

// Phase B: 反馈控制器
feedbackCtrl := kernel.NewMemFeedbackController(kernel.DefaultPerformanceSpec(), stateObserver)
mgr.controller = feedbackCtrl
go feedbackCtrl.Start(serveCtx)

// Phase C: 自稳定器
stabilizer := kernel.NewMemSelfStabilizer(stateObserver, kernel.DefaultCouplingMatrix())
mgr.stabilizer = stabilizer
go stabilizer.Start(serveCtx)
```

**设计原则**：所有组件均为指针/接口类型，字段在 `runManager` 中声明为可选（注释"可为 nil"）。当用户不需要控制论框架时，不创建这些对象即可，原有行为完全不受影响。

## 7. 设计约束

1. **stdlib-only**: 控制论框架不使用任何第三方依赖，仅使用 Go 标准库
2. **零第三方 CLI 依赖**: 与项目整体约束一致，flag 解析使用标准库 `flag`
3. **goroutine-safe**: 所有实现均使用 `sync.RWMutex`，支持并发快照和评估
4. **低侵入**: 控制论组件作为"外挂"层存在，nil 时不影响原有行为
5. **可观测**: 所有控制动作和稳定性问题通过 `fmt.Fprintf(os.Stderr, ...)` 输出，后续可接入结构化日志
