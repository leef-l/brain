## 二、与 Brain SDK v3 的关系

量化系统的三个大脑是 Brain SDK v3 的**消费者**，不修改 SDK 任何代码：

```
Brain SDK v3（不动）                量化系统（新增）
─────────────────                 ──────────────────
protocol/  — stdio JSON-RPC       ← Data/Quant/Central 遵守
kernel/    — Orchestrator          ← 调度三个大脑
sidecar/   — BrainHandler 接口     ← 三个大脑都实现它
agent/     — Kind 注册             ← 新增 "data", "quant" Kind
loop/      — Agent Loop 引擎       ← Central Brain 复用
tool/      — Tool Registry         ← 注册新工具（data.*, quant.*）
llm/       — LLM Provider          ← Central Brain 审查时调用
```

三个大脑与 Code Brain、Browser Brain 等同，是按 v3 Manifest-driven、Sidecar 架构接入的**专精大脑**。

## 三、设计原则

| 原则 | 说明 |
|------|------|
| **数据源插件化** | 所有数据源实现统一 `DataProvider` 接口，配置驱动热插拔 |
| **数据零拷贝** | 高频数据走共享内存 ring buffer，不经过 Kernel 中转 |
| **关键路径无 LLM** | tick→策略→下单全链路不调 LLM，延迟 < 5ms |
| **多账号隔离** | 每个账号独立风控、独立仓位、独立损益核算 |
| **多账号聚合** | 跨账号全局风控 + 中央大脑统一视图 |
| **故障隔离** | 三个大脑独立进程，任一崩溃不影响其他 |
| **双通道通信** | 快路径（共享内存）+ 慢路径（JSON-RPC） |
| **数据质量守门** | 所有数据经过校验层，异常数据不进入策略 |
| **全链路可追溯** | 每笔交易决策完整记录，支持回放审计 |
| **品种标准化** | 统一品种 ID，屏蔽各交易所命名差异 |

## 四、三脑分工

| 大脑 | Kind | 职责 | 关键路径 |
|------|------|------|---------|
| **Data Brain** | `data` | 数据采集、清洗、校验、特征计算、实时分发 | 是（快路径） |
| **Quant Brain** | `quant` | 策略计算、信号聚合、风控、多账号执行、审计 | 是（快路径） |
| **Central Brain** | `central` | LLM 审查、每日复盘、异常处理、全局监控 | 否（慢路径） |

### 为什么是三脑而不是四脑

考虑过独立 Risk Brain，但当前规模不需要。**GlobalRiskGuard 在 Quant Brain 内部模块化设计**，未来如果风控复杂度增长，可以零修改拉出来变成独立 sidecar。

## 五、双通道通信

### 5.1 快路径 vs 慢路径

```
┌──────────────────────────────────────────────────────────────────┐
│                       通信拓扑（单机部署）                         │
│                                                                  │
│                    Central Brain                                 │
│                   ┌────────────┐                                 │
│            JSON-RPC│  Kernel   │JSON-RPC                         │
│           (stdio)  │           │(stdio)                          │
│            ┌──────┘            └──────┐                          │
│            │                          │                          │
│     Data Brain                  Quant Brain                     │
│     ┌──────────┐                ┌──────────┐                    │
│     │          ├───────────────►│          │                    │
│     │          │  /dev/shm      │          │                    │
│     │          │  Ring Buffer   │          │                    │
│     └──────────┘  (< 0.1ms)    └──────────┘                    │
│                                                                  │
│  快路径: mmap ring buffer     慢路径: JSON-RPC via Kernel        │
│  延迟 < 0.1ms                 延迟 1-5ms                         │
│  无序列化                      完整审计                           │
│  99% 流量（tick）              1% 流量（控制信号）                 │
└──────────────────────────────────────────────────────────────────┘
```

**单机优势**: 共享内存 = 零网络开销的最优解。未来如需分布式，ring buffer 接口预留了可替换传输层。

### 5.2 通信场景矩阵

| 场景 | 发起方 | 接收方 | 通道 | 入口 | 说明 | 延迟 |
|------|--------|--------|------|------|------|------|
| 实时行情/特征推送 | Data | Quant | 共享内存 | Ring Buffer 直读 | 高频快路径，按 `WriteSeq` 增量消费 | < 0.1ms |
| 历史 K 线查询 | Quant | Data | JSON-RPC | `specialist.call_tool` → `data.get_candles` | 查询型工具，不进入 Agent Loop | < 50ms |
| 当前快照/特征查询 | Quant / Central | Data | JSON-RPC | `specialist.call_tool` → `data.get_snapshot` / `data.get_feature_vector` | 幂等读取 | < 50ms |
| 向量相似度查询 | Quant | Data | JSON-RPC | `specialist.call_tool` → `data.get_similar_patterns` | 走 ANN 检索 | < 100ms |
| LLM 审查请求 | Quant | Central | JSON-RPC | `specialist.call_tool` → `central.review_trade` | 同步返回审查结果 | < 10s |
| 数据质量告警 | Data | Central | JSON-RPC | `specialist.call_tool` → `central.data_alert` | 告警上报 | < 1s |
| 账号执行器故障 | Quant | Central | JSON-RPC | `specialist.call_tool` → `central.account_error` | 账号级异常上报 | < 1s |
| 宏观事件通知 | Data | Central | JSON-RPC | `specialist.call_tool` → `central.macro_event` | 事件型通知 | < 1s |
| 全局视图/审计查询 | Central | Quant | JSON-RPC | `specialist.call_tool` → `quant.global_portfolio` / `quant.trace_query` | 监控与复盘查询 | < 200ms |
| 全局/单品种暂停 | Central | Quant | JSON-RPC | `specialist.call_tool` → `quant.pause_trading` / `quant.pause_instrument` | 确定性控制命令 | < 100ms |
| 强制平仓 | Central | Quant | JSON-RPC | `specialist.call_tool` → `quant.force_close` | 立即执行的控制命令 | < 200ms |
| Provider 健康检查 | Central | Data | JSON-RPC | `specialist.call_tool` → `data.provider_health` | 轻量查询 | < 1s |
| 每日复盘数据收集 | Central | Quant / Data | JSON-RPC | `subtask.delegate` → `brain/execute {instruction: "collect_review_input"}` | 长任务，允许内部多步处理 | < 30s |
| 回测启动/停止 | Central | Data | JSON-RPC | `subtask.delegate` → `brain/execute {instruction: "replay_start" \| "replay_stop"}` | 状态型任务 | < 1s |
| 宏观数据预警 | Data | Quant | 共享内存 | Ring Buffer `ContextFlags` 位 | 低频上下文直接下沉到快路径 | < 0.1ms |

### 5.3 跨脑通信协议（关键约束）

Brain SDK 有两条跨脑调用路径，按“工具面”和“编排面”分工：

```
路径 A: specialist.call_tool（工具面）
  限制: 需要 SpecialistToolCallAuthorizer 授权
  方向: Quant / Data / Central → 任意 brain 的公开工具
  流程: 发起方 sidecar → Kernel → SpecialistToolCallAuthorizer → 目标 sidecar → tools/call
  用途: 查询、短命令、告警、同步审查
  要求: 输入输出 schema 稳定；幂等；可重试

路径 B: subtask.delegate（编排面）
  限制: 仅 Central Brain 可调用（SDK 硬编码鉴权）
  方向: Central → Data / Central → Quant
  流程: Central Brain → Kernel → 目标 sidecar → HandleMethod("brain/execute", ...)
  用途: 长任务、状态任务、需要 sidecar 内部多步处理的任务
  要求: instruction 必须文档化、可恢复、可重复执行
```

**授权策略配置**（需要在 Kernel 层注册，不修改 SDK 代码）:

```go
orch.SetSpecialistToolCallAuthorizer(
    kernel.NewStaticSpecialistToolCallAuthorizer([]kernel.SpecialistToolCallRule{
        {
            Caller: agent.Kind("quant"),
            Target: agent.Kind("data"),
            ToolPrefixes: []string{
                "data.get_",
                "data.validation_",
            },
        },
        {
            Caller:       agent.Kind("quant"),
            Target:       agent.KindCentral,
            ToolPrefixes: []string{"central.review_trade", "central.account_error"},
        },
        {
            Caller:       agent.Kind("data"),
            Target:       agent.KindCentral,
            ToolPrefixes: []string{"central.data_alert", "central.macro_event"},
        },
        {
            Caller:       agent.KindCentral,
            Target:       agent.Kind("quant"),
            ToolPrefixes: []string{"quant."},
        },
        {
            Caller:       agent.KindCentral,
            Target:       agent.Kind("data"),
            ToolPrefixes: []string{"data."},
        },
    }),
)
```

**设计决策**:
- Quant Brain 调用 Central Brain 的 LLM 审查，必须走 `specialist.call_tool("central.review_trade")`，绝不走 `subtask.delegate`
- `tools/call` 承担稳定的公开 API；`brain/execute` 只承担长任务与内部维护指令
- Central Brain 既可以做 `specialist.call_tool` 监控/控制，也可以对长任务发起 `subtask.delegate`

### 5.4 HandleMethod 路由总览

统一约定：
- `tools/call` 暴露稳定的跨脑公开 API
- `brain/execute` 只承载长任务、恢复任务、关闭任务等内部维护指令
- 同一能力只暴露一个主入口，不同时提供“同名 tool + 同名 instruction”两套公共协议

三个大脑的 `HandleMethod` 路由建议如下：

```go
// Data Brain HandleMethod 路由
func (b *DataBrain) HandleMethod(ctx context.Context, method string, params json.RawMessage) (interface{}, error) {
    switch method {
    case "tools/call":
        return b.handleToolCall(ctx, params)
    case "brain/execute":
        var req sidecar.ExecuteRequest
        json.Unmarshal(params, &req)
        switch req.Instruction {
        case "collect_review_input":
            return b.collectReviewInput(ctx, req.Context)
        case "replay_start":
            return b.replayStart(ctx, req.Context)
        case "replay_stop":
            return b.replayStop(ctx)
        case "shutdown_prepare":
            return b.shutdownPrepare(ctx)
        }
    }
    return nil, sidecar.ErrMethodNotFound
}

// Quant Brain HandleMethod 路由
func (q *QuantBrain) HandleMethod(ctx context.Context, method string, params json.RawMessage) (interface{}, error) {
    switch method {
    case "tools/call":
        return q.handleToolCall(ctx, params)
    case "brain/execute":
        var req sidecar.ExecuteRequest
        json.Unmarshal(params, &req)
        switch req.Instruction {
        case "collect_review_input":
            return q.collectReviewInput(ctx, req.Context)
        case "reconcile_state":
            return q.reconcileState(ctx, req.Context)
        case "shutdown_prepare":
            return q.shutdownPrepare(ctx)
        }
    }
    return nil, sidecar.ErrMethodNotFound
}

// Central Brain HandleMethod 路由
func (c *CentralBrain) HandleMethod(ctx context.Context, method string, params json.RawMessage) (interface{}, error) {
    switch method {
    case "tools/call":
        return c.handleToolCall(ctx, params) // central.review_trade / central.data_alert / central.account_error / ...
    case "brain/execute":
        var req sidecar.ExecuteRequest
        json.Unmarshal(params, &req)
        switch req.Instruction {
        case "daily_review_run":
            return c.dailyReview(ctx, req.Context)
        case "update_config":
            return c.updateConfig(ctx, req.Context)
        case "health_check":
            return c.healthCheck(ctx, req.Context)
        }
    }
    return nil, sidecar.ErrMethodNotFound
}
```

### 5.5 Tool 与 `brain/execute` 选型规则

- 读操作、短命令、告警上报，一律优先 `tools/call`
- 需要 sidecar 内部多步处理、可能跨多账号或多 provider 的任务，才使用 `brain/execute`
- `shutdown_prepare`、`collect_review_input`、`reconcile_state` 属于保留 instruction 名称，不参与普通业务编排
- 外部系统接入只面向 tool 名称；instruction 视为内部协议，必须在文档中维护版本

### 5.6 Kind 注册

Brain SDK 的 `agent.Kind` 是 `string` 类型（非枚举），可直接使用新字符串：

## 六、延迟预算

### 6.1 内部处理链路

```
OKX WebSocket 消息到达
    │
    ├─ [0.1ms] Frame 解析 + 反序列化
    ├─ [0.1ms] 数据质量校验
    ├─ [0.2ms] 数据清洗 + K线聚合
    ├─ [0.3ms] 特征向量计算
    ├─ [0.1ms] 写入 ring buffer ──── Data Brain 完成
    │
    ├─ [0.1ms] Quant Brain 读取 ring buffer
    ├─ [0.5ms] 4 策略并行计算
    ├─ [0.2ms] 信号聚合 + 阈值判断
    ├─ [0.1ms] 全局风控检查
    ├─ [0.2ms] per-account 风控检查
    ├─ [0.1ms] 仓位计算
    ├─ [0.5ms] 多账号并行下单 ──── Quant Brain 完成
    │
    └─ 内部总计: ~2.5ms（P99 < 5ms）
```

### 6.2 端到端延迟（含网络）

```
[~200ms]  OKX WS → 服务器（VPN 网络延迟，不可压缩）
[  ~2ms]  内部处理（数据→策略→风控→下单）
[~200ms]  下单请求 → OKX（网络延迟）
[ ~50ms]  OKX 撮合

端到端总计: ~452ms
```

**结论**: 内部处理仅占 0.4%，99% 延迟在网络。对秒级/分钟级策略完全够用。要降延迟应优化网络（如服务器迁至 OKX 机房附近）。

## 七、单机资源估算

| 组件 | 内存 | CPU |
|------|------|-----|
| Data Brain 进程 | ~50 MB | < 5% |
| Ring Buffer（3 品种 × 1.75 MiB） | ~5 MB | — |
| Quant Brain 进程 | ~30 MB | < 5% |
| Central Brain 进程（大部分时间空闲） | ~20 MB | < 1% |
| Kernel 主进程 | ~30 MB | < 1% |
| PostgreSQL 连接池 | ~50 MB | < 3% |
| **总计** | **~210 MB** | **< 15%** |

- 网络带宽: < 100 KB/s
- 磁盘写入: < 10 MB/天
- **最低要求: 2 核 4G VPS**

## 八、启动时序

```
T+0s    Kernel 启动
        │
T+0.1s  ├── 启动 Data Brain
        │   ├── 初始化 InstrumentNormalizer
        │   ├── 初始化 DataValidator
        │   ├── 注册所有 DataProvider（当前: okx_swap）
        │   ├── 创建 /dev/shm ring buffers
        │   ├── 各 Provider.Start()
        │   │   ├── OKX WebSocket 连接（超时 30s，失败 → 退避重试 3 次）
        │   │   └── REST 回填历史 K 线（超时 60s，失败 → 标记 degraded 继续）
        │   └── 报告 "data_ready"（含 shm 路径）─── ~3-5s
        │
        │   ★ 超时处理: Kernel 等待 data_ready 最多 60s
        │     超时 → 日志 CRITICAL + 退出（数据层是一切前提）
        │
T+5s    ├── 启动 Quant Brain（仅在 data_ready 后启动）
        │   ├── 打开 ring buffers（只读 mmap）
        │   │   └── 失败 → 日志 CRITICAL + 退出（无数据无法运行）
        │   ├── 初始化 GlobalRiskGuard
        │   ├── 初始化所有 AccountState
        │   │   ├── Account A: Risk Guard + OKX Executor + Private WS
        │   │   ├── Account B: Risk Guard + OKX Executor + Private WS
        │   │   └── Account C: Risk Guard + Paper Executor
        │   │   └── 单账号初始化失败 → 标记该账号 status="error"，不阻塞其他
        │   ├── 从 PostgreSQL 恢复各账号 MemoryState
        │   ├── 初始化 SignalTraceStore
        │   └── 报告 "quant_ready" ─── ~1s
        │
        │   ★ 超时处理: Kernel 等待 quant_ready 最多 30s
        │     超时 → 日志 CRITICAL + 退出
        │
T+6s    ├── 启动 Central Brain（仅在 quant_ready 后启动）
        │   ├── 初始化 LLM Client
        │   ├── 启动定时器（复盘、监控轮询）
        │   └── 报告 "central_ready" ─── ~0.5s
        │
        │   ★ 超时处理: Kernel 等待 central_ready 最多 15s
        │     超时 → 日志 WARNING（Central 非关键路径，Data+Quant 可独立运行）
        │
T+6.5s  └── 系统就绪，tick loop 开始
```

### 8.1 关闭时序

```
T+0s    Kernel 收到 SIGTERM
        │
T+0.1s  ├── Phase 1: 软关闭（grace period = 5s）
        │   ├── 通知 Quant Brain: brain/execute { instruction: "shutdown_prepare" }
        │   │   → Quant 停止开新仓
        │   │   → 完成当前 tick 周期
        │   │   → 保存所有 MemoryState 到 system_state
        │   │   → 关闭所有 Private WS 连接
        │   │   → 报告持仓状态快照
        │   │
        │   ├── 通知 Central Brain 停止定时器
        │   │   → 如有复盘正在进行，等待完成或 5s 超时取消
        │   │
        │   └── 通知 Data Brain: brain/execute { instruction: "shutdown_prepare" }
        │       → 停止所有 Provider 的 WebSocket 连接
        │       → 刷写内存中的未持久化 K 线
        │
T+5s    ├── Phase 2: 强制关闭
        │   ├── 如 Quant Brain 未响应 → Kernel 发送 SIGKILL
        │   ├── 如 Central Brain 未响应 → Kernel 发送 SIGKILL（无数据风险）
        │   └── 如 Data Brain 未响应 → Kernel 发送 SIGKILL
        │
T+5.5s  └── Kernel 退出
        │
★ 关闭顺序: Quant → Central → Data（Data 最后，确保 Ring Buffer 到最后一刻可读）
★ 不变量: 即使 SIGKILL，持仓数据已在 PostgreSQL 中持久化（每次成交后写入）
```

## 九、故障处理

### 9.1 故障检测与恢复状态机

```
                    ┌─────────────────────────────────────────────┐
                    │           Kernel 监控层（heartbeat 1s）      │
                    │                                             │
                    │  每个 sidecar 1s 一次 heartbeat             │
                    │  连续 3 次无响应 → 判定崩溃                  │
                    │  进程退出码非 0 → 立即判定崩溃               │
                    └───┬─────────────────┬─────────────────┬─────┘
                        │                 │                 │
                   Data Brain          Quant Brain      Central Brain
                   崩溃检测             崩溃检测          崩溃检测
```

### 9.2 Data Brain 崩溃

```go
// Quant Brain 内部检测（不依赖 Kernel 通知，自己也能发现）
func (q *QuantBrain) detectDataStale() {
    for instID, ring := range q.rings {
        lastSeq := ring.LastReadSeq()
        currentSeq := ring.PeekWriteSeq()
        if currentSeq == lastSeq && time.Since(ring.LastReadTime()) > 5*time.Second {
            q.enterDegradedMode(instID, "data_brain_stale")
        }
    }
}

// 降级模式状态机
const (
    ModeActive     = "active"      // 正常运行
    ModeDegraded   = "degraded"    // 数据停滞 — 停止开新仓，保持止损监控
    ModeRecovering = "recovering"  // Data Brain 已重启，等待数据恢复
)

func (q *QuantBrain) enterDegradedMode(instID, reason string) {
    q.mode = ModeDegraded
    // 1. 停止所有新开仓
    q.tradingEnabled = false
    // 2. 保持 processAccountOrders 运行（止损/止盈用最后已知价格）
    // 3. 通知 Central Brain
    q.callTool("central.data_alert", map[string]any{
        "level": "critical", "type": "data_stale", "symbol": instID, "reason": reason,
    })
}

// Data Brain 重启后
func (q *QuantBrain) onDataRecovered(instID string) {
    q.mode = ModeRecovering
    // 等待 ring buffer 连续 3 个有效写入后恢复
    if q.ringRecoveryCount[instID] >= 3 {
        q.mode = ModeActive
        q.tradingEnabled = true
    }
}
```

**恢复流程**: Kernel 重启 Data Brain → WS 重连 → Ring Buffer 恢复写入 → Quant 检测到 WriteSeq 增长 → 连续 3 个有效 tick 后恢复交易。

### 9.3 Quant Brain 崩溃（★ 最大风险点）

```go
// Kernel 检测到 Quant Brain 崩溃后的恢复序列
func (k *Kernel) recoverQuantBrain() {
    // Step 1: 立即重启 sidecar 进程（< 1s）
    q := k.restartSidecar("quant")

    // Step 2: Quant Brain 内部 recoverState()（详见 Doc 37）
    // 2a. 从 PostgreSQL trades 表恢复各账号持仓
    // 2b. 从 system_state 恢复 MemoryState 快照
    // 2c. 对比交易所持仓与本地记录，以交易所为准

    // Step 3: 重连 Private WS
    // 3a. 各 live 账号重新建立认证连接
    // 3b. 获取交易所返回的实时持仓列表
    // 3c. 用交易所持仓校正本地 MemoryState（以交易所为准）

    // Step 4: 恢复止损/止盈监控
    // 4a. 检查所有持仓的止损止盈挂单是否在交易所侧存活
    // 4b. 如果挂单丢失 → 重新提交

    // Step 5: 恢复 tick loop
    // 5a. 重新 mmap ring buffer
    // 5b. 恢复 market loop + housekeeping loop
    // 5c. 前 20 个有效 snapshot 仅做监控，不开新仓（warming up）
}

// 恢复期状态: 前 20 个有效 snapshot 不开新仓（约 100-300ms）
// 这确保策略指标有足够数据稳定
```

**不变量**: 崩溃期间的风险兜底依赖交易所侧的止损止盈挂单。设计要求：每笔开仓**必须同时在交易所提交止损挂单**（而非仅靠本地 tick 监控）。

### 9.4 Central Brain 崩溃（最低风险）

```
Central Brain 崩溃
  → Data + Quant 完全不受影响（独立运行）
  → Quant Brain 的 specialist.call_tool("central.review_trade")
    → 调用超时 10s → 降级为自动决策（Approved=true, Reason="central_unavailable"）
  → 每日复盘跳过（第二天补跑）
  → Kernel 重启 Central Brain → 无状态，直接恢复
```

### 9.5 单个账号执行器故障

```go
func (r *AccountRouter) handleAccountError(acct *AccountState, err error) {
    acct.Status = "error"
    acct.LastError = err.Error()
    acct.ConsecutiveErrors++

    if acct.ConsecutiveErrors >= 3 {
        // 连续 3 次失败 → 暂停该账号，通知 Central
        acct.Status = "paused"
        r.callTool("central.account_error", map[string]any{
            "level": "critical",
            "type":  "account_executor_failure",
            "detail": fmt.Sprintf("账号 %s 连续 %d 次执行失败: %s",
                acct.Config.Label, acct.ConsecutiveErrors, err),
        })
    }

    // 其他账号不受影响
    // 该账号已有持仓的止损挂单在交易所侧保持有效
}
```

### 9.6 数据质量异常

```go
func (b *DataBrain) onValidatorAlert(alert ValidatorAlert) {
    // 1. 持久化告警
    b.store.ValidatorAlerts().Save(ctx, alert)

    // 2. 通知 Central Brain
    b.callTool("central.data_alert", map[string]any{
        "level": alert.Level, "type": alert.Type,
        "symbol": alert.Symbol, "detail": alert.Detail,
    })

    // 3. 按类型处理
    switch alert.Type {
    case "price_spike":
        // 在 ring buffer 中设置该品种的 Flags |= FLAG_PRICE_SPIKE
        // Quant Brain 读到后自动跳过该品种的策略计算
        b.rings[alert.Symbol].SetFlag(FlagPriceSpike)
    case "gap":
        // 触发 REST 回填
        b.backfiller.RequestBackfill(alert.Symbol, alert.Detail)
    case "stale":
        // 5 分钟无数据 → 品种标记 degraded
        b.rings[alert.Symbol].SetFlag(FlagStaleData)
    }
}
```

## 十一、实现优先级

| Phase | 内容 | 目标 |
|-------|------|------|
| **Phase 1** | Data Brain MVP + OKX 合约 | 稳定产出 MarketSnapshot |
| **Phase 2** | Quant Brain + 多账号 + 全局风控 | 从 ring buffer 读数据跑策略 |
| **Phase 3** | 微观结构 + LLM 审查 + 回测 | OrderFlow 数据源、回测路径 |
| **Phase 4** | OKX 实盘 + 更多数据源 + 监控 | 上线实盘交易 |

各 Phase 细节见各模块文档。

## 十二、关键设计决策备忘

| 决策 | 选择 | 理由 |
|------|------|------|
| 三脑 vs 四脑 | 三脑 | GlobalRiskGuard 模块化预留拆分 |
| 共享内存 vs Socket | /dev/shm mmap | 单机最优解；接口预留替换 |
| Private WS 归属 | Quant Brain | 包含凭据，属于执行层 |
| 特征向量维度 | 192 维（配置可调） | 当前够用；扩展数据源时调整区段分配 |
| 信号审计粒度 | 每个信号完整追踪 | 复盘调参的基础 |
| 回测方案 | ReplayProvider + 同一 Pipeline | Quant Brain 零修改 |
