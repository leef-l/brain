## 二、与 Brain SDK v3 的关系

量化系统的三个大脑是 Brain SDK v3 的**消费者**，不修改 SDK 任何代码：

```
Brain SDK v3（不动）                量化系统（新增）
─────────────────                 ──────────────────
protocol/  — stdio JSON-RPC       ← Data/Quant/Central 遵守
kernel/    — Orchestrator          ← 调度三个大脑
sidecar/   — BrainHandler 接口     ← 三个大脑都实现它
agent/     — Kind 注册             ← 新增 "data", "quant" Kind
loop/      — Agent Loop 引擎       ← Central Brain 复用
tool/      — Tool Registry         ← 注册新工具（data.*, quant.*）
llm/       — LLM Provider          ← Central Brain 审查时调用
```

三个大脑与 Code Brain、Browser Brain 等同，是按 v3 Manifest-driven、Sidecar 架构接入的**专精大脑**。

## 三、设计原则

| 原则 | 说明 |
|------|------|
| **数据源插件化** | 所有数据源实现统一 `DataProvider` 接口，配置驱动热插拔 |
| **数据零拷贝** | 高频数据走共享内存 ring buffer，不经过 Kernel 中转 |
| **关键路径无 LLM** | tick→策略→下单全链路不调 LLM，延迟 < 5ms |
| **多账号隔离** | 每个账号独立风控、独立仓位、独立损益核算 |
| **多账号聚合** | 跨账号全局风控 + 中央大脑统一视图 |
| **故障隔离** | 三个大脑独立进程，任一崩溃不影响其他 |
| **双通道通信** | 快路径（共享内存）+ 慢路径（JSON-RPC） |
| **数据质量守门** | 所有数据经过校验层，异常数据不进入策略 |
| **全链路可追溯** | 每笔交易决策完整记录，支持回放审计 |
| **品种标准化** | 统一品种 ID，屏蔽各交易所命名差异 |

## 四、三脑分工

| 大脑 | Kind | 职责 | 关键路径 |
|------|------|------|---------|
| **Data Brain** | `data` | 数据采集、清洗、校验、特征计算、实时分发 | 是（快路径） |
| **Quant Brain** | `quant` | 策略计算、信号聚合、风控、多账号执行、审计 | 是（快路径） |
| **Central Brain** | `central` | LLM 审查、每日复盘、异常处理、全局监控 | 否（慢路径） |

### 为什么是三脑而不是四脑

考虑过独立 Risk Brain，但当前规模不需要。**GlobalRiskGuard 在 Quant Brain 内部模块化设计**，未来如果风控复杂度增长，可以零修改拉出来变成独立 sidecar。

## 五、双通道通信

### 5.1 快路径 vs 慢路径

```
┌──────────────────────────────────────────────────────────────────┐
│                       通信拓扑（单机部署）                         │
│                                                                  │
│                    Central Brain                                 │
│                   ┌────────────┐                                 │
│            JSON-RPC│  Kernel   │JSON-RPC                         │
│           (stdio)  │           │(stdio)                          │
│            ┌──────┘            └──────┐                          │
│            │                          │                          │
│     Data Brain                  Quant Brain                     │
│     ┌──────────┐                ┌──────────┐                    │
│     │          ├───────────────►│          │                    │
│     │          │  /dev/shm      │          │                    │
│     │          │  Ring Buffer   │          │                    │
│     └──────────┘  (< 0.1ms)    └──────────┘                    │
│                                                                  │
│  快路径: mmap ring buffer     慢路径: JSON-RPC via Kernel        │
│  延迟 < 0.1ms                 延迟 1-5ms                         │
│  无序列化                      完整审计                           │
│  99% 流量（tick）              1% 流量（控制信号）                 │
└──────────────────────────────────────────────────────────────────┘
```

**单机优势**: 共享内存 = 零网络开销的最优解。未来如需分布式，ring buffer 接口预留了可替换传输层。

### 5.2 通信场景矩阵

| 场景 | 发起方 | 接收方 | 通道 | 延迟 |
|------|--------|--------|------|------|
| 实时行情/特征推送 | Data | Quant | 共享内存 | < 0.1ms |
| 历史K线查询 | Quant | Data | JSON-RPC | < 50ms |
| 向量相似度查询 | Quant | Data | JSON-RPC | < 100ms |
| LLM 审查请求 | Quant | Central | JSON-RPC | < 10s |
| 强制平仓/暂停 | Central | Quant | JSON-RPC | < 100ms |
| 策略权重调整 | Central | Quant | JSON-RPC | < 100ms |
| 数据源健康检查 | Central | Data | JSON-RPC | < 1s |
| 数据质量告警 | Data | Central | JSON-RPC | < 1s |
| 每日复盘 | Quant | Central | JSON-RPC | < 30s |
| 宏观数据预警 | Data | Quant | ring buffer ContextFlags | < 0.1ms |
| 多账号状态汇总 | Quant | Central | JSON-RPC | < 100ms |
| 信号审计查询 | Central | Quant | JSON-RPC | < 200ms |

## 六、延迟预算

### 6.1 内部处理链路

```
OKX WebSocket 消息到达
    │
    ├─ [0.1ms] Frame 解析 + 反序列化
    ├─ [0.1ms] 数据质量校验
    ├─ [0.2ms] 数据清洗 + K线聚合
    ├─ [0.3ms] 特征向量计算
    ├─ [0.1ms] 写入 ring buffer ──── Data Brain 完成
    │
    ├─ [0.1ms] Quant Brain 读取 ring buffer
    ├─ [0.5ms] 4 策略并行计算
    ├─ [0.2ms] 信号聚合 + 阈值判断
    ├─ [0.1ms] 全局风控检查
    ├─ [0.2ms] per-account 风控检查
    ├─ [0.1ms] 仓位计算
    ├─ [0.5ms] 多账号并行下单 ──── Quant Brain 完成
    │
    └─ 内部总计: ~2.5ms（P99 < 5ms）
```

### 6.2 端到端延迟（含网络）

```
[~200ms]  OKX WS → 服务器（VPN 网络延迟，不可压缩）
[  ~2ms]  内部处理（数据→策略→风控→下单）
[~200ms]  下单请求 → OKX（网络延迟）
[ ~50ms]  OKX 撮合

端到端总计: ~452ms
```

**结论**: 内部处理仅占 0.4%，99% 延迟在网络。对秒级/分钟级策略完全够用。要降延迟应优化网络（如服务器迁至 OKX 机房附近）。

## 七、单机资源估算

| 组件 | 内存 | CPU |
|------|------|-----|
| Data Brain 进程 | ~50 MB | < 5% |
| Ring Buffer（3 品种 × 1.75 MiB） | ~5 MB | — |
| Quant Brain 进程 | ~30 MB | < 5% |
| Central Brain 进程（大部分时间空闲） | ~20 MB | < 1% |
| Kernel 主进程 | ~30 MB | < 1% |
| PostgreSQL 连接池 | ~50 MB | < 3% |
| **总计** | **~210 MB** | **< 15%** |

- 网络带宽: < 100 KB/s
- 磁盘写入: < 10 MB/天
- **最低要求: 2 核 4G VPS**

## 八、启动时序

```
T+0s    Kernel 启动
        │
T+0.1s  ├── 启动 Data Brain
        │   ├── 初始化 InstrumentNormalizer
        │   ├── 初始化 DataValidator
        │   ├── 注册所有 DataProvider（当前: okx_swap）
        │   ├── 创建 /dev/shm ring buffers
        │   ├── 各 Provider.Start()
        │   │   ├── OKX WebSocket 连接
        │   │   └── REST 回填历史 K 线
        │   └── 报告 "data_ready"（含 shm 路径）─── ~3-5s
        │
T+5s    ├── 启动 Quant Brain
        │   ├── 打开 ring buffers（只读 mmap）
        │   ├── 初始化 GlobalRiskGuard
        │   ├── 初始化所有 AccountState
        │   │   ├── Account A: Risk Guard + OKX Executor + Private WS
        │   │   ├── Account B: Risk Guard + OKX Executor + Private WS
        │   │   └── Account C: Risk Guard + Paper Executor
        │   ├── 从 PostgreSQL 恢复各账号 MemoryState
        │   ├── 初始化 SignalTraceStore
        │   └── 报告 "quant_ready" ─── ~1s
        │
T+6s    ├── 启动 Central Brain
        │   └── 报告 "central_ready" ─── ~0.5s
        │
T+6.5s  └── 系统就绪，tick loop 开始

关闭时序（反向）:
T+0s    Kernel 收到 SIGTERM
T+0.1s  ├── 通知 Quant Brain 停止开新仓
T+0.5s  ├── Quant Brain 完成当前 tick，报告所有持仓状态
T+1s    ├── 关闭 Quant Brain
T+1.5s  ├── 关闭 Central Brain
T+2s    └── 关闭 Data Brain（最后关闭，确保其他脑能读到最终数据）
```

## 九、故障处理

```
Data Brain 崩溃:
  → Quant Brain 检测 ring buffer 停滞（WriteSeq 不增长）
  → 所有账号停止开新仓，保持止损/止盈监控
  → DataValidator 最后已知价格用于止损判断
  → Kernel 重启 Data Brain，自动恢复

Quant Brain 崩溃:
  → ★ 最大风险点：持仓暂时无人监控
  → Kernel 立即重启
  → 从 PostgreSQL 恢复各账号 MemoryState
  → Private WS 重连，获取最新持仓状态
  → 恢复止损/止盈监控

Central Brain 崩溃:
  → Data + Quant 不受影响（独立运行）
  → LLM 审查超时降级为自动决策
  → Kernel 重启 Central Brain

单个账号执行器故障:
  → 仅该账号标记为 "error"
  → 其他账号不受影响
  → Central Brain 收到告警

数据质量异常:
  → DataValidator 拦截异常数据
  → 告警发送到 Central Brain
  → 价格 spike → 该品种暂停交易
  → K线 gap → 触发 REST 回填
```

## 十一、实现优先级

| Phase | 内容 | 目标 |
|-------|------|------|
| **Phase 1** | Data Brain MVP + OKX 合约 | 稳定产出 MarketSnapshot |
| **Phase 2** | Quant Brain + 多账号 + 全局风控 | 从 ring buffer 读数据跑策略 |
| **Phase 3** | 微观结构 + LLM 审查 + 回测 | OrderFlow 数据源、回测路径 |
| **Phase 4** | OKX 实盘 + 更多数据源 + 监控 | 上线实盘交易 |

各 Phase 细节见各模块文档。

## 十二、关键设计决策备忘

| 决策 | 选择 | 理由 |
|------|------|------|
| 三脑 vs 四脑 | 三脑 | GlobalRiskGuard 模块化预留拆分 |
| 共享内存 vs Socket | /dev/shm mmap | 单机最优解；接口预留替换 |
| Private WS 归属 | Quant Brain | 包含凭据，属于执行层 |
| 特征向量维度 | 192 维（配置可调） | 当前够用；扩展数据源时调整区段分配 |
| 信号审计粒度 | 每个信号完整追踪 | 复盘调参的基础 |
| 回测方案 | ReplayProvider + 同一 Pipeline | Quant Brain 零修改 |
