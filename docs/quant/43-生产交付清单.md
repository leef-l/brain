## 一、交付目标

本清单只回答一件事：

**把当前仓库中的 Data / Quant / Central 三脑代码，交付成可打包、可迁移、可核查、可回滚的产物。**

当前阶段仍以 `paper` 和受控环境为主，但以下项目必须满足后，才允许对外宣称“具备生产交付面”：

- release 包包含三脑与执行器二进制
- migration 文件可以被重复执行
- 配置模板和环境变量入口固定
- 启动与健康检查有脚本，不依赖人工临场敲命令
- smoke check 可以验证交付包完整性

## 二、Release 包内容

发布产物必须至少包含：

- `brain`
- `brain-central`
- `brain-data`
- `brain-quant`
- `exchange-executor`
- `brain-code`
- `brain-verifier`
- `brain-fault`
- `brain-browser`
- `config.example.json`
- `keybindings.example.json`
- `quant.example.yaml`
- `VERSION.json`
- `LICENSE`
- `README.md`
- `CHANGELOG.md`
- `SECURITY.md`
- `persistence/migrations/*.sql`
- `scripts/ops/*.sh`
- `docs/quant/43-生产交付清单.md`

验收：

- `./scripts/release/package.sh linux amd64 <version> <outdir>` 能产出归档
- 归档内存在 `MANIFEST.SHA256SUMS`
- `scripts/ops/health-check.sh <bundle-dir>` 只验证该 bundle 自身，不回退到源码树或本地 `bin/`
- 在存在 `MANIFEST.SHA256SUMS` 时，`health-check.sh` 必须实际执行校验；若宿主机缺少 `sha256sum` / `shasum`，应直接失败

## 三、迁移步骤

### 3.1 迁移前置

- PostgreSQL 已创建目标库
- `DATABASE_URL` 或 `PGHOST/PGPORT/PGUSER/PGDATABASE` 已准备
- 如需独立 schema，设置 `BRAIN_DB_SCHEMA`

### 3.2 执行命令

```bash
./scripts/ops/apply-migrations.sh status
./scripts/ops/apply-migrations.sh up
```

说明：

- migration 目录默认是 `persistence/migrations/`
- `apply-migrations.sh` 采用顺序执行，适合当前串行 SQL 基线
- 重复执行必须保持幂等；SQL 文件必须使用 `IF NOT EXISTS`

## 四、启动步骤

当前 sidecar 由 Kernel 统一拉起，不建议把 `brain-data` / `brain-quant` / `brain-central` 当成独立 daemon 手工后台托管。

推荐入口：

```bash
go build -o bin/brain ./cmd
./scripts/ops/start-kernel.sh
```

前台调试：

```bash
./scripts/ops/start-kernel.sh foreground
```

说明：

- `start-kernel.sh` 启动 `brain serve`
- sidecar 二进制需与 `brain` 同目录或位于 `PATH`
- Kernel 会在运行时按需拉起三脑 sidecar
- 常用环境变量：
  - `BRAIN_BINARY` 指定 `brain` 可执行文件
  - `BRAIN_LISTEN` 指定监听地址，默认 `127.0.0.1:7701`
- `BRAIN_LOG_DIR` / `BRAIN_LOG_FILE` / `BRAIN_PID_FILE` 指定日志与 PID 落位
- `BRAIN_PERMISSION_MODE` 指定 `serve` 运行模式
- `BRAIN_RUN_WORKDIR_POLICY` 指定请求级 workdir 策略
- `BRAIN_DATA_PERSIST_DRIVER` / `BRAIN_DATA_PERSIST_DSN` 控制 Data 快路径状态恢复后端；默认使用 `file`，落到 `$HOME/.brain/sidecars/brain-data.json`
- `BRAIN_CENTRAL_PERSIST_DRIVER` / `BRAIN_CENTRAL_PERSIST_DSN` 控制 Central control/review 状态恢复后端；默认使用 `file`，落到 `$HOME/.brain/sidecars/brain-central.json`
- `BRAIN_QUANT_PERSIST_DRIVER` / `BRAIN_QUANT_PERSIST_DSN` 控制 Quant 审计与 runtime 恢复后端；默认使用 `file`，落到 `$HOME/.brain/sidecars/brain-quant.json`

## 五、健康检查

静态检查：

```bash
./scripts/ops/health-check.sh
brain doctor
```

如需把 `brain doctor` 纳入同一条脚本 gate，可执行：

```bash
BRAIN_RUN_DOCTOR=1 ./scripts/ops/health-check.sh
```

说明：

- `health-check.sh` 在 `BRAIN_RUN_DOCTOR=1` 时默认以隔离的临时 `HOME` 运行 `brain doctor`，用于验证 release bundle 自身不会被宿主机已有 `~/.brain/config.json` 污染
- 如需直接核查部署机当前账号的真实配置、权限和凭据，使用 `BRAIN_DOCTOR_HOME_MODE=host BRAIN_RUN_DOCTOR=1 ./scripts/ops/health-check.sh` 或单独执行 `brain doctor`

HTTP 检查：

```bash
BRAIN_HEALTH_URL=http://127.0.0.1:7701/health ./scripts/ops/health-check.sh
```

最小 smoke check：

- release 包文件齐全
- `MANIFEST.SHA256SUMS` 可校验
- migration 可执行
- `brain doctor` 无失败项
- `/health` 返回 200
- `go test ./internal/central/... ./internal/data/... ./internal/quant/... ./persistence ./testing -count=1` 通过

## 六、回滚动作

- 停止 `brain serve`
- 回滚到上一个 release 包
- 如 schema 变更不兼容，必须准备独立 down migration 或备份恢复策略
- 未验证恢复流程前，不允许在 live 账号上直接切换版本

## 七、当前已知缺口

以下问题仍未关闭，不能因为交付面补齐就忽略：

- Quant 默认仍以 paper / 内存执行器为主，live 执行链未闭环
- Quant 的 `SignalTrace`、`pause_trading`、`pause_instrument`、最近一次 recovery 摘要已经通过 file-backed trace 默认落到 `$HOME/.brain/sidecars/brain-quant.json` 并可在 sidecar 重启后恢复
- Data 的最近 snapshot、validator 去重上下文和最近 provider health 摘要已默认落到 `$HOME/.brain/sidecars/brain-data.json` 并可在 sidecar 重启后恢复；但 provider lifecycle 与 live ingest attach 尚未闭环
- `brain-data` 现已支持静态 fixture provider 启动灌入和 provider drain 骨架，便于回放/联调；但这不是生产级真实交易所接入
- Central 的 control/review 状态已默认落到 `$HOME/.brain/sidecars/brain-central.json` 并可在 sidecar 重启后恢复；但 `daily_reviews` 的数据库化沉淀与更细粒度历史查询仍待补齐
- GlobalRiskGuard 已接入审查前后双检查，但当前 `PortfolioView` 只有汇总态，跨账号净敞口仍是保守近似，不是最终版持仓真值风控
- Data provider 生命周期和真实交易所接入仍不完整
- migration 已覆盖 `signal_traces`、`account_snapshots`、`daily_reviews`，但 `validator_alerts` 和更多数据侧表仍待继续落地
