# 核心逻辑与开发护栏

> 量化系统实现基线 — 单一状态归属 · 核心时序 · 不变量 · 禁区 · 验收标准

**上级文档**: [35-量化系统三脑架构总览](35-量化系统三脑架构总览.md)

---

## 一、文档目的

这份文档不重复三脑架构描述，只回答一件事：

**开发时什么是不能偏的核心逻辑。**

后续开发、评审、联调，以下内容应视为实现基线：
- 状态归属不能混
- 快慢路径不能串
- 责任边界不能越
- 核心链路顺序不能改
- 临时捷径不能破坏不变量

如果实现与本文件冲突，以本文件为准，再回写其他设计文档。

## 二、单一状态归属

| 状态 / 数据 | 唯一写入方 | 读取方 | 说明 |
|-------------|------------|--------|------|
| 原始市场数据 | Data Brain | Data / Quant | Data Brain 是公共行情唯一入口 |
| `MarketSnapshot` / Ring Buffer | Data Brain | Quant Brain | Quant 只读，不回写 |
| 特征向量 | Data Brain | Quant / Central | 对外查询走 `data.get_*` |
| 账号内存状态 `MemoryState` | Quant Brain | Central / 恢复流程 | 每账号独立维护 |
| 实盘持仓真值 | 交易所 + Private WS | Quant Brain | live 模式最终以交易所为准 |
| 审查结果 `ReviewDecision` | Central Brain | Quant Brain | Central 不直接执行订单 |
| 交易审计 `SignalTrace` | Quant Brain | Central Brain | 审计写入在 Quant 完成 |
| 每日复盘结果 | Central Brain | 人工 / Central | `daily_reviews` 唯一由 Central 写 |

核心规则：
- Data Brain 不持有账号凭据，不下单，不维护仓位
- Quant Brain 不接公共行情源，不写共享内存，不做每日复盘
- Central Brain 不直接操作执行器，不直接改 `MemoryState`

### 2.1 核心对象冻结

下面这些对象在 Phase 1 视为“冻结接口”，不能边写边换语义：

| 对象 | 归属 | 约束 |
|------|------|------|
| `stdID` | Data | 系统内部唯一品种标识，只有适配层可见 raw instID |
| `MarketSnapshot` | Data → Quant | Data 到 Quant 的唯一快路径载体，策略层不能直接消费 provider 原始 payload |
| `DispatchPlan` | Quant | 聚合信号后的唯一执行意图对象，执行层不接 `AggregatedSignal` 原始对象 |
| `ReviewDecision` | Central → Quant | 审查结果必须是结构化 JSON，不能在快路径里传播自由文本 |
| `SignalTrace` | Quant | 唯一审计对象，必须能解释“通过 / 拒绝 / 缩量 / 失败 / 不动作” |
| `MemoryState` | Quant | 本地交易状态视图，最终要和交易所真值对账，不能让数据库反向驱动 |

扩展规则：
- 这几个对象新增字段时，必须同步更新文档、schema、YAML 和对外接口说明
- 跨脑对象禁止退化成 `map[string]any` 长期流通
- 跨脑对象改语义时必须显式升版本，不能只改字段注释

## 三、唯一主链路

### 3.1 快路径主链路

正常交易路径必须严格保持以下顺序：

```text
Provider 原始事件
  -> DataValidator
  -> Processor / FeatureEngine
  -> Ring Buffer Write(MarketSnapshot, SeqNum++)
  -> Quant 发现 WriteSeq 变化
  -> MarketView 适配
  -> Strategy Pool
  -> Aggregator
  -> per-account 预检查，生成 DispatchPlan
  -> GlobalRiskGuard（审查前）
  -> 条件触发 review_trade
  -> 应用 size_factor
  -> GlobalRiskGuard（审查后复核）
  -> ExecutePlan
  -> 更新 MemoryState / 记录 SignalTrace
```

任何实现都不能跳过这几个关键节点：
- 不能跳过 `DataValidator`
- 不能跳过 `DispatchPlan`
- 不能把 `GlobalRiskGuard` 放到执行后
- 不能拿审查结果直接改信号，只能改候选单大小
- 不能执行完成后不写 `SignalTrace`

### 3.2 慢路径主链路

```text
异常 / 审查 / 复盘 / 配置变更
  -> specialist.call_tool 或 subtask.delegate
  -> Central Brain
  -> 分析 / 编排 / 建议
  -> 通过 quant.* / data.* 公开接口回调
```

慢路径规则：
- 慢路径永远不能绕过公开接口直接改内部状态
- `brain/execute` 仅用于长任务、恢复任务、关闭任务
- 监控、查询、告警、短控制命令优先走 `tools/call`

### 3.3 状态机冻结

开发阶段不允许各模块各自发明状态名，统一使用以下状态机。

#### Quant Brain 运行态

| 状态 | 含义 | 允许进入方式 | 允许动作 |
|------|------|--------------|----------|
| `booting` | 进程启动，资源初始化 | 进程启动 | 不开仓，不执行策略 |
| `recovering` | 对账、回补、重建本地状态 | `booting` 后或崩溃恢复 | 只允许恢复和校验，不开新仓 |
| `active` | 正常交易 | `recovering` 成功后 | 允许完整主链路 |
| `degraded` | 数据/依赖降级 | Data 异常、关键依赖异常 | 禁止新开仓，只允许减仓、平仓、止损、对账 |
| `paused` | 人工或中央脑暂停 | `active` / `degraded` | 禁止策略开仓，允许保护性动作 |
| `stopped` | 关闭完成 | 显式 shutdown | 不处理新请求 |

#### Account 运行态

| 状态 | 含义 | 规则 |
|------|------|------|
| `active` | 账号可正常接收候选单 | 完整执行 per-account risk + execute |
| `paused` | 账号被局部暂停 | 不能接收新开仓，可执行 reduce-only |
| `error` | 执行器或私有链路异常 | 必须上报 `central.account_error` |
| `recovering` | 私有状态回补和对账中 | 恢复完成前不得恢复新开仓 |

#### Data Provider 运行态

| 状态 | 含义 | 规则 |
|------|------|------|
| `starting` | provider 初始化 | 不向 ring buffer 写正式快照 |
| `syncing` | 首次拉齐、补 gap | 可以构建内部缓存，不向 Quant 宣称 ready |
| `active` | 数据正常 | 正常产出快照 |
| `degraded` | 数据源异常但仍有部分可用数据 | 必须带上质量标记，允许 Quant 进入降级 |
| `stopped` | 已停止 | 不再写入新序列 |

状态机规则：
- `degraded` 不是“还能正常交易”的别名，它代表只能保守运行
- `paused` 是控制动作，`degraded` 是系统事实，二者不能混用
- 新状态名如果不是这张表里的，必须先改文档再改代码

## 四、开发红线

### 4.1 数据红线

- 禁止 Quant Brain 直接连接 OKX Public WS
- 禁止 Data Brain 直接访问账号私有接口
- 禁止 Quant 用数据库轮询代替 Ring Buffer 读快照
- 禁止用 `symbol` 粒度做全局去重；必须按 `provider + topic + symbol`
- 禁止系统内部长期保留交易所 raw instID；内部统一使用 stdID

### 4.2 执行红线

- 禁止 Central Brain 直接下单或直接调用执行器
- 禁止 Data Brain 触发任何订单动作
- 禁止跳过 per-account 风控直接进入全局风控
- 禁止全局风控只看已有持仓，不看本次候选单
- 禁止 live 模式只靠本地止损监控；必须有交易所侧止损保护

### 4.3 协议红线

- 禁止 `Quant -> Central` 使用 `subtask.delegate`
- 禁止把公开能力同时做成“tool + 同名 instruction”双入口
- 禁止新增未文档化的 `instruction`
- 禁止在 PR 中随意扩展 tool 名称但不补 authorizer 规则

### 4.4 性能红线

- 禁止把数据库查询放入主交易快路径
- 禁止把 LLM 放在 Data Brain 内部或普通行情每 tick 必经路径
- 禁止把 Quant 主决策 loop 实现成固定 100ms 全量轮询

## 五、核心不变量

### 5.1 状态不变量

- Ring Buffer 只有 Data 写，Quant 只读
- `SignalTrace` 必须能解释“为什么没下单”
- live 模式下持仓最终以交易所返回为准
- 每次成交后，持仓状态与持久化状态必须可恢复

### 5.2 风控不变量

- 没有止损参数的开仓请求不能通过风控
- 审查缩量后必须重跑全局风控
- 同一信号对不同账号的处理结果必须可分别追踪
- 任何“自动降级放行”都必须留下机器可读原因

### 5.3 可恢复性不变量

- Data 崩溃后 Quant 必须进入 degraded，不得继续开新仓
- Quant 崩溃后恢复流程必须先对账再恢复开仓
- Central 崩溃不能影响 Data + Quant 正常运行

## 六、推荐实现顺序

### 6.1 第一步: 把接口面做死

先固定这些内容，再写业务：
- `data.*` / `quant.*` / `central.*` 工具名
- `brain/execute` instruction 白名单
- `SpecialistToolCallAuthorizer` 规则
- `MarketSnapshot` 字段和版本
- `DispatchPlan` / `ReviewDecision` / `SignalTrace` 结构

原因：
- 接口面不稳定，后面的代码一定返工

### 6.2 第二步: 先打通 paper 主链路

实现顺序必须优先：
1. Data Brain 生成稳定 `MarketSnapshot`
2. Quant Brain 跑 paper `DispatchPlan -> ExecutePlan`
3. SignalTrace 落库
4. 恢复流程可重启

先不要做：
- 宏观事件
- 新闻舆情
- 自动调权
- 复杂回测
- LLM 审查优化

### 6.3 第三步: 再接 LLM 与 slow path

只有在 paper 主链路稳定后，才做：
- `central.review_trade`
- `central.data_alert`
- `daily_review_run`
- `central.account_error`

### 6.4 第四步: 最后上实盘

只有以下条件全部满足后，才进入 live：
- Private WS 状态更新稳定
- 崩溃恢复已演练
- 止损保护能在交易所侧落地
- 信号审计和账号级追踪完整

## 七、模块边界建议

为了避免实现时“功能能跑，但代码落位越来越乱”，核心模块建议按职责切开：

| 模块 | 建议职责 | 不应出现的内容 |
|------|----------|----------------|
| `internal/data/provider` | 交易所/数据源接入、原始 payload 获取 | 策略逻辑、账号凭据、下单逻辑 |
| `internal/data/validator` | 去重、顺序检查、checksum、gap 检测 | 指标计算、数据库落库策略 |
| `internal/data/processor` | 标准化、聚合、特征、快照构造 | 账号风控、执行逻辑 |
| `internal/quant/view` | ring buffer 消费、MarketView 适配 | 交易所网络访问、LLM 调用 |
| `internal/quant/strategy` | 策略计算、信号聚合 | 落库、副作用 IO |
| `internal/quant/router` | 账号过滤、候选单构建、DispatchPlan | 直接下单 |
| `internal/quant/risk` | per-account 风控、GlobalRiskGuard | LLM 调用、数据库查询 |
| `internal/quant/execution` | executor、MemoryState、reconcile、stop-loss | 策略计算 |
| `internal/quant/audit` | SignalTrace、执行审计、状态持久化 | 决策逻辑改写 |
| `internal/central/review` | `central.review_trade`、模型调用、结构化审查结果 | 直接调 Quant 执行器 |
| `internal/central/control` | `data_alert`、`account_error`、daily review 编排 | 直接修改 Quant 内存状态 |

落位规则：
- 网络 IO 只出现在 provider、private ws、executor 相关模块
- 数据库 IO 不进入 `strategy`、`router`、`risk` 的主计算函数
- LLM 调用只允许出现在 Central Brain
- `SignalTrace` 的写入在 `audit` 收口，不要散落在策略和执行各处

## 八、模块完成定义

### 8.1 Data Brain 完成定义

以下才算完成，不是“能连上交易所”就算完：
- 能稳定产出 `MarketSnapshot`
- `SeqNum` 单调增长
- 校验异常不会污染快路径
- gap 能回填
- `data.get_snapshot` / `data.get_feature_vector` / `data.get_candles` 可用

### 8.2 Quant Brain 完成定义

- 以 `WriteSeq` 驱动决策
- 先出 `DispatchPlan`，再风控，再执行
- 每个账号可独立暂停 / 错误隔离
- `SignalTrace` 能覆盖通过、拒绝、缩量、失败四类路径
- 重启后能恢复到安全状态

### 8.3 Central Brain 完成定义

- `central.review_trade` 返回稳定 JSON
- `central.data_alert` / `central.account_error` 只是编排，不直接动执行器
- 每日复盘只通过公开接口取数
- 中央脑宕机不影响交易主链路

## 九、PR 自检清单

提交任何量化相关 PR 之前，至少自问这 14 个问题：

1. 这段改动是在快路径还是慢路径？
2. 有没有让 Data/Quant/Central 越界写对方状态？
3. 有没有新增未文档化的 tool / instruction？
4. 有没有把数据库或 LLM 放进主交易快路径？
5. 有没有绕过 `DispatchPlan` 直接执行？
6. 有没有绕过全局风控复核？
7. 审查缩量后的行为是否可追踪？
8. live 模式下是否仍然依赖交易所侧保护单？
9. 崩溃恢复后这段状态能不能重新构造？
10. `SignalTrace` 能否解释这次动作或不动作？
11. 新增字段是否同步更新 schema / YAML / 对外接口？
12. 这段实现是否把“临时方便”变成了长期协议债务？
13. 有没有偷偷引入新的运行状态名或状态跳转？
14. 这段代码是否放到了正确模块，而不是“哪里好写写哪里”？

只要其中有 1 个回答不清楚，就不应该合并。

## 十、禁止的“临时捷径”

这些做法开发期很诱人，但后面一定炸：

- “先让 Quant 直接读数据库，后面再换 ring buffer”
- “先让 Central 直接调执行器，后面再走 quant.*”
- “先把 review_trade 写成 instruction，后面再改 tool”
- “先不写 trace，等稳定了再补”
- “先用 raw instID，后面统一映射”
- “先按单账号实现，后面再抽多账号”

这些都不是临时方案，而是返工起点。

## 十一、推荐评审口径

Code review 时优先看这几类问题，而不是先看代码风格：
- 是否破坏状态归属
- 是否改乱快慢路径边界
- 是否让公开接口和内部指令混用
- 是否让恢复流程失真
- 是否让审计链断裂

如果这些问题没守住，代码即使能跑，也是在错误方向上加速。
