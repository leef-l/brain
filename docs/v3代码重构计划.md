
## Wave-2：架构整理

> **目标**：解决设计层面的冲突和不一致
>
> **依赖**：Wave-1 完成后开始

### 2.1 解决 TurnExecutor vs Runner 架构冲突

| 属性 | 值 |
|------|---|
| 文件 | `loop/turn_executor.go`（整文件）、`loop/turn.go:58-64`（Executor 接口）、`loop/runner.go` |
| 问题 | `Runner` 完全不使用 `Executor` 接口，`TurnExecutor` 是死代码 |
| 决策选项 | **选项 A**：删除 `TurnExecutor` + `Executor` 接口，Runner 直接驱动 Turn（简单）<br>**选项 B**：重构 Runner 通过 `Executor` 接口驱动（更灵活，支持测试替换） |
| 推荐 | 选项 B — 让 Runner 通过 Executor 接口驱动，TurnExecutor 作为默认实现 |
| 影响范围 | `loop/runner.go` 的 LLM 调用和工具分发逻辑迁移到 `TurnExecutor` 中 |

### 2.2 修复 Runner 状态机绕过

| 属性 | 值 |
|------|---|
| 文件 | `loop/runner.go:230,239` |
| 问题 | 直接赋值 `run.State = StateWaitingTool` 绕过状态机验证 |
| 修复 | 在 `run_helper.go` 中添加 `WaitTool(at time.Time)` 和 `ResumeTool(at time.Time)` 状态转换方法，Runner 通过方法调用而非直接赋值 |
| 额外 | 修复 `run.Fail()`/`run.Cancel()` 返回值被忽略的问题（runner.go:142,153,207） |

### 2.3 补全 LoopDetector 事件发送

| 属性 | 值 |
|------|---|
| 文件 | `loop/runner.go` |
| 问题 | 只发送 `tool_call` 事件，`content`/`llm_call` 事件从不发送 |
| 修复 | 在 Runner 的 LLM 调用前后发送 `llm_call` 事件，在解析 content block 时发送 `content` 事件 |
| 额外 | 在 Runner.Execute 的 defer 中调用 `LoopDetector.Forget(runID)` 防止内存泄漏 |
