### 🔴 C-1: Sandbox L2 容器沙箱校验未实现

**代码位置**：`sdk/security/hybrid_llm_access.go:187`
**原文位置**：`sdk/docs/23-安全模型.md` §5.2 "L2 容器沙箱"

```go
case SandboxL2:
    return brainerrors.New(brainerrors.CodeInvariantViolated,
        brainerrors.WithMessage("sandbox L2 (container) enforcement not yet implemented"))
```

**现状**：`SandboxEnforcer.ValidateLevel()` 中 L2 校验直接返回错误。任何配置 L2 的场景都会运行时失败。
**影响**：安全核心能力缺失，无法使用容器级沙箱。
**实现方向**：调研 Windows/Linux 容器方案（如 runc/windows container），实现 `ValidateLevel` 的 L2 分支。

### 🔴 C-3: SemanticApprover manifest 最小审批等级检查（Phase C 预留）

**代码位置**：`sdk/kernel/approval.go:87-97`
**原文位置**：`sdk/docs/23-安全模型.md` §6.1 "审批决策树"

```go
// 决策优先级（高到低）：
//  1. ApprovalRequest.Class 非空 → 直接使用（工具显式声明）
//  2. manifest 最小级别检查（预留接口，Phase C 填充）
//  3. 工具名前缀启发式规则（兜底）

// ManifestMinLevel 预留给 Phase C 的 manifest 最小审批等级查询。
ManifestMinLevel func(targetKind agent.Kind) ApprovalClass
```

**现状**：`DefaultSemanticApprover` 三层决策树的第二层（manifest 最小级别检查）完全为空。字段 `ManifestMinLevel` 已定义但从未赋值。
**影响**：审批系统仅依赖工具名前缀启发式规则，精确度低。
**实现方向**：
1. 在 `brain.json` manifest 中定义 `min_approval_class` 字段
2. 在 `NewOrchestrator` 或 brain 注册时读取该字段
3. 赋值给 `DefaultSemanticApprover.ManifestMinLevel`

### 🟢 C-6: `kernelLLMProvider.Stream` 真正的跨端实时流式 ✅

**代码位置**：`sdk/sidecar/kernel_provider.go` / `sdk/kernel/llm_proxy.go`
**原文位置**：`sdk/sidecar/kernel_provider.go`

**实现内容**：
1. 新增协议常量 `MethodLLMStreamDelta = "llm/stream/delta"`（host→sidecar 通知）
2. `LLMProxy.handleStream` 读取 provider StreamReader 时，若 sidecar 传了 `stream_id`，
   每读到一个 StreamEvent 立即通过 `rpc.Notify` 实时推送给 sidecar
3. `kernelLLMProvider.Stream` 生成 `stream_id`，注册 channel，goroutine 调用 `llm.stream`，
   返回 `channelStreamReader` 供 `loop.Runner` 实时消费
4. sidecar 在 stdio/WebSocket 双模式下均注册 `llm/stream/delta` handler，
   接收通知并写入 stream registry

**向后兼容**：旧 sidecar 不传 `stream_id` 时，host 回退到聚合模式，行为完全不变。
**性能**：`Notify` 为 fire-and-forget，每 token 零 RTT 等待。
