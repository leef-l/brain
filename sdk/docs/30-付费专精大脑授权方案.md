# 30. 付费专精大脑授权方案

> 目标：在不破坏开源主仓库定位的前提下，为付费 Specialist Brain
> 提供一套可离线分发、可验签、可过期、可限制能力范围的授权机制。

---

## 1. 设计结论

建议采用：

- **开源核心**：`brain` 主程序与基础内核保持开源
- **收费扩展**：付费专精大脑单独发布为 `Brain Package`，其中通常包含 sidecar runtime
- **授权介质**：本地 license 文件
- **校验方式**：runtime 内置公钥，离线验签
- **生效时机**：brain runtime 启动早期校验，通过后才进入 `initialize`

第一版**不要**做在线授权中心，也不要把授权耦合到 Kernel 主进程。

原因：

- 当前架构天然就是 sidecar 分发
- 离线 license 易部署，适合私有环境
- 验签逻辑收敛在收费二进制内部，不污染开源核心
- 出现授权问题时，失败范围只在该付费 brain，不影响全局系统

在 v3 术语里，可以把这套方案理解为：

- `Manifest` 声明这个 brain 需要授权
- `Package` 携带运行时、公钥和授权说明
- `Runtime` 在启动时执行真正的 license gate

---

## 2. 授权范围应该控制什么

授权不建议只判断“能不能启动”，而应该同时约束这些维度：

1. `customer`
   购买主体或组织名

2. `edition`
   版本档位，例如 `pro` / `enterprise`

3. `allowed_brains`
   允许启用哪些付费专精大脑

4. `features`
   某个专精大脑内部还能继续拆成功能开关

5. `expires_at`
   授权过期时间

6. `not_before`
   生效时间

7. `max_nodes`
   可选，限制部署节点数

8. `max_concurrency`
   可选，限制并发任务数

9. `metadata`
   订单号、销售渠道、签发备注

---

## 3. license 文件格式

推荐结构：

```json
{
  "schema_version": 1,
  "license_id": "lic_2026_000001",
  "customer": "Acme Inc.",
  "edition": "enterprise",
  "allowed_brains": ["browser-pro", "security", "data"],
  "features": {
    "browser-pro.recording": true,
    "browser-pro.parallel_tabs": true,
    "security.sast": true,
    "security.secrets_scan": true
  },
  "not_before": "2026-04-13T00:00:00Z",
  "expires_at": "2027-04-13T00:00:00Z",
  "max_nodes": 50,
  "max_concurrency": 20,
  "issued_at": "2026-04-13T00:00:00Z",
  "issuer": "leef-l",
  "metadata": {
    "order_id": "order_123456",
    "plan": "annual"
  },
  "signature": "base64(ed25519(signature over canonical payload))"
}
```

### 3.1 签名范围

`signature` 之外的全部字段参与签名。

为了避免不同语言 JSON 序列化顺序不一致，签名时必须使用：

- 固定字段顺序
- UTF-8
- RFC3339 UTC 时间
- sorted keys 的 canonical JSON

这和现有系统里对稳定输出、稳定 key 的要求一致。

---

## 4. 为什么选 Ed25519

第一版推荐 `Ed25519`：

- 标准库可直接实现
- 公私钥模型简单
- 签名短
- 校验快
- 适合离线 license 文件

不建议第一版就上：

- RSA：更重
- 在线 JWT introspection：太复杂
- 机器码强绑定：实施成本高、误伤率高

---

## 5. 授权文件放哪里

建议优先级：

1. `BRAIN_LICENSE_FILE`
2. `~/.brain/license.json`
3. 与 sidecar 同目录的 `license.json`
4. `brain config` 中的显式路径字段

推荐最终统一成一个 resolver：

```text
license_file flag
→ BRAIN_LICENSE_FILE
→ config.license.file
→ ~/.brain/license.json
→ <sidecar-dir>/license.json
```

第一版即使只实现前 3 条也够用。

---

## 6. 应该在什么时机校验

建议在**brain runtime 启动时**校验，而不是每次 `brain/execute` 时才校验。

推荐时序：

1. runtime 进程启动
2. 读取 license 文件
3. 验签
4. 检查时间窗、允许 brain、feature 集
5. 校验通过后再进入 `sidecar.Run(...)`
6. 校验失败则直接退出非零

这样做的好处：

- 失败更早，更容易定位
- 不会把未授权 brain 暴露给 orchestrator
- 不会出现 `initialize` 成功后再在执行期失败的半残状态

### 6.1 更严格的做法

如果你希望 host 在握手期就知道授权状态，可以让收费 sidecar 在
`initialize` 返回里附加：

```json
{
  "brainDescriptor": {
    "kind": "browser-pro",
    "version": "1.2.0",
    "license": {
      "status": "valid",
      "edition": "enterprise",
      "expires_at": "2027-04-13T00:00:00Z"
    }
  }
}
```

但这属于增强项，第一版不是必须。

## 7. 推荐的数据结构

建议新建一个独立包，例如：

```text
license/
  license.go
  signer.go
  verifier.go
  resolver.go
  errors.go
```

最小结构：

```go
type File struct {
    SchemaVersion  int                    `json:"schema_version"`
    LicenseID      string                 `json:"license_id"`
    Customer       string                 `json:"customer"`
    Edition        string                 `json:"edition"`
    AllowedBrains  []string               `json:"allowed_brains"`
    Features       map[string]bool        `json:"features,omitempty"`
    NotBefore      time.Time              `json:"not_before"`
    ExpiresAt      time.Time              `json:"expires_at"`
    MaxNodes       int                    `json:"max_nodes,omitempty"`
    MaxConcurrency int                    `json:"max_concurrency,omitempty"`
    IssuedAt       time.Time              `json:"issued_at"`
    Issuer         string                 `json:"issuer"`
    Metadata       map[string]interface{} `json:"metadata,omitempty"`
    Signature      string                 `json:"signature"`
}
```

验证结果：

```go
type Result struct {
    Valid          bool
    LicenseID      string
    Customer       string
    Edition        string
    AllowedBrains  []string
    Features       map[string]bool
    ExpiresAt      time.Time
    MaxNodes       int
    MaxConcurrency int
}
```

## 8. sidecar 启动接线建议

对收费专精大脑，建议 runtime 的 `main()` 变成：

```go
func main() {
    res, err := license.VerifyForBrain("browser-pro", license.Options{
        LicensePath: "",
        PublicKeyPEM: embeddedKey,
        Now: time.Now(),
    })
    if err != nil {
        fmt.Fprintf(os.Stderr, "brain-browser-pro: license: %v\n", err)
        os.Exit(2)
    }

    _ = res

    if err := sidecar.Run(newBrowserProHandler()); err != nil {
        fmt.Fprintf(os.Stderr, "brain-browser-pro: %v\n", err)
        os.Exit(1)
    }
}
```

也就是：

- 未授权：直接退出
- 已授权：正常进入 sidecar runtime

### 8.1 feature 下沉

`newBrowserProHandler(res)` 可以拿到验证结果，把 feature 开关直接下沉到工具注册：

```go
if res.Features["browser-pro.parallel_tabs"] {
    reg.Register(NewParallelTabsTool())
}
```

这样就能做到：

- 同一个二进制
- 不同客户 license 开不同功能

## 9. 错误码建议

当前 `errors/codes.go` 里还没有授权相关错误码。

建议新增这组：

- `license_not_found`
- `license_invalid_signature`
- `license_expired`
- `license_not_yet_valid`
- `license_brain_not_allowed`
- `license_feature_not_allowed`
- `license_schema_unsupported`

建议分类：

- 大多数归 `Permanent`
- 如果是授权服务不可达这种联网场景错误，才归 `Transient`

第一版既然是离线 license，就不需要网络类错误码。

## 10. CLI / 配置怎么接

### 10.1 不建议先塞进主配置核心路径

第一版不要强耦合进 `brain run` 主流程。

更合理的是给 sidecar 自己读：

- `BRAIN_LICENSE_FILE`
- 本地默认路径

等稳定后再加到主配置：

```json
{
  "license": {
    "file": "/etc/brain/license.json"
  }
}
```

### 10.2 可选增加 doctor 检查

后续可以在 `brain doctor` 增加：

- 是否发现 license 文件
- 过期时间
- 是否允许当前已安装的付费 brains

但第一版不是必须。

### 11.1 开源核心 release

包含：

- `brain`

### 11.2 付费 brain release

单独发布：

- `brain-browser-pro`
- `brain-security`

客户拿到：

- package 资产
- `manifest.json`
- runtime 二进制
- `license.json`
- 校验说明

推荐进一步按 [34-Brain-Package与Marketplace规范.md](./34-Brain-Package与Marketplace规范.md) 打包，这样边界更清晰，也更利于 Marketplace 分发。

## 12. 续费、升级、吊销怎么做

第一版建议只做这 3 件事：

1. **续费**
   重新签一个新的 `expires_at`

2. **升级**
   换一个新 license，增加 `allowed_brains` 或 `features`

3. **吊销**
   第一版先不做在线吊销；如果必须吊销，就通过到期时间较短 + 重发 license 控制

为什么先不做吊销列表：

- 离线场景不好同步
- 复杂度明显升高
- 容易把第一版拖成授权平台项目

---

## 13. 不建议第一版做的东西

这些都可以以后再上：

- 在线激活
- 授权服务器心跳
- 机器码硬绑定
- USB Dongle
- 按调用量实时扣费
- 远程吊销列表
- 混淆和反调试

这些不是不能做，而是第一版性价比很低。

---

## 14. 推荐落地顺序

### 第一步

先做离线签名 license：

- `license.File`
- canonical JSON
- Ed25519 sign / verify
- sidecar 启动拦截

### 第二步

做 feature gating：

- `allowed_brains`
- `features`
- `expires_at`

### 第三步

做运维可见性：

- `brain doctor` 展示 license 状态
- sidecar initialize 回传授权摘要

### 第四步

再考虑商业平台化：

- 客户后台
- 自动签发
- 发票/订单系统
- 在线续费

---

## 15. 一句话方案

对这个项目，最合适的商业化路径不是“把开源仓库许可证改成企业禁用”，而是：

- **主仓库开源**
- **收费能力放在付费专精大脑 package**
- **用离线签名 license 文件做授权**

这条路线和你现在的架构是最匹配的。
