## 1. L1 协作级学习算法

### 1.1 核心问题

L1 回答一个问题：**给定一类任务，应该委托给哪个 brain？**

这不是静态路由（"数据任务给 data brain"），而是动态能力画像——随着各 brain 的实际表现，持续调整委托偏好。

### 1.2 Brain 能力画像数据结构

```go
// BrainCapabilityProfile 是 Central 对某个 brain 的能力认知模型
type BrainCapabilityProfile struct {
    BrainKind   string    // "quant" | "data" | "code" | "browser" | "verifier" | "fault"
    UpdatedAt   time.Time

    // 按任务类型分组的评分记录
    // key: taskType，如 "code_review", "data_fetch", "trade_analysis"
    TaskScores  map[string]*TaskTypeScore

    // 全局能力指标（跨任务类型的综合评估）
    GlobalMetrics BrainGlobalMetrics

    // 冷启动状态
    ColdStart bool // true = 接入 < MinSamplesForRanking 个样本
}

// TaskTypeScore 某 brain 对某类任务的综合评分
type TaskTypeScore struct {
    TaskType    string
    SampleCount int       // 样本数（越大越可信）

    // 四维评分（各维度独立存储，支持单独查询和权重动态调整）
    Accuracy    EWMAScore // 准确率：输出质量符合预期的比例
    Speed       EWMAScore // 速度：任务耗时的倒数归一化（快=高分）
    Cost        EWMAScore // 成本：token 消耗量的倒数归一化（省=高分）
    Stability   EWMAScore // 稳定性：连续成功率（无错误/重试/超时）

    // 置信区间（Wilson score interval，用于小样本修正）
    ConfidenceLow  float64
    ConfidenceHigh float64

    // 滑动窗口时间戳（用于淘汰过期数据）
    LastUpdated time.Time
    WindowStart time.Time // 当前统计窗口起点
}

// EWMAScore 指数加权移动平均评分
type EWMAScore struct {
    Value   float64   // 当前 EWMA 值，范围 [0, 1]
    Alpha   float64   // 衰减系数，0.1～0.3，根据任务频率动态调整
    Updated time.Time
}

// BrainGlobalMetrics 来自 L0 上报的 BrainMetrics 聚合
type BrainGlobalMetrics struct {
    OverallSuccessRate float64
    AvgLatency         time.Duration
    ConfidenceTrend    float64 // 正值=能力上升，负值=下降
    DomainMetrics      map[string]float64
}

// BrainRanking 是 RankBrainsForTask 的返回值
type BrainRanking struct {
    BrainKind    string
    Score        float64 // 综合评分 [0, 1]
    Confidence   float64 // 评分可信度 [0, 1]（样本越少越低）
    Explanation  string  // 可读解释，如 "准确率 0.92，速度 0.78，样本 43"
    IsColdStart  bool
}
```

### 1.3 多维评分模型

#### 1.3.1 四维指标定义

**准确率（Accuracy）**：delegate 结果是否符合预期质量标准。
- 计算方式：每次 delegate 完成后，由 Central 的结果评审（见 §1.4）给出 0-1 分，EWMA 累积。
- 权重初始值：`w_accuracy = 0.40`（最重要，质量优先）

**速度（Speed）**：任务耗时的归一化倒数。
- 计算方式：`speed_score = ref_latency / actual_latency`，上限 1.0。`ref_latency` 为同类任务历史中位数。
- 权重初始值：`w_speed = 0.25`

**成本（Cost）**：LLM token 消耗量的归一化倒数。
- 计算方式：`cost_score = ref_tokens / actual_tokens`，上限 1.0。`ref_tokens` 为同类任务历史中位数。
- 权重初始值：`w_cost = 0.15`

**稳定性（Stability）**：无错误/重试/超时的连续成功率。
- 计算方式：基于最近 N 次 delegate 的滑动窗口（N=20），无异常计为 1，有异常计为 0，取均值。
- 权重初始值：`w_stability = 0.20`

#### 1.3.2 权重动态调整机制

权重不是固定的，根据**任务上下文**动态调整：

```go
// WeightPolicy 定义权重调整策略
type WeightPolicy struct {
    // 用户/任务标注的优先级信号
    LatencyPriority bool // "尽快完成" → speed 权重 x2
    CostPriority    bool // "节省成本" → cost 权重 x2
    QualityPriority bool // "高质量优先" → accuracy 权重 x1.5，speed/cost 各 x0.5
}

func ComputeWeights(policy WeightPolicy) (wAcc, wSpd, wCst, wStab float64) {
    // 基础权重
    wAcc, wSpd, wCst, wStab = 0.40, 0.25, 0.15, 0.20

    if policy.LatencyPriority {
        wSpd *= 2.0
    }
    if policy.CostPriority {
        wCst *= 2.0
    }
    if policy.QualityPriority {
        wAcc *= 1.5
        wSpd *= 0.5
        wCst *= 0.5
    }

    // 归一化，保证总和为 1
    total := wAcc + wSpd + wCst + wStab
    return wAcc / total, wSpd / total, wCst / total, wStab / total
}
```

#### 1.3.3 综合评分计算

```
// 伪代码：计算某 brain 对某 taskType 的综合评分
func ComputeScore(profile TaskTypeScore, weights [4]float64) float64:
    raw_score = weights[0] * profile.Accuracy.Value
             + weights[1] * profile.Speed.Value
             + weights[2] * profile.Cost.Value
             + weights[3] * profile.Stability.Value

    // Wilson score 修正：样本少时向 0.5 收缩
    confidence = WilsonConfidence(profile.SampleCount)
    adjusted_score = confidence * raw_score + (1 - confidence) * 0.5

    return adjusted_score

// Wilson score interval 下界（保守估计）
func WilsonConfidence(n int) float64:
    if n == 0: return 0.0
    if n < 5:  return float64(n) / 5.0 * 0.6  // 线性插值到 0.6
    if n < 20: return 0.6 + (float64(n)-5)/15.0 * 0.3  // 插值到 0.9
    return 0.9 + min(float64(n)-20, 80)/80.0 * 0.1  // 最终收敛到 1.0
```

### 1.4 Delegate 结果评分标准

#### 1.4.1 什么算"好的 delegate"？

一次 delegate 从以下五个维度综合评判，最终产出 [0, 1] 的综合分：

```go
// DelegateResult 是一次 delegate 完成后的结构化评估
type DelegateResult struct {
    DelegateID  string
    FromBrain   string
    ToBrain     string
    TaskType    string
    StartedAt   time.Time
    FinishedAt  time.Time

    // 五维评判
    Completion  CompletionScore  // 任务完成度：是否输出了预期的结果
    Quality     QualityScore     // 输出质量：输出是否准确、完整、无幻觉
    Timeliness  TimelinessScore  // 时效性：是否在时限内完成
    Efficiency  EfficiencyScore  // 效率：token 消耗是否合理
    SideEffects SideEffectScore  // 副作用：是否有异常错误、重试、工具失败

    FinalScore  float64          // 加权综合分
    ScoredBy    string           // "auto" | "central_review" | "user_feedback"
}

type CompletionScore struct {
    Score   float64 // 0=未完成，0.5=部分完成，1=完全完成
    Reason  string
}

type QualityScore struct {
    Score   float64 // 0-1，由 Central LLM 复审或对比预期输出计算
    Method  string  // "llm_review" | "schema_match" | "metric_compare"
}

type TimelinessScore struct {
    Score    float64
    Actual   time.Duration
    Deadline time.Duration // 0 = 无时限，自动满分
}

type EfficiencyScore struct {
    Score       float64
    TokensUsed  int
    TokensRef   int   // 同类任务历史中位数
}

type SideEffectScore struct {
    Score      float64 // 1=无副作用，<1=有问题
    ErrorCount int
    RetryCount int
    TimeoutCount int
}
```

#### 1.4.2 评分计算规则

```
// 0-1 评分的具体评判逻辑（伪代码）
func ScoreDelegate(result RawDelegateResult) DelegateResult:

    // 1. 完成度评分（最重要，不完成其他都不算）
    if result.Status == "error" or result.Output == nil:
        completion = 0.0
    elif result.OutputComplete:
        completion = 1.0
    else:
        completion = 0.5  // 部分完成（如返回了部分数据）

    // 2. 质量评分（依赖任务类型）
    quality = EvaluateQuality(result.TaskType, result.Output, result.Expected)
    // quality 的具体实现见下方分任务类型规则

    // 3. 时效性评分（软性约束）
    if result.Deadline == 0:
        timeliness = 1.0  // 无时限
    elif result.ActualDuration <= result.Deadline:
        timeliness = 1.0
    elif result.ActualDuration <= result.Deadline * 2:
        timeliness = 0.5  // 超时但在 2x 以内
    else:
        timeliness = 0.0  // 严重超时

    // 4. 效率评分
    if result.TokensUsed <= result.TokensRef * 1.2:
        efficiency = 1.0
    elif result.TokensUsed <= result.TokensRef * 2.0:
        efficiency = 0.7
    else:
        efficiency = 0.4  // 异常高消耗

    // 5. 副作用评分
    side_effects = 1.0
    side_effects -= result.ErrorCount * 0.2   // 每次错误扣 0.2
    side_effects -= result.RetryCount * 0.1   // 每次重试扣 0.1
    side_effects -= result.TimeoutCount * 0.3 // 每次超时扣 0.3
    side_effects = max(side_effects, 0.0)

    // 加权综合分
    // 完成度权重最高，副作用次之，质量、时效、效率平衡
    final = 0.35 * completion
          + 0.30 * quality
          + 0.15 * timeliness
          + 0.10 * efficiency
          + 0.10 * side_effects

    // 完成度为 0 时，整体最高只能得 0.1（意味着有过程价值）
    if completion == 0.0:
        final = min(final, 0.1)

    return DelegateResult{FinalScore: final, ...}
```

#### 1.4.3 分任务类型的质量评判规则

| 任务类型 | 质量评判方法 | 满分条件 |
|----------|-------------|---------|
| `code_review` | LLM 复审：检查是否指出了真实问题，无幻觉问题 | 问题覆盖率 ≥ 80%，误报率 < 10% |
| `data_fetch` | Schema 验证：输出是否符合预期 JSON/表格格式 + 数据完整性 | 字段完整、时间范围覆盖、无 null 关键字段 |
| `trade_analysis` | 指标对比：信号方向是否一致，关键指标误差 < 5% | 方向准确 + 误差在阈值内 |
| `web_scrape` | 内容验证：提取的关键字段是否存在且合理 | 关键字段提取率 ≥ 95% |
| `fault_diagnosis` | 恢复验证：问题是否解决，服务是否恢复正常 | 服务健康检查通过 |
| `generic` | LLM 复审（Central 执行）：对比输出和任务描述的匹配程度 | 综合评分 ≥ 0.7 |

### 1.5 EWMA 更新算法

```go
// UpdateEWMA 用新观测值更新 EWMA 评分
// alpha 越大，新数据权重越高，历史数据衰减越快
func UpdateEWMA(current EWMAScore, newValue float64, taskFreq TaskFrequency) EWMAScore:
    // 根据任务频率动态调整 alpha
    // 高频任务（每天 >10 次）：alpha=0.3，快速响应变化
    // 中频任务（每天 1-10 次）：alpha=0.2
    // 低频任务（每天 <1 次）：alpha=0.1，更保守
    alpha = AlphaForFrequency(taskFreq)

    newEWMA = alpha * newValue + (1 - alpha) * current.Value
    return EWMAScore{Value: newEWMA, Alpha: alpha, Updated: now()}
```

### 1.6 RankBrainsForTask 算法

```go
// RankBrainsForTask 对所有可用 brain 排序，返回推荐委托列表
func RankBrainsForTask(ctx, taskType string) ([]BrainRanking, error):
    candidates = GetActiveBrains()  // 从 Brain Pool 获取健康的 brain 列表
    rankings   = []BrainRanking{}

    policy = ExtractWeightPolicy(ctx)  // 从任务上下文提取权重偏好
    weights = ComputeWeights(policy)

    for each brain in candidates:
        profile = LoadProfile(brain.Kind, taskType)

        if profile == nil or profile.IsColdStart:
            // 冷启动处理（见 §1.7）
            rankings = append(rankings, ColdStartRanking(brain))
            continue

        score = ComputeScore(profile, weights)
        confidence = WilsonConfidence(profile.SampleCount)

        rankings = append(rankings, BrainRanking{
            BrainKind:   brain.Kind,
            Score:       score,
            Confidence:  confidence,
            Explanation: FormatExplanation(profile, weights),
        })

    // 按 score 降序排列，confidence 低的排后面（置信区间修正已内含在 score 中）
    sort.Slice(rankings, func(i, j int) bool {
        return rankings[i].Score > rankings[j].Score
    })

    return rankings
```

### 1.7 冷启动策略

新 brain 首次接入时，历史数据为零，无法用历史评分排名。冷启动策略：

```go
// 冷启动阶段定义
const (
    MinSamplesForRanking = 5   // 至少 5 个样本才进入正式排名
    ColdStartScore       = 0.6 // 冷启动期间使用的默认评分（略低于平均水平）
    ExplorationBudget    = 0.2 // 探索预算：20% 的委托强制分配给冷启动 brain
)

func ColdStartRanking(brain BrainInfo) BrainRanking:
    return BrainRanking{
        BrainKind:   brain.Kind,
        Score:       ColdStartScore,
        Confidence:  0.0, // 信心为零，提示调用方
        IsColdStart: true,
        Explanation: fmt.Sprintf("新 brain，样本不足（当前 %d / 需要 %d）", brain.SampleCount, MinSamplesForRanking),
    }

// 委托决策时，保留探索预算
func SelectBrainForDelegate(rankings []BrainRanking, rng *rand.Rand) string:
    coldStartBrains = filter(rankings, r => r.IsColdStart)
    normalBrains    = filter(rankings, r => !r.IsColdStart)

    // 20% 概率强制选择冷启动 brain（加速冷启动）
    if len(coldStartBrains) > 0 and rng.Float64() < ExplorationBudget:
        return RandomChoice(coldStartBrains).BrainKind

    // 80% 概率选择最优 brain
    if len(normalBrains) > 0:
        return normalBrains[0].BrainKind  // 已排序，取最高分

    // 兜底：只有冷启动 brain
    return RandomChoice(coldStartBrains).BrainKind
```

**冷启动期间的默认能力假设**：根据 brain 的 Manifest `capabilities` 标签，对任务类型做预设评分。例如：`browser` brain 对 `web_scrape` 类任务预设 `accuracy=0.7`，对 `trade_analysis` 预设 `accuracy=0.3`。这减少了盲目探索的成本。

#### 1.8.2 淘汰策略

数据过多时，按以下规则淘汰：

| 淘汰维度 | 规则 | 原因 |
|----------|------|------|
| **时间窗口** | 默认 30 天滚动窗口，30 天前的 `delegate_log` 条目丢弃 | brain 能力会变化，过老数据失效 |
| **样本上限** | 每个 taskType 最多保留 500 条原始记录（超出后 FIFO） | 存储上限，EWMA 已摘要化 |
| **brain 下线** | brain 下线超过 7 天，profile 标记 `archived`，不再参与排名 | 避免过期数据影响新接入的同名 brain |

EWMA 本身是对历史的摘要，不需要保留所有原始样本——`profiles/*.json` 是稳态存储，`delegate_log.jsonl` 是原始事件流用于审计和离线分析。

## 2. L2 策略级学习算法

### 2.1 核心问题

L2 回答：**对于某类任务，哪种 brain 调用序列/组合效果最好？**

L1 解决"选谁"，L2 解决"怎么排兵布阵"——多步任务时，先 data 后 quant 还是并行？code 和 verifier 哪个先？

### 2.2 ExecutionChain 数据结构

```go
// ExecutionChain 描述一次多步任务的完整执行链路
type ExecutionChain struct {
    ChainID     string
    TaskType    string    // 顶层任务类型，如 "strategy_research"
    StartedAt   time.Time
    FinishedAt  time.Time

    // 链路节点图（有向无环图，DAG）
    Nodes    []ChainNode
    Edges    []ChainEdge

    // 链路摘要（用于快速模式匹配）
    Fingerprint string // 规范化的节点序列哈希，如 "data→quant→verifier"
}

// ChainNode 执行链路中的一个节点（一次 delegate 或一次工具调用）
type ChainNode struct {
    NodeID    string
    Kind      NodeKind  // "delegate" | "tool_call" | "llm_turn"
    BrainKind string    // 执行的 brain（delegate 时有效）
    ToolName  string    // 工具名（tool_call 时有效）
    TaskType  string    // 该节点的子任务类型

    // 执行结果
    Score     float64   // 该节点的评分（来自 L1 DelegateResult.FinalScore）
    Duration  time.Duration
    TokensUsed int
    Status    string    // "success" | "failed" | "timeout" | "skipped"

    // 节点位置信息
    Depth     int       // 在 DAG 中的深度（根节点=0）
}

// ChainEdge 节点间的依赖关系
type ChainEdge struct {
    From string  // NodeID
    To   string  // NodeID
    Kind EdgeKind // "sequential" | "parallel" | "conditional"
    Condition string // conditional 时：触发条件描述（如 "data_fetched=true"）
}

// Outcome 整条链路的最终结果
type Outcome struct {
    OverallScore float64  // 整链综合评分 [0, 1]
    UserFeedback float64  // 用户反馈分（-1=无反馈）
    Succeeded    bool
    FailedAt     string   // 失败的 NodeID（如有）
    TotalDuration time.Duration
    TotalTokens   int
}
```

#### 2.2.1 链路指纹（Fingerprint）

链路指纹是 ExecutionChain 的规范化表示，用于快速相似度匹配：

```
// 生成链路指纹
func ComputeFingerprint(chain ExecutionChain) string:
    // 拓扑排序后，按节点类型和顺序生成字符串
    // 并行节点用 "{}" 包裹，顺序节点直接连接
    // 示例：
    //   顺序链：data→quant→verifier
    //   并行段：{data,browser}→code→verifier
    //   条件分支：data→[quant|code]→verifier
    sorted = TopologicalSort(chain.Nodes, chain.Edges)
    return EncodeDAGToString(sorted, chain.Edges)
```

### 2.4 因果推断：区分真因果 vs 虚假关联

**问题**："data→quant 成功率高"可能是因为这个顺序本身好，也可能是因为选择这个顺序的任务本来就更简单。

```
// 因果推断框架（简化版反事实推断）
func AssessCausality(pattern ChainPattern, alternativePatterns []ChainPattern):

    // Step 1: 样本量检查——小样本不做因果推断
    if pattern.SampleCount < MinSamplesForCausality (default=10):
        return CausalityResult{Confidence: "insufficient_data"}

    // Step 2: 置信区间重叠检查
    // 如果最优模式和次优模式的 95% CI 重叠，则差异不显著
    best = pattern
    for alt in alternativePatterns:
        if alt.Fingerprint != best.Fingerprint:
            overlap = CIOverlap(best.ScoreCI, alt.ScoreCI)
            if overlap > 0:
                // CI 重叠，记录为"可能差异"而非"确定优势"
                best.CausalConfidence = "weak"

    // Step 3: Simpson's Paradox 检查（按任务复杂度分层）
    // 将历史样本按 ComplexityBucket (low/medium/high) 分层
    // 在每个分层内分别计算 pattern 成功率
    // 如果分层后模式消失，则可能是混淆变量导致
    stratified = StratifyByComplexity(pattern.Samples)
    for bucket in [low, medium, high]:
        bucket_rate = stratified[bucket].SuccessRate
        // 如果某个 bucket 内成功率反转，标记为 confounded
        if bucket_rate < alternative_bucket_rate:
            return CausalityResult{Confidence: "possibly_confounded"}

    // Step 4: 时间稳定性检查
    // 将历史数据分为前半段和后半段，比较成功率是否一致
    first_half_rate = pattern.Samples[:n/2].SuccessRate
    second_half_rate = pattern.Samples[n/2:].SuccessRate
    if abs(first_half_rate - second_half_rate) > 0.15:
        pattern.Trend = "unstable"  // 效果不稳定，可能是外部因素

    return CausalityResult{
        Confidence: "strong",
        BestPattern: best,
    }
```

#### 2.4.1 置信区间计算

使用 Wilson score interval 的变体（适用于非二值指标的连续分数）：

```go
// 连续评分的置信区间（基于 t 分布，当 n<30 时）
func ScoreCI(scores []float64, confidence float64) [2]float64:
    n    = len(scores)
    mean = Mean(scores)
    std  = StdDev(scores)
    se   = std / sqrt(float64(n))  // 标准误差

    // t 分布临界值（confidence=0.95，df=n-1）
    t = TDistCritical(confidence, n-1)

    return [2]float64{mean - t*se, mean + t*se}
```

### 2.5 链路模式匹配算法

给定 `taskType`，从历史模式库中找到最相似的成功模式：

```
// SuggestChain 建议最优执行链路
func SuggestChain(ctx, taskType string) (ExecutionChain, float64, error):

    // Step 1: 精确匹配——相同 taskType 且样本充足
    exactPatterns = QueryPatterns(taskType, MinSamples=5)
    if len(exactPatterns) > 0:
        best = SelectBestPattern(exactPatterns)
        return best.ToExecutionChain(), best.AvgScore, nil

    // Step 2: 语义相似匹配——taskType 相近的模式
    // 通过预定义的 taskType 相似度矩阵（或 embedding 距离）找到相近任务类型
    similarTypes = FindSimilarTaskTypes(taskType, topK=3)
    for st in similarTypes:
        patterns = QueryPatterns(st, MinSamples=3)
        if len(patterns) > 0:
            // 打折使用（相似度作为权重折扣）
            similarity = TaskTypeSimilarity(taskType, st)
            best = SelectBestPattern(patterns)
            return best.ToExecutionChain(), best.AvgScore * similarity, nil

    // Step 3: 兜底——返回默认链路（单节点，基于 L1 排名选 brain）
    topBrain = RankBrainsForTask(ctx, taskType)[0]
    return SingleNodeChain(topBrain), 0.5, ErrNoHistoricalPattern

// SelectBestPattern 在多个候选模式中选最优
func SelectBestPattern(patterns []ChainPattern) ChainPattern:
    // 综合评分 = 成功率 * 0.5 + 平均分 * 0.3 + 置信度 * 0.2
    // 置信度 = WilsonConfidence(SampleCount)
    return MaxBy(patterns, p => 0.5*p.SuccessRate + 0.3*p.AvgScore + 0.2*WilsonConfidence(p.SampleCount))
```

### 2.6 探索与利用平衡（ε-贪心策略）

```go
// ExplorationConfig 探索配置
type ExplorationConfig struct {
    // ε-贪心中的 ε：以此概率选择随机链路（探索）
    // 1-ε 的概率选择当前最优链路（利用）
    Epsilon float64 // 默认 0.15（15% 探索）

    // ε 衰减：随样本量增加，探索比例逐渐降低
    EpsilonDecay   float64 // 默认 0.99（每次更新后 ε *= 0.99）
    EpsilonMinimum float64 // 最小探索率，默认 0.05（永远保持 5% 探索）

    // 置信上界（UCB）修正：低置信度模式额外加分，鼓励探索不确定区域
    UCBConstant float64 // 默认 1.0
}

// ChainSelector 综合探索策略的链路选择器
func SelectChain(patterns []ChainPattern, config ExplorationConfig, totalEpisodes int) ChainPattern:

    // UCB 分数 = AvgScore + UCBConstant * sqrt(ln(totalEpisodes) / SampleCount)
    // UCB 让采样少的模式得到额外加分，自然实现探索
    for p in patterns:
        ucb_bonus = config.UCBConstant * sqrt(ln(float64(totalEpisodes)) / float64(p.SampleCount))
        p.UCBScore = p.AvgScore + ucb_bonus

    // ε-贪心选择
    if rand.Float64() < config.Epsilon:
        // 探索：从所有模式中随机选（但倾向于低样本量的模式）
        // 权重 = 1 / SampleCount（样本越少越可能被选中探索）
        return WeightedRandomChoice(patterns, w => 1.0/float64(w.SampleCount))
    else:
        // 利用：选 UCB 分最高的
        return MaxBy(patterns, p => p.UCBScore)
```

## 3. L3 用户级学习算法

### 3.1 核心问题

L3 回答：**这个用户是谁，他/她的工作风格和偏好是什么，怎么更好地服务他/她？**

L3 是最细粒度也最个性化的学习层。它不改变"系统能做什么"，而是改变"系统怎么和用户协作"。

### 3.4 偏好更新算法

#### 3.4.1 EWMA 偏好更新

```go
// UpdatePreference 根据信号更新某个偏好维度
func UpdatePreference(current float64, signal PreferenceSignal) float64:
    // 将信号转换为目标值（target）和学习率（lr）
    target, lr = SignalToUpdate(signal)

    // EWMA 更新：current 向 target 移动，步长 = lr * signal.Strength
    step   = lr * signal.Strength
    updated = current + step * (target - current)

    // 边界约束
    return Clamp(updated, 0.0, 1.0)

// SignalToUpdate 将信号类型转换为具体更新参数
func SignalToUpdate(signal PreferenceSignal) (target float64, lr float64):
    switch signal.Kind:
    case "accept":
        return current, 0.0  // 接受 = 当前偏好是对的，无需更新（但记录正向反馈）
    case "reject":
        return 1.0 - current, 0.15  // 拒绝 = 往反方向移动，步长较大
    case "modify":
        return 0.5, 0.05  // 修改 = 轻微调整，步长小（方向由 AnalyzeModify 确定）
    case "expand":
        return 1.0, 0.10  // 要求更详细 = DetailLevel 往 1 方向移动
    case "summarize":
        return 0.0, 0.10  // 要求总结 = DetailLevel 往 0 方向移动
    case "silence":
        return current, 0.01  // 沉默 = 极小正向更新（惰性学习）
    case "retry":
        return 1.0 - current, 0.20  // 重试 = 强烈负面信号
```

#### 3.4.2 时间衰减（偏好随时间变化的处理）

用户偏好不是永久固定的。用三种机制处理偏好变化：

```go
// TemporalDecayConfig 时间衰减配置
type TemporalDecayConfig struct {
    // 向均值回归：超过 HalfLife 没有信号，偏好向 0.5 收缩
    // 不同维度有不同半衰期
    HalfLifeByDimension map[string]time.Duration

    // 急剧转变检测：短期内连续出现反向信号，触发快速学习
    ReversalThreshold float64  // 默认 3 次连续反向信号
    ReversalBoostLR   float64  // 急剧转变时的学习率倍数（默认 3x）
}

// 各偏好维度的半衰期（无信号情况下偏好向 0.5 衰减的半衰期）
var DefaultHalfLife = map[string]time.Duration{
    "detail_level":              30 * 24 * time.Hour,  // 30 天（稳定偏好）
    "speed_preference":          7 * 24 * time.Hour,   // 7 天（可能随项目阶段变化）
    "automation_level":          60 * 24 * time.Hour,  // 60 天（最稳定）
    "communication.directness":  90 * 24 * time.Hour,  // 个人风格，高度稳定
    "tool_preference":           14 * 24 * time.Hour,  // 工具偏好相对易变
}

// ApplyTemporalDecay 对所有偏好维度应用时间衰减
func ApplyTemporalDecay(profile UserProfile, now time.Time):
    elapsed = now.Sub(profile.UpdatedAt)
    for dim, halfLife in DefaultHalfLife:
        // 衰减函数：value = 0.5 + (value - 0.5) * 2^(-elapsed/halfLife)
        decayFactor = pow(2.0, -elapsed.Hours() / halfLife.Hours())
        current = GetDimension(profile, dim)
        decayed = 0.5 + (current - 0.5) * decayFactor
        SetDimension(profile, dim, decayed)
```

**两种衰减模式**：

| 场景 | 衰减模式 | 原因 |
|------|----------|------|
| 长期无使用（>30天） | 向 0.5 回归 | 用户可能已改变，不要强持旧偏好 |
| 活跃使用期 | 不衰减，实时 EWMA 更新 | 信号已足够，不需要人工衰减 |
| 连续反向信号（≥3次） | 快速学习（3x LR） | 用户偏好发生了转变，快速适应 |

```go
    hour = session.StartedAt.UTC().Hour()
    day  = session.StartedAt.Weekday()

    // 更新活跃小时（EWMA 频率计数）
    profile.TemporalPattern.HourlyTaskRate[hour] =
        0.9 * profile.TemporalPattern.HourlyTaskRate[hour] + 0.1 * float64(session.TaskCount)

    // 更新活跃周（滑动窗口计数，保留最近 12 周）
    profile.TemporalPattern.ActiveWeekdays = UpdateWeekdayStats(day, profile.TemporalPattern.ActiveWeekdays)

    // 更新会话特征（EWMA）
    profile.TemporalPattern.AvgSessionDuration =
        EWMADuration(profile.TemporalPattern.AvgSessionDuration, session.Duration, alpha=0.1)
```

### 3.5 GetUserProfile 算法

```go
// GetUserProfile 返回当前用户画像，应用时间衰减后返回
func GetUserProfile(ctx, userID string) (UserProfile, error):
    profile = LoadFromStorage(userID)

        return DefaultProfile(userID), nil  // 新用户返回全 0.5 的默认 profile

    // 应用时间衰减（距上次更新越久，偏好越向中性收缩）
    ApplyTemporalDecay(&profile, now())

    return profile, nil
```

## 6. 评估指标：如何衡量学习系统本身的效果

### 6.1 总体评估框架

学习系统的效果评估分三个维度：**有效性**（学到了正确的东西）、**收敛性**（学得足够快）、**稳定性**（不会过拟合或振荡）。

### 6.2 L1 评估指标

```go
// L1LearningMetrics L1 学习效果评估指标
type L1LearningMetrics struct {
    // 有效性指标
    DelegateSuccessRateBefore float64  // 引入 L1 前的委托成功率（基线）
    DelegateSuccessRateAfter  float64  // 引入 L1 后的委托成功率
    RankingAccuracy           float64  // 排名第一的 brain 实际也是最优的比例
    RegretRate                float64  // 选错 brain 导致的性能损失（0=最优选择，1=最差选择）

    // 收敛性指标
    SamplesToColdStartExit    int      // 新 brain 从冷启动到稳定排名所需样本数
    EWMAStabilizationEpisodes int      // EWMA 评分达到稳定（标准差 <0.05）所需样本数

    // 稳定性指标
    RankingFlipRate           float64  // 连续两次排名第一的 brain 不同的比例（越低越稳定）
    ScoreVariance             float64  // 各 brain 评分方差（过大 = 不稳定）
}
```

**评估方法**：A/B 对比——同类任务随机一半走 L1 排名委托，一半走随机委托（对照组），对比最终 DelegateResult.FinalScore 均值。
