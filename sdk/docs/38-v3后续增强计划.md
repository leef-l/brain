## Phase F：质量加固

### F-1: protocol frameWriter race 修复
- **优先级**：P0（已知 bug）
- **问题**：`sdk/protocol/frame.go` 中 `frameWriter` 的 channel close 与 send 存在竞态。`bidirRPC.readLoop()` 关闭 channel 时，`bidirRPC.Notify()` 可能正在发送
- **影响**：`-race` 检测间歇性失败（`TestOrchestratorDelegate_ProcessRunner_RealBrainCodeBinary`）
- **修复方案**：`frameWriter.WriteFrame()` 在发送前检查 closed 标志，或用 mutex 保护 close/send
- **涉及文件**：`sdk/protocol/frame.go`（~20 行改动）
- **预估**：0.5 天

## Phase G：安全加固

### G-1: 远程 brain 多租户 workdir 隔离
- **优先级**：P0（团队共享模式必须）
- **问题**：远程 brain 共享模式下，用户 A 的请求可能操作用户 B 的文件
- **方案**：
  - `ExecuteRequest` 必须携带 `workdir`，远程 brain 强制 chroot/sandbox
  - `sidecar.ListenAndServe` 模式下拒绝无 workdir 的 execute 请求
  - 工具执行前验证路径在 workdir 内
- **涉及文件**：`sdk/sidecar/serve.go`、各 brain 的 `buildRegistry`
- **预估**：2 天

### G-2: 远程连接 mTLS
- **优先级**：P1
- **内容**：
  - `RemoteBrainConfig` 增加 `CAPath`、`ClientCertPath`、`ClientKeyPath`
  - `ListenAndServe` 支持 `--tls-cert` / `--tls-key` 参数
  - 双向证书认证替代 Bearer token
- **涉及文件**：`sdk/kernel/remote.go`、`sdk/sidecar/serve.go`
- **预估**：2 天

### G-3: 审计日志导出
- **优先级**：P2

- **涉及文件**：`cmd/brain/command/` 新增 audit 子命令

## Phase H：性能优化

### H-1: gRPC 传输层
- **优先级**：P2
- **动机**：WebSocket JSON 编解码开销大，高频调用场景（如 quant 实时数据）需要更低延迟
- **方案**：
  - 新增 `sdk/protocol/brain.proto` protobuf 定义
  - 实现 `GRPCFrameReader/Writer` 适配 BidirRPC
  - `--listen` 支持 `--grpc` 标志切换协议
  - 保持 WebSocket 作为默认（兼容性好）
- **依赖**：`google.golang.org/grpc`、`google.golang.org/protobuf`
- **预估**：4-5 天

### H-2: 连接池复用与负载均衡
- **优先级**：P2

  - round-robin / least-connections / weighted 策略
  - `DiscoverableBrainPool` 集成多端点
- **涉及文件**：`sdk/kernel/discovery.go`、`sdk/kernel/remote.go`
- **预估**：2 天

### H-3: SQLite 性能调优
- **优先级**：P3
- **内容**：

  - 查询索引优化（EXPLAIN QUERY PLAN 分析）

  - WAL checkpoint 策略调优
- **涉及文件**：`sdk/persistence/driver_sqlite.go`
- **预估**：1.5 天

## Phase J：交互与体验

### J-1: serve 模式审批通道
- **优先级**：P3（当前无产品需要，等上层产品真正需要 Brain 层审批时再做）
- **背景**：chat 有 6 种 permission mode（plan/default/accept-edits/auto/restricted/bypass），但 serve 下 `plan` 和 `accept-edits` 无法使用——没有交互通道让 HTTP 调用方确认审批
- **方案**：
  - 创建 run 时可指定 `mode: "plan"`
  - run 执行到需要审批时暂停，状态变为 `waiting_approval`
  - WebSocket/SSE 推送审批请求事件给调用方
  - 调用方 POST `/v1/runs/{id}/approve` 或 `/v1/runs/{id}/reject`
  - run 继续执行或取消
- **为什么现在不做**：EasyMVP 等上层产品在自己的领域层做审批（方案审核、任务合同），发给 Brain 执行时已经过了审批，Brain 层用 `auto` 即可
- **触发条件**：当有产品需要"Brain 执行过程中动态审批某个工具调用"时再实现
- **涉及文件**：`cmd/brain/cmd_serve.go`、`sdk/kernel/execution.go`（新增 `StateWaitingApproval`）、`sdk/loop/runner.go`（暂停/恢复机制）
- **预估**：3-4 天

### J-2: chat 输入体验升级
- **优先级**：P3
- **问题**：当前用 `bufio.Scanner`，没有行编辑、历史翻阅、Tab 补全、多行输入
- **方案**：
  - 替换为 `github.com/charmbracelet/bubbletea` 或 `github.com/chzyer/readline`
  - 支持上下键翻历史、左右移动光标
  - 支持多行输入（Shift+Enter 或反引号块）
  - 支持斜杠命令 Tab 补全
  - 可选：Markdown 渲染（代码块高亮）
- **预估**：3-4 天

### J-3: chat 会话持久化
- **优先级**：P3
- **问题**：退出 chat 后对话历史丢失
- **方案**：
  - 会话自动保存到 `~/.brain/sessions/{id}.json`
  - `brain chat --resume` 恢复上次会话
  - `brain chat --session <id>` 恢复指定会话
- **预估**：1.5 天
