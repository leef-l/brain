> **状态**：v1 · 2026-04-16
> **上位规格**：[32-v3-Brain架构.md](./32-v3-Brain架构.md)
> **适用范围**：`/www/wwwroot/project/brain-v3` 当前代码库向 v3 架构的平滑迁移

## 0. 前言：衔接当前仓库的核心矛盾

v3 架构文档（32号）定义了四个正交概念：**TaskExecution + Capability Lease + Dispatch Policy + Flow Edge**。
当前代码库的关键矛盾只有一个：

> **每个 `brain serve` 的 Run 都在 `executeRun` 里各自创建一个 `Orchestrator`，导致每个 run 都 fork 一套独立的 sidecar 进程。6 个并发 run = 6 × N 个 sidecar 进程。这是 Phase A 必须修的架构缺陷，不是优化。**

其余改动（LeaseManager、统一 EventBus、Dashboard）都是增量新增，对现有 API 无破坏性影响。

---

## 1. 重构顺序（依赖关系严格排序）

```
[基础层，无前置依赖]
  Step 1: 定义 BrainPool 接口 + 从 Orchestrator 抽取 active pool 逻辑
  Step 2: 定义 EventBus 接口 + 替换 runtimeaudit.Sink 的调用点

[依赖 Step 1]
  Step 3: 将全局 BrainPool 注入 cmd_serve.go 的 runServe（修复 per-run Orchestrator 缺陷）
  Step 4: 将 Orchestrator 瘦身为纯路由层（不再持有 active pool）

[依赖 Step 2]
  Step 5: EventBus SSE 端点 GET /v1/executions/{id}/events

[依赖 Step 3 + Step 4]
  Step 6: TaskExecution 模型（重命名 Run → TaskExecution，添加 policy 字段）
  Step 7: HTTP API 统一（/v1/executions 作为规范入口，/v1/runs 作为别名）

[依赖 Step 6]
  Step 8: LeaseManager + ToolConcurrencySpec（turn_executor 调度内核）
  Step 9: Dashboard API + 前端 SPA（/v1/dashboard/*）

[依赖 Step 8]
  Step 10: Dispatch Policy（BatchPlanner，冲突分析 + AcquireSet）

[Phase B 入口，依赖 Step 10]
  Step 11: Context Engine（可插拔上下文装配层）
  Step 12: MCP-backed Runtime 参考实现
  Step 13: 语义审批分级（5 级审批类，替换静态白名单）

[依赖 Step 6，Phase A 收尾]
  Step 14: Brain Manifest 落地（4 个基础脑 + 3 个业务脑 brain.json）

[依赖 Step 14，Phase B 增强]
  Step 15: 工具 ToolConcurrencySpec 并发约束声明
  Step 16: BrainLearner L0 标准接口迁移
  Step 17: Dashboard 前端 SPA（embed.FS 嵌入）
```

### Step 1：定义 BrainPool 接口

**涉及文件**

| 操作 | 文件 |
|------|------|
| 新增 | `sdk/kernel/pool.go` |
| 修改 | `sdk/kernel/orchestrator.go`（抽取 active pool 逻辑） |
| 修改 | `sdk/kernel/kernel.go`（添加 `BrainPool` 字段） |

**关键类型变化**

```go
// sdk/kernel/pool.go（新增）
type BrainPool interface {
    GetBrain(ctx context.Context, kind agent.Kind) (protocol.BidirRPC, error)
    Status() map[agent.Kind]BrainStatus
    AutoStart(ctx context.Context)
    Shutdown(ctx context.Context) error
}

// ProcessBrainPool 是用 ProcessRunner 实现的 BrainPool
// 将 Orchestrator.active map + getOrStartSidecar + waitForSidecar + removeSidecar
// 逐字移入此处，不做逻辑改动
type ProcessBrainPool struct {
    runner      BrainRunner
    binResolver func(agent.Kind) (string, error)
    available   map[agent.Kind]bool
    mu          sync.Mutex
    active      map[agent.Kind]agent.Agent
    registrations map[agent.Kind]*BrainRegistration
}
```

**Orchestrator 保留**：仅 `Delegate`、`CallTool`、`HandleSubtaskDelegate`、`HandleSpecialistCallToolFrom`，内部调 `pool.GetBrain()`。

**API 兼容性**：无破坏，`BrainPool` 是内部类型，对 HTTP 端点无影响。

### Step 3：全局 BrainPool 注入 cmd_serve.go

这是**修复 per-run Orchestrator 架构缺陷**的核心步骤。

**涉及文件**

| 操作 | 文件 |
|------|------|
| 修改 | `cmd/brain/cmd_serve.go`（runServe + executeRun） |
| 修改 | `cmd/brain/helpers.go`（buildOrchestrator 改为 buildBrainPool） |

**关键改动**

```go
// 现在（每个 run 各建 Orchestrator）：
func executeRun(...) {
    var orch *kernel.Orchestrator
    if req.Brain == "central" {
        orch = buildOrchestrator(...)   // ← 每次 fork 新 sidecar
    }
    defer orch.Shutdown(...)
}

// 改后（全局 BrainPool，run 之间共享）：
func runServe(...) {
    pool := buildBrainPool(cfg)         // ← 全局一个，serve 生命周期持有
    pool.AutoStart(serveCtx)
    defer pool.Shutdown(...)
    // pool 注入 mgr
    mgr := &runManager{pool: pool, ...}
}

func executeRun(ctx, entry, mgr, ...) {
    orch := kernel.NewOrchestrator(mgr.pool, llmProxy)  // ← 复用全局 pool，不 fork 进程
    ...
}
```

**API 兼容性**：`POST /v1/runs`、`GET /v1/runs/{id}`、`DELETE /v1/runs/{id}` 行为完全不变，只是 sidecar 进程从"每 run 一套"变为"全局共享"。

### Step 6：TaskExecution 模型

**涉及文件**

| 操作 | 文件 |
|------|------|
| 新增 | `sdk/kernel/execution.go` |
| 修改 | `sdk/loop/run.go`（`Run` 添加 policy 字段，保持向后兼容） |

**关键类型变化**

```go
// sdk/kernel/execution.go（新增）
type TaskExecution struct {
    ID          string
    ParentID    string
    BrainID     string
    Budget      loop.Budget
    Mode        ExecutionMode
    Lifecycle   LifecyclePolicy
    Restart     RestartPolicy
    Status      ExecutionStatus
    CreatedAt   time.Time
    EndedAt     *time.Time

    // 内部：关联的 loop.Run（不是新类型，是现有 Run 的包装）
    run    *loop.Run

}

type ExecutionMode   string  // "interactive" / "background"
type LifecyclePolicy string  // "oneshot" / "daemon" / "watch"
type RestartPolicy  string   // "never" / "on-failure" / "always"
```

**迁移策略**：`loop.Run` 不删除，`TaskExecution` 通过 `run *loop.Run` 内部持有它。
`runEntry`（cmd_serve.go）逐步迁移为 `TaskExecution`，两者可以共存一个版本周期。

### Step 8：LeaseManager + ToolConcurrencySpec

**涉及文件**

| 操作 | 文件 |
|------|------|
| 新增 | `sdk/kernel/lease.go` |
| 修改 | `sdk/tool/schema.go`（添加 `Concurrency *ToolConcurrencySpec` 字段） |
| 修改 | `sdk/loop/turn_executor.go`（预留 LeaseManager 注入位，Phase B 时填充） |

**关键类型变化**

```go
// sdk/kernel/lease.go（新增）
type AccessMode  string  // "shared-read" / "shared-write-append" / "exclusive-write" / "exclusive-session"
type LeaseScope  string  // "turn" / "task" / "daemon"

type LeaseRequest struct {
    Capability  string
    ResourceKey string
    AccessMode  AccessMode
    Scope       LeaseScope
}

type Lease interface {
    Release()
}

type LeaseManager interface {
    AcquireSet(ctx context.Context, reqs []LeaseRequest) ([]Lease, error)
    ReleaseAll(leases []Lease)
}

// sdk/tool/schema.go 新增字段（可选，nil 表示无并发约束）
type ToolConcurrencySpec struct {
    Capability          string
    ResourceKeyTemplate string
    AccessMode          AccessMode
    Scope               LeaseScope
    AcquireTimeout      time.Duration
}
```

**Phase A 实现**：`MemLeaseManager`（内存实现），AcquireSet 按 canonical order（Capability+ResourceKey 字典序）排序后依次申请，任意失败则全部释放 + 退避重试。

- `go test ./sdk/kernel/lease_test.go` 覆盖以下场景：
  - 并发 SharedRead 不冲突

  - ExclusiveWrite 不同 ResourceKey 不冲突
  - AcquireSet 失败后释放已获取的 lease
  - Canonical order 防止同等条件下的排列差异

## 6. 风险点和回退方案

### 风险 1：PR-A2 全局 BrainPool 引入共享状态竞争

**风险描述**：多个并发 run 共享同一个 `BrainPool`，`getOrStartSidecar` 的 nil-placeholder 并发机制是否在高并发下仍然正确？

**缓解措施**：
- PR-A1 的 `ProcessBrainPool` 直接复制现有 `Orchestrator` 中已经过测试的并发逻辑，不做改动
- PR-A1 必须包含 `go test -race -count=5 ./sdk/kernel/...` 在并发测试下通过

**回退方案**：
- `cmd_serve.go` 保留 `--no-pool` flag（默认关闭），开启时退回到 per-run Orchestrator 行为
- 回退只需改一行：`if *noPool { orch = buildOrchestrator(...) } else { orch = kernel.NewOrchestratorWithPool(mgr.pool, ...) }`

### 风险 2：`runEntry` → `TaskExecution` 迁移破坏持久化格式

**风险描述**：`runtimeStore` 持久化 `runEntry` 的 JSON，新增 policy 字段可能影响旧记录读取。

**缓解措施**：
- `Mode`、`Lifecycle`、`Restart` 字段使用 `omitempty`，旧记录读取时为零值（对应默认行为：interactive + oneshot + never）
- 迁移前在 `runtimeStore.get` 中加 default value 填充逻辑

**回退方案**：字段添加不是 breaking change，旧二进制读取新格式只是忽略未知字段，新二进制读取旧格式只是得到零值默认。可随时回滚到旧二进制。

### 风险 3：AcquireSet 死锁（LeaseManager）

**风险描述**：Phase B 的 BatchPlanner + AcquireSet 如果 canonical order 实现有 bug，可能导致跨 batch 死锁。

**缓解措施**：
- PR-A6 只是骨架，`turn_executor` 实际接入在 PR-B1，中间有完整测试周期
- `MemLeaseManager` 内置 `AcquireTimeout`（默认 10s），超时后返回 `ErrAcquireTimeout`，不会永久阻塞
- canonical order 是纯函数，排序逻辑单独单测

**回退方案**：
- `BatchPlanner` 有 `Disabled bool` flag；开启时退回串行执行（当前 `Runner.dispatchTools` 行为）
- 单个 `TurnExecutor.BatchPlanner == nil` 时自动退回串行

### 风险 4：Dashboard 前端 SPA 拖延 Phase A 进度

**风险描述**：前端开发耗时不可控，可能阻塞 PR-A7。

**缓解措施**：
- Dashboard API（`/v1/dashboard/overview` 等）是纯后端工作（PR-A7 后端部分 = 0.5 人天）
- 前端 SPA 单独拆出（PR-A7b），不阻塞主链路
- 即使没有前端，`curl /v1/dashboard/brains | jq` 也可以给运维使用

## 7. Step 14：Brain Manifest 落地

> **状态**：2026-04-17 新增
> **优先级**：P0（Phase A 收尾，所有 v3 能力匹配/学习/审计依赖 manifest 声明）
> **依赖**：Step 6（TaskExecution 模型已就绪）

### 7.1 背景

v3 内核的 CapabilityMatcher、LearningEngine、SemanticApprover 已在代码中完整实现，
但因为 7 个 brain 都没有 `brain.json` manifest 文件，这些模块处于**空转状态**：
- CapabilityIndex 无任何 brain 注册 → 匹配算法无候选
- Orchestrator 无法从 manifest 读取 policy.approval_class → SemanticApprover 只能走启发式兜底
- BrainPool 无法从 manifest 推导进程策略（shared-service / exclusive / ephemeral）

### 7.2 交付物

| 优先级 | Brain | 文件路径 | 说明 |
|--------|-------|---------|------|
| P0-1 | central | `central/brain.json` | 协调者，不是 sidecar，runtime.type = native |
| P0-2 | code | `brains/code/brain.json` | 文件读写 + 命令执行，ephemeral-worker |
| P0-3 | browser | `brains/browser/brain.json` | 浏览器操控，exclusive-session |
| P0-4 | verifier | `brains/verifier/brain.json` | 只读验证，ephemeral-worker |
| P1-1 | data | `brains/data/brain.json` | 行情数据，shared-service |
| P1-2 | quant | `brains/quant/brain.json` | 策略执行，shared-service |
| P1-3 | fault | `brains/fault/brain.json` | 混沌测试，ephemeral-worker |

### 7.3 Manifest 字段规范

每个 brain.json 必须包含 7 个核心字段：
```json
{
  "schema_version": 1,
  "kind": "<agent.Kind>",
  "name": "<显示名称>",
  "brain_version": "0.1.0",
  "capabilities": ["<capability_tag>", ...],
  "runtime": { "type": "native", "entrypoint": "<sidecar_binary>" },
  "policy": { "approval_class": "<readonly|workspace-write|exec-capable|control-plane>" }
}
```

### 7.4 各脑 Capability 标签设计

| Brain | Capabilities | 审批等级 | 进程策略 |
|-------|-------------|---------|---------|
| central | `domain.orchestration`, `mode.interactive`, `mode.background` | control-plane | — (非 sidecar) |
| code | `file.read`, `file.write`, `file.delete`, `code.search`, `code.execute` | exec-capable | ephemeral-worker |
| browser | `web.navigate`, `web.interact`, `web.screenshot`, `web.evaluate` | external-network | exclusive-session |
| verifier | `verify.read`, `verify.test`, `verify.check`, `verify.browse` | readonly | ephemeral-worker |
| data | `market.candles`, `market.snapshot`, `market.features`, `market.replay` | readonly | shared-service |
| quant | `trading.portfolio`, `trading.risk`, `trading.execute`, `trading.learn` | control-plane | shared-service |
| fault | `chaos.inject_error`, `chaos.inject_latency`, `chaos.kill_process` | exec-capable | ephemeral-worker |
