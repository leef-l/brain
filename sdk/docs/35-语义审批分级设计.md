# 35. §7.3 语义审批分级设计

> **⚠️ 类型名差异（2026-04-24）：** 实际代码用 `ApprovalClass`（string 类型），非文档的 `ApprovalLevel`（int 类型）。常量名：`ApprovalReadonly`（非 `ApprovalLevelReadonly`）。五级语义一致。

## 0. 设计背景与目标

### 0.1 现状问题

当前 `SpecialistToolCallAuthorizer`（`sdk/kernel/specialist_tool_policy.go`）是一个**静态白名单**：

```
调用方 kind × 目标 kind × 工具前缀 → 允许 / 拒绝
```

这个设计有四个结构性缺陷：

1. **无法感知操作语义**：`data.get_candles`（只读）和 `quant.place_order`（有副作用）被同等对待，都是"在白名单里就放行"
2. **与执行模式无关**：`interactive` 模式（用户在看）和 `background` 模式（无人值守）使用相同门禁，不合理
3. **无审计上下文**：放行即放行，没有操作记录可供追溯
4. **扩展性差**：添加新 brain 或新工具，必须修改 Go 代码并重新编译

### 0.2 v3 目标

用**五级语义审批系统**替换静态白名单：

| 等级 | 名称 | 含义 | 默认审批方式 |
|------|------|------|------------|
| L0 | `readonly` | 只读操作，无副作用 | 自动通过 |
| L1 | `workspace-write` | 工作区文件修改（本地沙箱内） | 跟随当前 ExecutionMode |
| L2 | `exec-capable` | 执行外部命令或子进程 | 需要确认 |
| L3 | `control-plane` | 系统控制操作（进程管理、配置变更、金融下单） | 需要确认 + 审计记录 |
| L4 | `external-network` | 外部网络请求（HTTP、WebSocket、第三方 API） | 需要确认 + 审计记录 + 可选人工审批 |

## 1. 核心类型定义

```go
// sdk/kernel/approval.go

package kernel

import (
    "context"
    "time"

    "github.com/leef-l/brain/sdk/agent"
)

// ApprovalLevel 是工具调用的语义审批等级。
// 等级数值越大，要求越严格。
type ApprovalLevel int

const (
    // L0: 只读操作，自动通过。
    // 示例：data.get_candles, data.get_snapshot, repo.read_file
    ApprovalLevelReadonly      ApprovalLevel = 0

    // L1: 工作区写操作（本地沙箱内），跟随 ExecutionMode。
    // 示例：code.write_file, code.patch_file, browser.save_screenshot
    ApprovalLevelWorkspaceWrite ApprovalLevel = 1

    // L2: 执行外部命令，需要确认。
    // 示例：code.execute_shell, code.run_tests, fault.run_diagnostic
    ApprovalLevelExecCapable   ApprovalLevel = 2

    // L3: 控制面操作，需要确认 + 审计。
    // 示例：quant.place_order, quant.cancel_order, brain.restart_sidecar
    ApprovalLevelControlPlane  ApprovalLevel = 3

    // L4: 外部网络请求，需要确认 + 审计 + 可选人工审批。
    // 示例：browser.navigate, browser.form_submit, fetch.post_webhook
    ApprovalLevelExternalNetwork ApprovalLevel = 4
)

// approvalLevelNames 供日志和序列化使用。
var approvalLevelNames = map[ApprovalLevel]string{
    ApprovalLevelReadonly:       "L0:readonly",
    ApprovalLevelWorkspaceWrite: "L1:workspace-write",
    ApprovalLevelExecCapable:    "L2:exec-capable",
    ApprovalLevelControlPlane:   "L3:control-plane",
    ApprovalLevelExternalNetwork: "L4:external-network",
}

func (l ApprovalLevel) String() string {
    if name, ok := approvalLevelNames[l]; ok {
        return name
    }
    return "unknown"
}

// ApprovalRequest 是一次工具调用的审批请求。
// 由 SemanticApprover 的调用者填充，在审批决策树中流转。
type ApprovalRequest struct {
    // 调用链身份
    CallerKind agent.Kind // 发起调用的 brain kind
    TargetKind agent.Kind // 被调用的目标 brain kind
    ToolName   string     // 完整工具名，例如 "quant.place_order"

    // 语义信息
    Level      ApprovalLevel // 从 tool schema 推导出的审批等级
    InputJSON  []byte        // LLM 传入的原始参数（审计用）

    // 执行上下文
    Mode       ExecutionMode // interactive / background
    TaskID     string        // 所属 TaskExecution ID（审计用）
    TurnIndex  int           // 当前 turn 编号（审计用）
}

// ApprovalDecision 是审批结论。
type ApprovalDecision struct {
    Approved  bool
    Reason    string        // 拒绝时的原因，通过时可为空
    AuditID   string        // 审计记录 ID（L3/L4 有值）
    Timestamp time.Time
}

// ExecutionMode 标识当前任务的执行模式。
// 与 TaskExecution 状态机的 Mode 字段保持一致（§7.7.1）。
type ExecutionMode string

const (
    ModeInteractive ExecutionMode = "interactive" // 有用户在场
    ModeBackground  ExecutionMode = "background"  // 无人值守，后台运行
)
```

## 2. 审批决策树

### 2.1 从 tool schema 推导 ApprovalLevel 的算法

工具 schema（`sdk/tool/schema.go`）新增 `ApprovalClass` 字段：

```go
// sdk/tool/schema.go 新增字段

// ApprovalClass 是工具的语义审批类别声明。
// 工具作者在注册工具时填写；为空时 Kernel 根据工具名前缀做启发式推断。
//
// 合法值："readonly" / "workspace-write" / "exec-capable" / "control-plane" / "external-network"
//
// 示例：
//   ApprovalClass: "readonly"          // data.get_candles
//   ApprovalClass: "control-plane"     // quant.place_order
//   ApprovalClass: "external-network"  // browser.navigate
type ApprovalClass string

const (
    ApprovalClassReadonly        ApprovalClass = "readonly"
    ApprovalClassWorkspaceWrite  ApprovalClass = "workspace-write"
    ApprovalClassExecCapable     ApprovalClass = "exec-capable"
    ApprovalClassControlPlane    ApprovalClass = "control-plane"
    ApprovalClassExternalNetwork ApprovalClass = "external-network"
)
```

### 2.2 推导算法实现

```go
// sdk/kernel/approval.go

// resolveApprovalLevel 从工具 schema 推导审批等级。
//
// 决策优先级（高到低）：
//   1. tool.ApprovalClass 字段（工具显式声明，最高优先）
//   2. manifest policy.min_approval_level（brain 级别最低门槛，向上取整）
//   3. brain Capability 标签推断（见 35-BrainCapability标签与匹配算法 §6.3）
//   4. 工具名前缀启发式规则（兜底）
func resolveApprovalLevel(
    toolName string,
    toolApprovalClass ApprovalClass,
    manifestMinLevel ApprovalLevel,
    brainCapabilities []string, // Brain 的 Capability 标签列表（可选）
) ApprovalLevel {

    // Step 1：工具 schema 显式声明
    if toolApprovalClass != "" {
        declared := classToLevel(toolApprovalClass)
        // Step 2：与 manifest policy 取较大值（manifest 只升不降）
        return maxLevel(declared, manifestMinLevel)
    }

    // Step 3：从 Brain Capability 标签推断
    // 当工具未显式声明 ApprovalClass 且 manifest 未设 min_approval_level 时，
    // 根据 Brain 的 Capability 类型做兜底推断。
    //
    // 映射规则：
    //   function:code_*      → L2 (workspace-write)    代码操作涉及文件写入
    //   function:search_*    → L1 (read-only)          搜索操作只读
    //   function:execute_*   → L3 (control-plane)      执行操作有副作用
    //   domain:finance_*     → L3 (control-plane)      金融领域操作需审慎
    //   resource:network_*   → L4 (external-network)   网络访问需最高审批
    //   mode:background      → 基础等级 +1             后台执行用户无法实时监控
    if len(brainCapabilities) > 0 {
        capLevel := inferLevelByCapability(brainCapabilities)
        if capLevel > ApprovalLevelUnset {
            return maxLevel(capLevel, manifestMinLevel)
        }
    }

    // Step 4：启发式推断（未声明 ApprovalClass 时的最终兜底）
    inferred := inferLevelByName(toolName)
    return maxLevel(inferred, manifestMinLevel)
}

// classToLevel 将字符串 ApprovalClass 转换为 ApprovalLevel。
func classToLevel(class ApprovalClass) ApprovalLevel {
    switch class {
    case ApprovalClassReadonly:
        return ApprovalLevelReadonly
    case ApprovalClassWorkspaceWrite:
        return ApprovalLevelWorkspaceWrite
    case ApprovalClassExecCapable:
        return ApprovalLevelExecCapable
    case ApprovalClassControlPlane:
        return ApprovalLevelControlPlane
    case ApprovalClassExternalNetwork:
        return ApprovalLevelExternalNetwork
    default:
        // 未知 class 保守处理：等同于 L2
        return ApprovalLevelExecCapable
    }
}

// inferLevelByName 基于工具名前缀的启发式规则。
// 这是未声明 ApprovalClass 工具的兜底逻辑，有明确优先级。
//
// 规则表（按特殊性从高到低排列，先匹配先生效）：
//
//   前缀模式                    推断等级
//   ──────────────────────────────────────
//   *.place_order               L3 control-plane
//   *.cancel_order              L3 control-plane
//   *.withdraw                  L3 control-plane
//   browser.navigate            L4 external-network
//   browser.form_submit         L4 external-network
//   browser.*                   L4 external-network
//   fetch.*                     L4 external-network
//   *.execute_shell             L2 exec-capable
//   *.run_*                     L2 exec-capable
//   code.write_*                L1 workspace-write
//   code.patch_*                L1 workspace-write
//   data.get_*                  L0 readonly
//   data.list_*                 L0 readonly
//   *.read_*                    L0 readonly
//   *.list_*                    L0 readonly
//   （其他）                    L1 workspace-write（保守默认）
func inferLevelByName(toolName string) ApprovalLevel {
    for _, rule := range heuristicRules {
        if matchToolName(toolName, rule.pattern) {
            return rule.level
        }
    }
    // 未匹配任何规则：保守默认 L1
    return ApprovalLevelWorkspaceWrite
}

// heuristicRules 是启发式规则表，顺序决定优先级。
var heuristicRules = []struct {
    pattern string
    level   ApprovalLevel
}{
    // L3: 金融交易控制面
    {"*.place_order",   ApprovalLevelControlPlane},
    {"*.cancel_order",  ApprovalLevelControlPlane},
    {"*.withdraw",      ApprovalLevelControlPlane},
    {"quant.review_*",  ApprovalLevelControlPlane},

    // L4: 外部网络
    {"browser.*",       ApprovalLevelExternalNetwork},
    {"fetch.*",         ApprovalLevelExternalNetwork},
    {"http.*",          ApprovalLevelExternalNetwork},

    // L2: 执行命令
    {"*.execute_shell", ApprovalLevelExecCapable},
    {"*.run_*",         ApprovalLevelExecCapable},
    {"*.exec_*",        ApprovalLevelExecCapable},

    // L1: 工作区写
    {"code.write_*",    ApprovalLevelWorkspaceWrite},
    {"code.patch_*",    ApprovalLevelWorkspaceWrite},
    {"code.delete_*",   ApprovalLevelWorkspaceWrite},

    // L0: 只读
    {"data.get_*",      ApprovalLevelReadonly},
    {"data.list_*",     ApprovalLevelReadonly},
    {"*.read_*",        ApprovalLevelReadonly},
    {"*.list_*",        ApprovalLevelReadonly},
    {"*.get_*",         ApprovalLevelReadonly},
}

func maxLevel(a, b ApprovalLevel) ApprovalLevel {
    if a > b {
        return a
    }
    return b
}
```

## 3. 与 Brain Manifest policy 的绑定

### 3.1 Manifest policy 扩展字段

在 `33-Brain-Manifest规格.md` §7 的 `policy` 结构中新增两个字段：

```json
{
  "policy": {
    "tool_scope": "delegate.browser",
    "approval_mode": "default",
    "sandbox_profile": "workspace-write",
    "active_tools_profile": "safe",

    "min_approval_level": "workspace-write",
    "require_human_approval_above": "control-plane"
  }
}
```

| 字段 | 类型 | 默认值 | 含义 |
|------|------|--------|------|
| `min_approval_level` | string (ApprovalClass) | `"readonly"` | 该 brain 所有工具的**最低**审批等级下限。即使工具自己声明了更低的等级，也会被提升到此值。 |
| `require_human_approval_above` | string (ApprovalClass) | `""` （不启用）| 超过此等级的操作**必须**等待人工确认，忽略 ExecutionMode 设置。 |

### 3.2 约束逻辑说明

`min_approval_level` 是**只升不降**的约束：

```
最终等级 = max(tool.ApprovalClass 等级, manifest.min_approval_level 等级)
```

**示例**：

- browser brain 的 manifest 声明 `min_approval_level: "external-network"`
- 即使 browser brain 内某个工具声明了 `ApprovalClass: "readonly"`（理论上如 `browser.get_title`）
- 实际审批等级仍然被提升为 L4 external-network

这个设计保证了：**brain 维护者可以通过 manifest 为整个 brain 设置最严格的安全基线，用户无法通过单个工具声明绕过 brain 级别的门禁**。

## 4. 与 LeaseRequest 的映射

### 4.1 审批等级与 LeaseRequest 的关系

`LeaseRequest`（`sdk/kernel/lease.go`，§7.8）控制**并发资源访问**，`ApprovalLevel` 控制**操作权限**。两者正交但存在一个核心交汇点：

> **L3/L4 操作在 AcquireSet 之前必须通过审批门禁**

```
                    ┌──────────────────┐
                    │  BatchPlanner    │
                    │  .Plan()         │
                    └──────┬───────────┘
                           │ 推导 LeaseRequest 集合
                           │
                    ┌──────▼───────────┐
                    │  ApprovalGate    │  ← 新增：AcquireSet 前的审批门禁
                    │  .CheckBatch()   │
                    └──────┬───────────┘
                           │ 全部通过才继续
                           │
                    ┌──────▼───────────┐
                    │  LeaseManager    │
                    │  .AcquireSet()   │
                    └──────────────────┘
```

### 4.2 从 ApprovalLevel 到 AccessMode 的参考映射

| ApprovalLevel | 推荐 AccessMode | 说明 |
|---------------|----------------|------|
| L0 readonly | `shared-read` | 只读，可多并发 |
| L1 workspace-write | `shared-write-append` 或 `exclusive-write` | 取决于具体工具的资源 key |
| L2 exec-capable | `exclusive-write` | 执行命令通常占用独占资源 |
| L3 control-plane | `exclusive-session` | 控制面操作需要最强隔离 |
| L4 external-network | `exclusive-write` | 外部网络请求需要独占网络资源槽位 |

这个映射不是强制的——`ToolConcurrencySpec` 仍然是工具自己声明的。上表只是推荐的最佳实践，在工具没有声明 `ToolConcurrencySpec` 时由 `BatchPlanner` 做默认推断。

### 4.3 ApprovalGate 接口

```go
// sdk/kernel/approval_gate.go

// ApprovalGate 是 AcquireSet 前的批量审批门禁。
// BatchPlanner 在构建 ToolBatch 后、调用 LeaseManager.AcquireSet 前调用。
type ApprovalGate interface {
    // CheckBatch 对一批 tool_call 做批量审批检查。
    // 返回 nil 表示整批通过，否则返回第一个被拒绝的错误。

// 支持链式组合多个 gate（例如：静态矩阵 → 语义审批 → 人工确认）。
type ApprovalGateMiddleware func(next ApprovalGate) ApprovalGate
```

## 5. 审批上下文：ExecutionMode 下的差异

### 5.1 核心策略表

| ApprovalLevel | interactive 模式 | background 模式 |
|---------------|----------------|----------------|
| L0 readonly | 自动通过 | 自动通过 |
| L1 workspace-write | 自动通过 | 自动通过（同 interactive） |
| L2 exec-capable | 提示用户确认 | **自动通过**（background 默认信任） |
| L3 control-plane | 提示用户确认 | **拒绝**（background 禁止控制面操作，除非 manifest 显式授权） |
| L4 external-network | 提示用户确认 | **拒绝**（background 禁止外部网络，除非 manifest 显式授权） |

> **设计理由**：background 模式（无人值守）对 L3/L4 默认拒绝，是为了防止后台任务在用户不知情的情况下执行金融操作或外部网络请求。这是"安全默认"原则的体现。

### 5.2 background 模式的豁免机制

Brain Manifest 可以为 background 模式显式声明豁免：

```json
{
  "policy": {
    "min_approval_level": "control-plane",
    "background_allow_levels": ["control-plane"],
    "background_deny_levels": ["external-network"]
  }
}
```

`background_allow_levels` 仅当 brain 在设计上就是为了后台执行（如 quant brain 的自动交易模式）时才应该声明。

### 5.3 ExecutionMode 感知的审批策略实现

```go
// sdk/kernel/approval.go

// ModeAwareApprover 是感知 ExecutionMode 的审批策略实现。
type ModeAwareApprover struct {
    // ConfirmFunc 在 interactive 模式需要用户确认时调用。
    // 返回 true 表示用户确认，false 表示用户拒绝。
    // background 模式不调用此函数。
    ConfirmFunc func(ctx context.Context, req ApprovalRequest) (bool, error)

    // BackgroundAllowLevels 是 background 模式下允许通过的等级集合。
    // 默认只允许 L0/L1，background 模式的 L2-L4 需要显式豁免。
    BackgroundAllowLevels map[ApprovalLevel]bool
}

// Approve 实现核心审批策略。
func (a *ModeAwareApprover) Approve(ctx context.Context, req ApprovalRequest) (ApprovalDecision, error) {
    // L0: 无条件通过
    if req.Level == ApprovalLevelReadonly {
        return ApprovalDecision{Approved: true, Timestamp: time.Now()}, nil
    }

    switch req.Mode {
    case ModeInteractive:
        return a.approveInteractive(ctx, req)
    case ModeBackground:
        return a.approveBackground(ctx, req)
    default:
        // 未知 mode 保守处理：走 interactive 路径
        return a.approveInteractive(ctx, req)
    }
}

func (a *ModeAwareApprover) approveInteractive(ctx context.Context, req ApprovalRequest) (ApprovalDecision, error) {
    // L1: interactive 模式自动通过
    if req.Level == ApprovalLevelWorkspaceWrite {
        return ApprovalDecision{Approved: true, Timestamp: time.Now()}, nil
    }

    // L2/L3/L4: 请求用户确认
    confirmed, err := a.ConfirmFunc(ctx, req)
    if err != nil {
        return ApprovalDecision{}, err
    }
    return ApprovalDecision{
        Approved:  confirmed,
        Reason:    ifElse(confirmed, "", "用户拒绝"),
        Timestamp: time.Now(),
    }, nil
}

func (a *ModeAwareApprover) approveBackground(ctx context.Context, req ApprovalRequest) (ApprovalDecision, error) {
    // L0/L1: background 自动通过
    if req.Level <= ApprovalLevelWorkspaceWrite {
        return ApprovalDecision{Approved: true, Timestamp: time.Now()}, nil
    }

    // 检查 manifest 豁免
    if a.BackgroundAllowLevels[req.Level] {
        return ApprovalDecision{
            Approved:  true,
            Reason:    "manifest background_allow_levels 豁免",
            Timestamp: time.Now(),
        }, nil
    }

    // L2-L4: background 模式默认拒绝（L2 稍宽松：默认通过，除非被 manifest 限制）
    if req.Level == ApprovalLevelExecCapable {
        // L2 在 background 模式默认通过（后台任务常需要执行命令）
        return ApprovalDecision{Approved: true, Timestamp: time.Now()}, nil
    }

    // L3/L4: background 模式默认拒绝
    return ApprovalDecision{
        Approved:  false,
        Reason:    fmt.Sprintf("background 模式禁止 %s 操作，请切换到 interactive 模式或在 manifest 中声明豁免", req.Level),
        Timestamp: time.Now(),
    }, nil
}
```

## 6. 审批接口定义与中间件设计

### 6.1 核心接口

```go
// sdk/kernel/approval.go

// SemanticApprover 是语义审批的核心接口。
// 替代 SpecialistToolCallAuthorizer，成为跨脑工具调用的唯一授权入口。
type SemanticApprover interface {
    // Approve 对单个工具调用做审批决策。
    // 实现必须：
    //   1. 对 L0 操作立即返回，不阻塞
    //   2. 对 L3/L4 操作在返回前写入审计日志
    //   3. 线程安全（可被并发调用）
    Approve(ctx context.Context, req ApprovalRequest) (ApprovalDecision, error)
}

// SemanticApproverFunc 是函数适配器，方便测试和简单场景使用。
type SemanticApproverFunc func(context.Context, ApprovalRequest) (ApprovalDecision, error)

func (f SemanticApproverFunc) Approve(ctx context.Context, req ApprovalRequest) (ApprovalDecision, error) {
    return f(ctx, req)
}
```

### 6.2 中间件链

```go
// sdk/kernel/approval_middleware.go

// ApproverMiddleware 是 SemanticApprover 的中间件包装类型。
type ApproverMiddleware func(next SemanticApprover) SemanticApprover

// ChainApprovers 将多个中间件按顺序组合成一个 SemanticApprover。
// 执行顺序：middlewares[0] → middlewares[1] → ... → base
//
// 推荐的标准链：
//   1. StaticMatrixMiddleware     （静态授权矩阵：caller×target 白名单，向后兼容层）
//   2. ManifestPolicyMiddleware   （manifest policy：min_approval_level 约束）
//   3. LevelResolverMiddleware    （审批等级推导）
//   4. ModeAwareMiddleware        （ExecutionMode 感知策略）
//   5. AuditMiddleware            （L3/L4 审计记录）
func ChainApprovers(base SemanticApprover, middlewares ...ApproverMiddleware) SemanticApprover {
    result := base
    for i := len(middlewares) - 1; i >= 0; i-- {
        result = middlewares[i](result)
    }
    return result
}

// StaticMatrixMiddleware 是向后兼容层。
// 它将现有 SpecialistToolCallAuthorizer 的静态白名单作为第一道门禁。
// 不在矩阵中的 caller×target 组合，直接拒绝，不进入后续审批层。
func StaticMatrixMiddleware(matrix SpecialistToolCallAuthorizer) ApproverMiddleware {
    return func(next SemanticApprover) SemanticApprover {
        return SemanticApproverFunc(func(ctx context.Context, req ApprovalRequest) (ApprovalDecision, error) {
            // 先做静态矩阵检查
            if err := matrix.AuthorizeSpecialistToolCall(ctx, req.CallerKind, req.TargetKind, req.ToolName); err != nil {
                return ApprovalDecision{
                    Approved: false,
                    Reason:   fmt.Sprintf("静态授权矩阵拒绝：%v", err),
                }, nil
            }
            // 通过后交给下一层
            return next.Approve(ctx, req)
        })
    }
}

// ManifestPolicyMiddleware 将 manifest policy 的约束注入 ApprovalRequest。
// 它查询 BrainRegistration.Policy，将 MinApprovalLevel 合并进 req.Level。
func ManifestPolicyMiddleware(pool BrainPool) ApproverMiddleware {
    return func(next SemanticApprover) SemanticApprover {
        return SemanticApproverFunc(func(ctx context.Context, req ApprovalRequest) (ApprovalDecision, error) {
            // 查询目标 brain 的 manifest policy
            reg := pool.GetRegistration(req.TargetKind)
            if reg != nil && reg.Policy.MinApprovalLevel > req.Level {
                req.Level = reg.Policy.MinApprovalLevel // 只升不降
            }
            return next.Approve(ctx, req)
        })
    }
}

// AuditMiddleware 在 L3/L4 操作通过审批后写入审计日志。
// 它必须在 ModeAwareMiddleware 之后（审批结果确定后才记录）。
func AuditMiddleware(bus EventBus) ApproverMiddleware {
    return func(next SemanticApprover) SemanticApprover {
        return SemanticApproverFunc(func(ctx context.Context, req ApprovalRequest) (ApprovalDecision, error) {
            decision, err := next.Approve(ctx, req)
            if err != nil {
                return decision, err
            }
            // L3/L4: 无论通过还是拒绝，都写审计记录
            if req.Level >= ApprovalLevelControlPlane {
                auditID := publishApprovalAuditEvent(ctx, bus, req, decision)
                decision.AuditID = auditID
            }
            return decision, nil
        })
    }
}
```

### 6.3 默认审批器的构建

```go
// sdk/kernel/approval.go

// DefaultSemanticApprover 构建 v3 的标准审批器。
// 它将现有 StaticSpecialistToolCallAuthorizer 作为第一层，
// 再叠加语义审批、ExecutionMode 感知、审计三层中间件。
func DefaultSemanticApprover(
    pool     BrainPool,
    bus      EventBus,
    confirm  func(ctx context.Context, req ApprovalRequest) (bool, error),
) SemanticApprover {

    // 最内层：ExecutionMode 感知的核心审批策略
    base := &ModeAwareApprover{
        ConfirmFunc: confirm,
        BackgroundAllowLevels: map[ApprovalLevel]bool{
            // 默认 background 不豁免任何 L3/L4
            // brain 可以在 manifest 中声明豁免，由 ManifestPolicyMiddleware 扩展
        },
    }

    // 由外到内：静态矩阵 → manifest policy → 审计 → mode-aware
    return ChainApprovers(
        base,
        AuditMiddleware(bus),
        ManifestPolicyMiddleware(pool),
        StaticMatrixMiddleware(DefaultSpecialistToolCallAuthorizer()),
    )
}
```

## 7. 从 SpecialistToolCallAuthorizer 的迁移路径

### 7.1 迁移策略：三阶段向后兼容

迁移分三个阶段，每个阶段可独立部署，不破坏现有功能：

#### Phase 1：适配层（当前阶段可立即执行）

保留 `SpecialistToolCallAuthorizer` 接口不变，新增 `SemanticApprover` 接口。
`Orchestrator` 同时持有两个字段，新字段为 nil 时回退到旧字段：

```go
// sdk/kernel/orchestrator.go 修改

type Orchestrator struct {
    // ... 原有字段 ...

    // toolCalls 是旧的静态授权矩阵（向后兼容，deprecated）。
    // 当 semanticApprover 不为 nil 时，toolCalls 被忽略。
    toolCalls   SpecialistToolCallAuthorizer

    // semanticApprover 是新的语义审批器（v3 升级目标）。
    // 为 nil 时回退到 toolCalls。
    semanticApprover SemanticApprover
}

// authorizeToolCall 是统一的授权入口，自动选择新/旧实现。
func (o *Orchestrator) authorizeToolCall(ctx context.Context, req ApprovalRequest) error {
    if o.semanticApprover != nil {
        decision, err := o.semanticApprover.Approve(ctx, req)
        if err != nil {
            return err
        }
        if !decision.Approved {
            return fmt.Errorf("语义审批拒绝 [%s]: %s", req.Level, decision.Reason)
        }
        return nil
    }

    // 回退到旧的静态矩阵
    return o.toolCalls.AuthorizeSpecialistToolCall(ctx, req.CallerKind, req.TargetKind, req.ToolName)
}
```

#### Phase 2：语义升级（v3.1 阶段）

- 工具作者为现有工具添加 `ApprovalClass` 字段
- 各 brain manifest 添加 `min_approval_level`
- `Orchestrator.SetSemanticApprover()` API 稳定，文档化
- 现有测试补充 `ApprovalRequest` 路径验证

#### Phase 3：静态矩阵下线（v3.2 阶段）

- `SpecialistToolCallAuthorizer` 接口标记为 `@deprecated`
- `toolCalls` 字段仅保留空实现

## 8. 审批日志与审计：与 EventBus 的集成

### 8.1 审计事件类型

```go
// sdk/kernel/approval_audit.go

// ApprovalAuditEvent 是写入 EventBus 的审批审计事件。
// L3/L4 操作的每次审批（通过或拒绝）都会生成一条。
type ApprovalAuditEvent struct {
    // 基础字段（所有事件都有）
    EventID   string    // UUID v4
    Timestamp time.Time
    TaskID    string    // 关联的 TaskExecution ID
    TurnIndex int       // 关联的 turn 编号

    // 调用链信息
    CallerKind string // 发起方 brain kind
    TargetKind string // 目标 brain kind
    ToolName   string // 工具全名

    // 审批结果
    Level     string // ApprovalLevel.String()，如 "L3:control-plane"
    Decision  string // "approved" / "rejected"
    Reason    string // 拒绝原因（通过时可为空）
    Mode      string // "interactive" / "background"

    // 操作内容摘要（隐私敏感字段需脱敏）
    InputSummary string // InputJSON 的摘要，长度限制 512 字符

    // 人工审批信息（L4 human_approval 场景）
    HumanApprovalRequired bool
    HumanApprovalAt       *time.Time // nil 表示未经人工审批
    HumanApprovalBy       string     // 操作者身份标识（如有）
}

// publishApprovalAuditEvent 将审批结果发布到 EventBus，返回审计 ID。
func publishApprovalAuditEvent(
    ctx  context.Context,
    bus  EventBus,
    req  ApprovalRequest,
    dec  ApprovalDecision,
) string {
    event := ApprovalAuditEvent{
        EventID:      newUUID(),
        Timestamp:    dec.Timestamp,
        TaskID:       req.TaskID,
        TurnIndex:    req.TurnIndex,
        CallerKind:   string(req.CallerKind),
        TargetKind:   string(req.TargetKind),
        ToolName:     req.ToolName,
        Level:        req.Level.String(),
        Decision:     ifElse(dec.Approved, "approved", "rejected"),
        Reason:       dec.Reason,
        Mode:         string(req.Mode),
        InputSummary: summarizeInput(req.InputJSON, 512),
    }

    bus.Publish(ctx, EventTypeApprovalAudit, event)
    return event.EventID
}
```

### 8.2 EventBus 集成要点

| 关注点 | 设计决策 |
|--------|---------|
| 事件类型 | `EventTypeApprovalAudit = "approval.audit"`（新增到 EventBus 事件类型枚举） |
| 写入时机 | **在 AuditMiddleware 中**，审批决策返回后立即发布，不等待工具执行完成 |
| 写入失败处理 | EventBus 写入失败**不应阻塞工具执行**；记录到 stderr 日志，继续执行 |
| 数据保留 | 审计事件应持久化，建议保留 90 天以上（由 EventBus 实现决定） |
| 隐私敏感 | `InputSummary` 必须截断并脱敏（不得包含完整的账户信息、API Key 等） |
| 查询支持 | Dashboard `/v1/dashboard/events` 端点应支持按 `task_id` / `level` / `decision` 过滤 |

### 8.3 审计事件与 Dashboard 的联动

```
EventBus.Publish(ApprovalAuditEvent)
         │
         ├── 内存环形缓冲区（最近 1000 条）
         │         │
         │         └── Dashboard SSE 推送 → /v1/dashboard/events
         │
         └── 持久化存储（可选，由 EventBus 后端决定）
                   │
                   └── GET /v1/dashboard/audit?task_id=xxx&level=L3
```

### 不做的事（范围边界）

- **不实现人工审批工作流队列**：`require_human_approval_above` 在 v3.1 阶段的实现是简单的 CLI 交互确认（`ConfirmFunc`），不实现异步审批队列。异步队列是 v3.2+ 的议题。
- **不修改 brain 间通信协议**：审批在 Kernel 侧完成，不下沉到 sidecar 协议。
- **不做审批决策缓存**：每次调用都完整走审批链，不缓存结果（避免过期策略被复用）。

## 12. 一句话结论

语义审批分级用**五级操作语义**替代"在不在白名单里"的静态判断，让 Kernel 能感知"这个操作到底有多大风险"，并根据当前执行模式（用户在场/无人值守）、Brain Manifest 声明（brain 级安全基线）、审批等级（工具自我声明）三个维度联合做出精准的放行/拒绝/确认决策——同时所有 L3/L4 操作都留有可追溯的审计记录。
