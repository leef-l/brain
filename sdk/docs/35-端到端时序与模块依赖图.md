### 1.2 依赖关系简明表

| 调用方 | 被调用方 | 关系类型 | 说明 |
|--------|---------|---------|------|
| Dashboard | BrainPool, LeaseManager, EventBus, TaskExecution | 观测 | 只读查询+订阅，不干预执行 |
| TaskExecution | TurnExecutor, ContextEngine, EventBus, Learning | 组合 | TaskExec 是执行主循环 |
| TurnExecutor | LeaseManager, SemanticApprover | 门禁+资源 | Plan→Approve→AcquireSet→Execute |
| Orchestrator | BrainPool, ProcessRunner | 路由 | Delegate 只做路由，不管锁不管进程 |
| BrainPool | ProcessRunner, ManifestParser | 管理 | 进程生命周期管理 |
| LeaseManager | WaitForGraph | 兜底 | P2 死锁检测仅 daemon/streaming 场景 |

## 2. 端到端时序：用户请求 → 结果返回

```mermaid
sequenceDiagram
    autonumber
    participant User as User
    participant CB as CentralBrain
    participant CE as ContextEngine
    participant TE as TurnExecutor<br/>(Dispatch Policy)
    participant SA as SemanticApprover
    participant LM as LeaseManager
    participant BP as BrainPool
    participant SB as SpecialistBrain
    participant DB as Dashboard<br/>(EventBus)

    Note over User,DB: ═══ 阶段1: 任务理解 ═══

    User->>CB: POST /v1/runs {prompt}
    CB->>CB: 创建 TaskExecution(Interactive+OneShot)
    CB->>DB: Publish(task_created)

    Note over User,DB: ═══ 阶段2: 上下文装配 ═══

    CB->>CE: Assemble(AssembleRequest)
    CE->>CE: 委托头部注入 → 跨Turn记忆 → L3 History → BrainKind过滤 → token压缩
    CE-->>CB: []llm.Message

    Note over User,DB: ═══ 阶段3: LLM 推理 + 脑选择 ═══

    CB->>CB: LLM 推理 → 返回 tool_calls[]
    CB->>DB: Publish(turn_started)

    alt LLM 返回最终答复（无 tool_call）
        CB-->>User: 结果文本
        CB->>DB: Publish(task_completed)
    else LLM 返回 tool_calls（含 delegate 或直接工具）
        Note over User,DB: ═══ 阶段4: 调度规划 ═══

        CB->>TE: dispatchTools(tool_calls[])
        TE->>TE: 从 ToolConcurrencySpec 推导 LeaseRequest[]
        TE->>TE: 构建冲突图(ConflictGraph)
        TE->>TE: Welsh-Powell 贪心着色 → Batch[]

        Note over User,DB: ═══ 阶段5: 语义审批（逐 Batch） ═══

        loop 每个 Batch
            TE->>SA: Approve(ApprovalRequest[])
            SA->>SA: 决策树: tool.ApprovalClass → manifest.min_level → 前缀启发式
            alt 审批拒绝
                SA-->>TE: ApprovalDenied
                TE->>DB: Publish(approval_denied)
                TE-->>CB: 工具被拒绝
            else 审批通过
                SA-->>TE: ApprovalGranted
                SA->>DB: Publish(audit.approval)

                Note over User,DB: ═══ 阶段6: 资源获取 ═══

                TE->>LM: AcquireSet(LeaseRequest[])
                LM->>LM: canonical ordering → 逐个获取
                alt 获取失败
                    LM->>LM: 全部回滚 → 指数退避+jitter → 重试
                    LM-->>TE: ErrAcquireTimeout
                else 获取成功
                    LM-->>TE: Lease[]

                    Note over User,DB: ═══ 阶段7: 并行执行 ═══

                    par Batch 内并行
                        TE->>BP: GetBrain(kind)
                        BP-->>TE: BidirRPC
                        TE->>SB: brain/execute(tool_call)
                        SB-->>TE: tool_result
                    end

                    Note over User,DB: ═══ 阶段8: 结果回收 ═══

                    TE->>LM: Release(lease) [scope=turn 时]
                end
            end
        end

        TE-->>CB: 结果按原 tool_call 顺序回填
        CB->>CE: Persist(runID, messages)
        CB->>DB: Publish(turn_completed)
        CB->>CB: 下一轮 LLM 推理（循环直到最终答复）
        CB-->>User: 最终结果
    end
```

## 3. 端到端时序：Delegate（中央脑委托专精脑）

```mermaid
sequenceDiagram
    autonumber
    participant CB as CentralBrain
    participant Orch as Orchestrator
    participant SA as SemanticApprover
    participant LM as LeaseManager
    participant BP as BrainPool
    participant CE as ContextEngine
    participant SB as SpecialistBrain
    participant EB as EventBus

    Note over CB,EB: ═══ 委托决策 ═══

    CB->>CB: LLM 推理 → delegate(target=quant, task=...)
    CB->>Orch: Delegate(SubtaskRequest)

    Note over CB,EB: ═══ 门禁序列 §10.2 ═══

    Orch->>Orch: ① 身份鉴权: callerKind == KindCentral
    Orch->>Orch: ② license 检查: 目标 brain 持有有效 license
    Orch->>Orch: ③ delegate_freeze 检查: brain 未被冻结
    Orch->>SA: ④ policy 检查: Approve(ApprovalRequest)
    SA-->>Orch: ApprovalGranted + 审计记录
    Orch->>BP: ⑤ HealthCheck(kind=quant)
    BP-->>Orch: healthy

    Note over CB,EB: ═══ Capability 匹配 + Lease 获取 ═══

    Orch->>Orch: ⑥ CapabilityMatcher.Match(required=[execution.order], preferred=[domain.finance])
    Note right of Orch: 三阶段匹配：硬匹配→软匹配→负载感知<br/>评分公式：Score=100+soft×50+idle×30
    Orch->>LM: ⑦ AcquireSet([{execution.order, account:paper-main, ExclusiveWrite, ScopeTask}])
    LM->>LM: canonical ordering → 获取
    LM-->>Orch: Lease[]{id=lease-001, holder=taskExec-42, ttl=300s}
    Orch->>EB: Publish(lease_acquired)

    Note over CB,EB: ═══ 获取 Brain 连接 ═══

    Orch->>BP: GetBrain(kind=quant)
    BP->>BP: shared-service 策略 → 返回长驻单例连接
    BP-->>Orch: BidirRPC

    Note over CB,EB: ═══ 上下文共享 ═══

    Orch->>CE: Share(from=central, to=quant, relevantMsgs)
    CE->>CE: 隐私过滤 → SubtaskContext 构建
    CE-->>Orch: ok

    Note over CB,EB: ═══ 执行 ═══

    Orch->>SB: brain/execute(SubtaskRequest + SubtaskContext)
    SB->>SB: 专精脑 Agent Loop 执行
    SB->>SB: [内部可能调用多个 tool_call]

    alt 执行成功
        SB-->>Orch: SubtaskResult{status=ok, result=...}
        Orch->>CE: merge 结果摘要(≤500 token) 回 central 上下文
    else 执行失败
        SB-->>Orch: SubtaskResult{status=failed, error=...}
        Orch->>EB: Publish(delegate_failed)
    end

    Note over CB,EB: ═══ Lease 释放 ═══

    Orch->>LM: Release(lease-001)
    LM-->>Orch: ok
    Orch->>EB: Publish(lease_released)
    Orch-->>CB: SubtaskResult
```

## 4. 端到端时序：并行 Batch 执行

```mermaid
sequenceDiagram
    autonumber
    participant LLM as LLM Response
    participant TE as TurnExecutor
    participant DP as BatchPlanner<br/>(dispatch/)
    participant SA as SemanticApprover
    participant LM as LeaseManager
    participant BP as BrainPool
    participant T1 as Tool_1 执行
    participant T2 as Tool_2 执行
    participant T3 as Tool_3 执行

    Note over LLM,T3: ═══ 接收多个 tool_call ═══

    LLM->>TE: tool_calls[call_1, call_2, call_3, call_4, call_5]

    Note over LLM,T3: ═══ 冲突图构建 ═══

    TE->>DP: Plan(toolCalls, concurrencySpecs)
    DP->>DP: 为每个 tool_call 推导 LeaseRequest<br/>(ToolConcurrencySpec + JSON参数 → ResourceKey)
    DP->>DP: 构建 ConflictGraph (N×N 邻接矩阵)
    Note right of DP: 冲突判定:<br/>同 ResourceKey +<br/>至少一方 Exclusive → 冲突

    Note over LLM,T3: ═══ Welsh-Powell 图着色分组 ═══

    DP->>DP: 节点按度数降序排列
    DP->>DP: 贪心着色 → 同色即同 Batch
    DP-->>TE: BatchPlan{<br/>  Batch0: [call_1, call_2, call_3],<br/>  Batch1: [call_4, call_5],<br/>  ErrorStrategy: ContinueBatch<br/>}

    Note over LLM,T3: ═══ Batch 0 执行 ═══

    TE->>SA: Approve(Batch0.calls)
    SA-->>TE: AllGranted

    TE->>LM: AcquireSet(Batch0.leaseRequests)
    LM-->>TE: Lease[]{lease-A, lease-B, lease-C}

    par Batch 0 并行执行
        TE->>T1: execute(call_1)
        T1-->>TE: result_1 ✓
    and
        TE->>T2: execute(call_2)
        T2-->>TE: result_2 ✓
    and
        TE->>T3: execute(call_3)
        T3-->>TE: result_3 ✗ error!
    end

    Note right of TE: ErrorStrategy 决策点:<br/>ContinueBatch → 不影响其他<br/>FailBatch → 取消本batch剩余<br/>FailAll → 终止整个dispatch

    TE->>LM: Release(Batch0.leases) [scope=turn]

    Note over LLM,T3: ═══ Batch 1 执行 ═══

    TE->>SA: Approve(Batch1.calls)
    SA-->>TE: AllGranted

    TE->>LM: AcquireSet(Batch1.leaseRequests)
    LM-->>TE: Lease[]{lease-D, lease-E}

    par Batch 1 并行执行
        TE->>T1: execute(call_4)
        T1-->>TE: result_4 ✓
    and
        TE->>T2: execute(call_5)
        T2-->>TE: result_5 ✓
    end

    TE->>LM: Release(Batch1.leases)

    Note over LLM,T3: ═══ 结果聚合 ═══

    TE->>TE: 按原 tool_call Index 排序回填
    TE-->>LLM: [result_1, result_2, error_3, result_4, result_5]
```

## 5. 端到端时序：故障恢复

```mermaid
sequenceDiagram
    autonumber
    participant SB as SpecialistBrain<br/>(sidecar 进程)
    participant PR as ProcessRunner
    participant BP as BrainPool
    participant LM as LeaseManager
    participant TE as TaskExecution
    participant EB as EventBus
    participant DB as Dashboard

    Note over SB,DB: ═══ Brain 崩溃 ═══

    SB->>SB: 进程 crash / panic / OOM
    PR->>PR: exit hook 检测到进程退出 (code != 0)
    PR->>BP: 通知 entry 状态 → dead

    Note over SB,DB: ═══ BrainPool 检测 (快路径 < 100ms) ═══

    BP->>BP: entry 状态机: in-use → dead
    BP->>EB: Publish(brain_crashed, kind=quant)
    EB->>DB: 实时推送 brain 状态变更

    Note over SB,DB: ═══ Lease 回收 (双通道) ═══

    rect rgb(255, 240, 240)
        Note right of LM: 快路径: ProcessRunner exit hook
        BP->>LM: ForceRevokeHolder(holder=taskExec-42, reason="brain_crashed")
        LM->>LM: 按 scope 倒序释放:<br/>daemon > task > turn
        LM->>LM: 通知等待者 waiterCh (唤醒阻塞的 AcquireSet)
        LM-->>BP: revoked_count=3
        LM->>EB: Publish(lease_force_revoked)
    end

    rect rgb(240, 240, 255)
        Note right of LM: 兜底: TTL 过期扫描 (每 10s)
        LM->>LM: 扫描 TTL 过期的 lease → 强制回收
        Note right of LM: 正常由快路径处理<br/>TTL 兜底防止 exit hook 丢失
    end

    Note over SB,DB: ═══ TaskExecution 状态转移 ═══

    LM->>TE: leaseRevokeCh 通知
    TE->>TE: ctx cancel → 状态: running → crashed
    TE->>EB: Publish(task_crashed)

    Note over SB,DB: ═══ RestartPolicy 判定 ═══

    alt RestartPolicy == "never"
        TE->>TE: 状态: crashed (终态)
        TE->>EB: Publish(task_terminal)
    else RestartPolicy == "on-failure" 且重启次数 < maxRetries
        TE->>TE: 状态: crashed → restarting
        TE->>TE: 退避计算: base * 2^n ± 20% jitter
        Note right of TE: 默认: 1s 起步, 最大 5min<br/>ElapsedTime 继续计时, Turn 不计
        TE->>TE: 等待退避计时器到期...
        TE->>TE: 状态: restarting → pending
        TE->>EB: Publish(task_rescheduled)

        Note over SB,DB: ═══ 重新调度 ═══

        TE->>BP: GetBrain(kind=quant)
        BP->>BP: 检测到 quant entry = dead
        BP->>PR: fork/exec 新 sidecar 进程
        PR-->>BP: 新进程 ready (状态: starting → idle)
        BP-->>TE: BidirRPC (新连接)
        TE->>TE: 状态: pending → running
        TE->>EB: Publish(task_restarted)
    else RestartPolicy 重启次数耗尽
        TE->>TE: 状态: restarting → failed (终态)
        TE->>EB: Publish(task_restart_exhausted)
    end
```

## 6. 端到端时序：热更新 Manifest

```mermaid
sequenceDiagram
    autonumber
    participant FS as FileSystem
    participant FW as FileWatcher
    participant MP as Manifest 解析器
    participant BP as BrainPool
    participant LM as LeaseManager
    participant Orch as Orchestrator
    participant EB as EventBus
    participant DB as Dashboard

    Note over FS,DB: ═══ 检测变更 ═══

    FS->>FW: brain.json 文件修改
    FW->>FW: 防抖等待 (500ms debounce)
    FW->>MP: 触发重新解析

    Note over FS,DB: ═══ 解析新 Manifest ═══

    MP->>MP: 读取 → JSON Schema v1 校验 → Go 四层验证
    alt 校验失败
        MP->>EB: Publish(manifest_parse_error)
        MP->>MP: 保留旧配置不变更
    else 校验通过
        MP->>MP: diff 对比: 新旧 Manifest 差异
        MP->>MP: 前向兼容检查:<br/>新增 capability OK<br/>删除 capability 告警
        MP->>EB: Publish(manifest_validated, diff=...)

        Note over FS,DB: ═══ Drain 旧实例 ═══

        MP->>BP: Drain(ctx, kind)
        BP->>BP: 停止接受新的 GetBrain 请求
        BP->>BP: 等待在途请求全部完成
        Note right of BP: Drain 超时: 默认 30s<br/>超时后强制 kill

        rect rgb(255, 250, 230)
            Note right of LM: Drain 期间的 Lease 处理
            LM->>LM: 在途 task 持有的 ScopeTask lease 继续有效
            LM->>LM: 新的 AcquireSet 对该 kind 的请求排队等待
            LM->>LM: Drain 完成后:<br/>- ScopeTurn lease 已随 turn 结束释放<br/>- ScopeTask lease 随 task 完成释放<br/>- ScopeDaemon lease 强制回收
        end

        BP->>BP: 关闭旧 sidecar 进程
        BP->>EB: Publish(brain_drained, kind=...)

        Note over FS,DB: ═══ 重注册 BrainPool ═══

        MP->>BP: Register(BrainPoolRegistration{新策略})
        BP->>BP: 更新 kindPool 配置
        Note right of BP: 策略可能变更:<br/>如 approval_class 变化<br/>导致进程策略切换

        MP->>Orch: 更新注册信息(capabilities, policy)
        Orch->>Orch: 更新 registrations map

        Note over FS,DB: ═══ 新 Capability 生效 ═══

        BP->>BP: WarmUp(ctx, kind, defaultN)
        BP->>BP: 启动新 sidecar 实例 → ready
        BP->>EB: Publish(brain_registered, newCapabilities=[...])

        Note over FS,DB: ═══ 回滚保底 ═══

        alt WarmUp 失败
            BP->>MP: 回滚到旧 Manifest
            MP->>BP: Register(旧 BrainPoolRegistration)
            BP->>EB: Publish(manifest_rollback)
        else WarmUp 成功
            DB->>DB: 新 brain 状态已上线, 新 capabilities 可用
            EB->>DB: 推送 brain_status_updated
        end
    end
```

## 7. 门禁执行顺序全景图

### 7.1 完整门禁时序（§10.2 在完整请求中的位置）

```mermaid
sequenceDiagram
    autonumber
    participant Req as 请求到达
    participant G1 as ① license
    participant G2 as ② delegate_freeze
    participant G3 as ③ policy
    participant G4 as ④ health
    participant G5 as ⑤ approve
    participant G6 as ⑥ acquireSet
    participant Exec as ⑦ execute

    Note over Req,Exec: 门禁链: 快速布尔判断(①-④) → 语义审批(⑤) → 资源获取(⑥) → 执行(⑦)
    Note over Req,Exec: 原则: 越便宜的检查越靠前, 门禁失败不进入 AcquireSet 避免无效占锁

    Req->>G1: subtask.delegate 请求
    G1->>G1: license 检查<br/>目标 brain 是否持有有效 license<br/>⏱ < 1μs (内存查表)
    alt 无 license
        G1-->>Req: ErrNoLicense (快速拒绝)
    end

    G1->>G2: 通过
    G2->>G2: delegate_freeze 检查<br/>brain 是否被冻结 (热更新/drain中)<br/>⏱ < 1μs (原子读)
    alt 已冻结
        G2-->>Req: ErrBrainFrozen (快速拒绝)
    end

    G2->>G3: 通过
    G3->>G3: policy 检查<br/>ExecutionPolicy + ToolPolicy<br/>callerKind/targetKind/toolName 矩阵<br/>⏱ < 10μs (规则匹配)
    alt 策略拒绝
        G3-->>Req: ErrPolicyDenied (快速拒绝)
    end

    G3->>G4: 通过
    G4->>G4: health 检查<br/>BrainPool.HealthCheck(kind)<br/>⏱ < 50ms (ping/pong)
    alt 不健康
        G4-->>Req: ErrBrainUnhealthy (快速拒绝, 触发替换)
    end

    G4->>G5: 通过
    G5->>G5: 语义审批 (SemanticApprover)<br/>三层决策树:<br/>tool.ApprovalClass → manifest.min_level → 前缀启发式<br/>⏱ < 1ms (可能需人工确认则阻塞)
    alt 审批拒绝
        G5-->>Req: ErrApprovalDenied (审计记录)
    end

    G5->>G6: 通过
    G6->>G6: AcquireSet 资源获取<br/>canonical ordering → 逐个 lease → 全部成功或全部回滚<br/>⏱ 0-5s (可能阻塞等待)
    alt 获取超时
        G6-->>Req: ErrAcquireTimeout (无副作用)
    end

    G6->>Exec: 所有门禁通过
    Exec->>Exec: delegateOnce → brain/execute<br/>真正执行
```

### 7.2 门禁速查表

| 序号 | 门禁 | 检查内容 | 典型延迟 | 失败处理 | 资源消耗 |
|------|------|---------|---------|---------|---------|
| ① | license | brain license 有效性 | < 1μs | 快速拒绝 | 无 |
| ② | delegate_freeze | brain 是否被冻结 | < 1μs | 快速拒绝 | 无 |
| ③ | policy | 执行策略+工具策略 | < 10μs | 快速拒绝 | 无 |
| ④ | health | sidecar 进程健康 | < 50ms | 拒绝+触发替换 | 一次 ping |
| ⑤ | approve | 语义审批等级 | < 1ms ~ 阻塞 | 拒绝+审计 | 可能人工 |
| ⑥ | acquireSet | Capability Lease 获取 | 0 ~ 5s | 超时+退避重试 | 锁资源 |
| ⑦ | execute | brain/execute RPC | 不限 | 工具层错误 | 完整执行 |

**设计原则**：门禁①-④是零成本布尔判断，任何一个失败立即返回，不消耗下游资源。⑤是策略判断，可能涉及人工确认但不消耗锁资源。⑥才真正获取锁。**越便宜越靠前。**

## 9. 关键不变量（Invariants）

以下不变量在系统运行的**任何时刻**都必须成立。违反任何一条即为 Bug。

### 9.1 Lease 不变量

| # | 不变量 | 执行机制 | 违反后果 |
|---|--------|---------|---------|
| L1 | **任何时刻，一个 ResourceKey 上最多一个 Exclusive lease** | LeaseManager 兼容性矩阵检查 (§13.2) | 数据竞争/资金安全事故 |
| L2 | **AcquireSet 要么全成功要么全回滚（原子性）** | AcquireSet 算法: 逐个获取, 失败回滚所有已获取的 | 半获取导致死锁 |
| L3 | **canonical ordering: AcquireSet 内部按 ResourceKey+Capability 字典序获取** | `sort.Sort(byCanonicalOrder(reqs))` | 乱序获取导致 AB-BA 死锁 |
| L4 | **Task 处于终态 (completed/failed/canceled) 时不持有任何 lease** | 状态转移 hook: 进入终态前 `ReleaseAll(holder, scope)` | 资源泄漏, 后续 task 永久阻塞 |
| L5 | **daemon scope lease 自动续租, 心跳间隔 = TTL/3** | heartbeat goroutine + 连续 3 次错过强制撤销 | lease 意外过期导致 daemon 中断 |
| L6 | **Exclusive lease 持有者崩溃后, lease 必须在 TTL 内被回收** | 双通道: exit hook 快路径 + TTL 扫描兜底 | 资源永久锁定 |

### 9.2 TaskExecution 不变量

| # | 不变量 | 执行机制 | 违反后果 |
|---|--------|---------|---------|
| T1 | **所有 TaskExecution 从 pending 状态开始, 无例外** | 构造函数强制 `Status = StatePending` | 状态机不一致 |
| T2 | **终态 (completed/failed/canceled) 不可逆——除非 RestartPolicy 允许从 failed/crashed 进入 restarting** | 状态转移表白名单 + `canTransition()` 检查 | 僵尸 task |
| T3 | **一个 TaskExecution 同一时刻只在一个 Runner 上执行** | BrainPool GetBrain + holder 绑定 | 重复执行, 数据冲突 |
| T4 | **父子传播: 当 ChildFailurePolicy=propagate_immediately 时, 子任务 failed → 父任务必须在当前 turn 结束前转 failed** | 父 Runner 监听子 TaskExecution 状态 channel | 父任务"假成功" |

### 9.3 Dispatch 不变量

| # | 不变量 | 执行机制 | 违反后果 |
|---|--------|---------|---------|
| D1 | **同一 batch 内的 tool_call 两两无资源冲突** | ConflictGraph + Welsh-Powell 着色保证同色无边 | 并发执行导致数据竞争 |
| D2 | **结果按原 tool_call Index 顺序回填, 对 LLM 透明** | BatchPlan 保留原始 Index, 聚合时排序 | LLM 把结果对错工具 |
| D3 | **禁止普通工具执行过程中再隐式追加跨资源 lease (P0 约束)** | AcquireSet 只在 batch 开始前调用, 执行期间 LeaseManager 拒绝该 holder 新请求 | 运行时死锁 |

### 9.4 Delegate 不变量

| # | 不变量 | 执行机制 | 违反后果 |
|---|--------|---------|---------|
| DE1 | **Delegate 调用必须先通过全部门禁 (license→freeze→policy→health→approve)** | Orchestrator.Delegate() 顺序执行门禁链 | 未授权执行 |
| DE2 | **只有 callerKind == KindCentral 才能发起 delegate** | `registerReverseHandlers()` 闭包内身份检查 | 非法横向委托 |
| DE3 | **MCP server 不能直接成为 delegate target** | `CanDelegate()` 只查 available[kind] brain | 绕过协议边界 |
| DE4 | **delegate 执行入口固定为 `brain/execute`** | `delegateOnce()` 硬编码方法名 + 协议常量 | 协议偏离 |

### 9.5 BrainPool 不变量

| # | 不变量 | 执行机制 | 违反后果 |
|---|--------|---------|---------|
| BP1 | **锁顺序: brainPool.mu > kindPool.mu > poolEntry.mu, 不可逆** | 代码 review + 注释标注 | Pool 内部死锁 |
| BP2 | **Drain 期间拒绝所有新的 GetBrain 请求** | kindPool.draining 原子标志 | 返回已关闭的连接 |
| BP3 | **ephemeral-worker 最大并发实例数 ≤ maxInstances (默认 8)** | poolEntry 计数器 + 信号量 | 进程爆炸, OOM |

### 9.6 ContextEngine 不变量

| # | 不变量 | 执行机制 | 违反后果 |
|---|--------|---------|---------|
| CE1 | **private 标记的消息绝不跨脑传递** | Share() 隐私过滤器 | 信息泄露 |
| CE2 | **Assemble 返回的消息 token 总量 ≤ TokenBudget** | Compress 三层策略兜底: 裁剪→摘要→贪心 | LLM API 报错 / 截断 |
| CE3 | **跨脑共享的 SubtaskContext 中 RelevantMessages ≤ 10 条, PriorResults 各 ≤ 200 token** | Share 协议硬上限 | 上下文爆炸 |

### 9.7 系统级不变量

| # | 不变量 | 执行机制 | 违反后果 |
|---|--------|---------|---------|
| S1 | **所有 brain 都是独立 sidecar 进程, 零例外** | BrainPool 强制通过 ProcessRunner fork/exec | 隔离性丧失 |
| S2 | **CAS 写入幂等: 同内容并发写入产生相同 ref** | blake3 content-addressing + atomic rename | 数据重复/不一致 |
| S3 | **EventBus 环形缓冲溢出时丢弃最旧事件, 不阻塞生产者** | 固定 10000 条 + 覆写策略 | 反压导致执行卡住 |
| S4 | **Manifest 热更新失败必须回滚到旧配置** | WarmUp 失败→回滚旧 Manifest→重新 Register | brain 不可用 |
