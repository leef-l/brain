## 二、重构方案

### 2.1 cmd/brain/ 拆分（P0，最严重）

#### 难点分析

所有文件都在 `package main` 里，符号可以自由互引。拆子目录 = 拆子 package = 必须 export 跨包引用的符号。

依赖分析发现 5 组严重的交叉依赖：
1. `brainConfig` — 被 12+ 文件依赖，是全局核心类型
2. `cliRuntime` / `persistedRunRecord` — runtime 组向外暴露给 chat 和 cmd
3. `approvalPrompter` / `approvalRequest` — chat 组暴露给 execution_env
4. `permissionMode` — chat_mode 暴露给 execution_env、cmd_serve、tool_profiles
5. `buildSystemPrompt` — chat_prompt 暴露给 cmd_run、cmd_serve

#### 符号导出规则

拆包后，跨包引用的 unexported 符号必须 export（首字母大写）：

| 原符号 | 新包 | 导出为 |
|--------|------|--------|
| `brainConfig` | `config` | `Config` |
| `loadConfig` | `config` | `Load` |
| `resolvedProvider` | `config` | `ResolvedProvider` |
| `cliRuntime` | `runtime` | `CLIRuntime` |
| `persistedRunRecord` | `runtime` | `RunRecord` |
| `loadPersistedRun` | `runtime` | `LoadRun` |
| `chatState` | `chat` | `State` |
| `permissionMode` | `chat` | `PermissionMode` |
| `approvalPrompter` | `chat` | `ApprovalPrompter` |
| `buildSystemPrompt` | `chat` | `BuildSystemPrompt` |
| `runChat` | `chat` | `Run` |

#### 依赖方向（严格单向）

```
main.go
  ├── dispatcher → command/*, chat/
  │
  config/ ← 所有包都可以 import
  │
  runtime/ ← command/, chat/, env/
  │
  env/ ← chat/, command/
  │
  chat/ → config, runtime, env, term, bridge
  │
  command/ → config, runtime, env, bridge
  │
  bridge/ → config, runtime (不依赖 chat/command)
  │
  term/ ← chat/ (纯终端，无业务依赖)
  │
  dashboard/ → runtime, config
```

**禁止**：chat ↔ command 互相引用。共享逻辑放 config/runtime/env。

### 2.3 sdk/kernel/ 暂不拆（P2 观察）

理由：
1. 已有 3 个子目录（manifest, mcphost, mcpadapter）
2. 顶层 30 个文件中，大多是接口+实现成对出现（approval.go + approval_test.go）
3. Go 的包语义要求同包内类型可互引，kernel 各模块（approval, dispatch, execution, lease）存在紧密耦合
4. 强行拆分会导致大量循环依赖或接口爆炸

**待观察**：如果后续 kernel 文件数增长到 40+，再考虑拆分 `kernel/dispatch/`, `kernel/execution/` 等。

## 四、执行计划（原始设计）

| 步骤 | 内容 | 风险等级 | 预估工作量 |
|------|------|---------|-----------|
| Step 1 | `config/` 包提取 — brainConfig 及相关类型 | 高（12+ 文件引用） | 大 |
| Step 2 | `runtime/` 包提取 — cliRuntime, store, persistence | 高（cmd/chat 双向引用） | 大 |
| Step 3 | `env/` 包提取 — executionEnvironment | 中 | 中 |
| Step 4 | `term/` 包提取 — rawterm, keybindings, spinner | 低（纯终端） | 小 |
| Step 5 | `chat/` 包提取 — 15 个 REPL 文件 | 高（approvalPrompter 跨引用） | 大 |
| Step 6 | `command/` 包提取 — 13 个子命令 | 中（主要依赖 config+runtime） | 中 |
| Step 7 | `bridge/` 包提取 — 适配器文件 | 低 | 小 |
| Step 8 | `dashboard/` 合并 dashboard_handler.go | 低 | 小 |
| Step 9 | `sdk/tool/builtin/` 提取 | 中（工具注册链） | 中 |
| Step 10 | `sdk/tool/sandbox/` 提取 | 低 | 小 |
| Step 11 | 全量编译验证 + 测试 | — | 中 |

**执行顺序**：Step 1 → 2 → 3 → 4（低依赖的先拆），然后 5 → 6 → 7 → 8（高依赖的后拆），最后 9 → 10 → 11。

## 四、风险与回退

- **最大风险**：`brainConfig` 提取为 `config.Config` 后，所有引用点都要改。建议用 type alias 过渡：`type brainConfig = config.Config`，逐步消除。
- **编译验证**：每完成一个 Step 都必须 `go build ./...` 通过。
- **回退策略**：每个 Step 一个 commit，如果某步失败可以 revert。
- **测试保障**：`go test ./cmd/brain/...` 在每步后都必须通过。
