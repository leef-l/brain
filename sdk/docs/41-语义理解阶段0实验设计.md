# 41. 语义理解阶段 0:实验设计

> **目的**:在为 [`40-Browser-Brain语义理解架构`](./40-Browser-Brain语义理解架构.md) 投入数周开发前,**用 1 周时间做一个低成本实验**,回答一个关键问题:
>
> **当前 LLM 水平下,从不同输入(结构/视觉/HTML)能多准确地推断出 L6-L8 层的语用信息?**
>
> 这是整份架构的**Go / No-Go 决策点**。实验不做就直接写代码,等于盲投。

---

## 1. 实验核心问题

### 1.1 要回答什么

**问题 1(能力上限)**:给定一个网页上的可交互元素,在不同输入组合下,LLM 能否准确推断它的:
- `action_intent`(意图:点这个按钮要达成什么)
- `reversibility`(可逆性:点了能不能撤销)
- `risk_level`(风险:是不是破坏性动作)
- `flow_role`(流程角色:主动作/备选/转义/导航)

**问题 2(成本/收益权衡)**:
- 结构 snapshot(最便宜)的准确率是多少?
- 加 screenshot(贵)能提升多少?
- 加 HTML 片段(中等)能提升多少?
- 哪种输入组合是**性价比最优**的?

**问题 3(边界识别)**:
- 哪些元素 LLM 判断**稳定准确**?(→ 可用纯 JS + LLM 批量预处理)
- 哪些元素 LLM **不稳定或错得离谱**?(→ 阶段 1 预处理无法覆盖,必须靠阶段 2 模式库或阶段 3 多模态兜底)

### 1.2 不回答什么(避免范围蔓延)

- 不评估**动作执行**的成功率(那是阶段 1+后的事)
- 不评估**模式库**的效果(阶段 2 才有意义)
- 不评估**异常处理**(单独文档)
- 不调 Browser Brain 的代码(实验独立,纯数据分析)

## 2. 测试集构建

### 2.1 10 个代表性网页

按 5 类各选 2 个,覆盖常见场景:

| 类别 | 样本 1 | 样本 2 | 说明 |
|---|---|---|---|
| **登录页** | GitHub login | 银行网银登录 | 简单 vs 高风险 |
| **电商详情** | Amazon 商品页 | 淘宝商品页 | 国外 vs 国内,按钮密度差异大 |
| **后台管理** | 一个开源 CMS 管理台 | 自建 admin | 结构化强 vs 自由发挥 |
| **搜索结果** | Google 搜索 | Bing 搜索 | 相似但不同 |
| **表单填写** | 多步注册表 | 报税/合同类表单 | 简单 vs 极复杂 |

**样本获取**:保存为静态 HTML + CSS + 截图三元组,放在 `sdk/docs/experiments/phase0/samples/`。
避免测试时依赖网络(保证可复现)。

### 2.2 元素抽样

每个页面抽取 **5-8 个可交互元素**作为标注样本,覆盖:
- 主动作按钮(primary action)
- 次要按钮(secondary)
- 危险按钮(如果有,如"删除")
- 导航链接
- 表单输入框
- 复选框/单选框
- 下拉菜单
- "忘记密码"类转义链接

**总样本量**:10 页 × 6 个元素 ≈ **60 个标注样本**。

## 3. 标注规范(人类 ground truth)

### 3.1 4 个维度的定义

由**两位熟悉网页交互的人独立标注**,不一致的条目讨论后统一。

#### action_intent(意图)

一句话描述"点这个元素想达成什么目标"。

**示例**:
- "提交登录凭据验证身份"
- "添加商品到购物车以便后续对比"
- "删除当前用户账户(不可恢复)"
- "跳转到密码重置流程"

#### reversibility(可逆性)

| 值 | 含义 |
|---|---|
| `reversible` | 点了可以撤销,无持久副作用(搜索/筛选/展开/筛选) |
| `semi_reversible` | 有副作用但可挽回(加购物车/收藏/草稿保存) |
| `irreversible` | 有持久副作用且不可撤销(下单/付款/发送消息/删除) |
| `conditional` | 取决于后续操作(下一步可能有"确认"页面) |

#### risk_level(风险)

| 值 | 含义 |
|---|---|
| `safe` | 无副作用或副作用完全可控(读取/搜索) |
| `safe_caution` | 有副作用但小范围(提交表单/登录尝试) |
| `destructive` | 破坏性(删除/清空/支付/发送不可撤回消息) |
| `external_effect` | 影响到网页之外(发邮件/发短信/调第三方 API) |

#### flow_role(流程角色)

| 值 | 含义 |
|---|---|
| `primary` | 主路径动作(登录页的"登录"按钮) |
| `secondary` | 备选路径("用手机验证码登录"作为替代) |
| `escape` | 转义/退出("忘记密码"/"取消") |
| `navigation` | 页面内导航(锚点/标签切换) |
| `cross_page_nav` | 跨页导航(去其他页) |
| `utility` | 辅助功能(帮助/反馈/语言切换) |

### 3.2 标注冲突解决

两人独立标注后比对:
- **完全一致**:直接采用
- **轻微分歧**(如 `safe` vs `safe_caution`):讨论后采用更保守的
- **根本分歧**:记录为 `ambiguous`,排除出准确率统计(同时作为"人类也不确定"的证据)

**预期结果**:大部分元素(80%+)两人会一致,少量(如"忘记密码"的 flow_role 是 escape 还是 cross_page_nav)会讨论。

### 3.3 标注文件格式

`sdk/docs/experiments/phase0/labels/{page_id}.json`:

```json
{
  "page_id": "github_login",
  "url": "https://github.com/login",
  "snapshot_file": "samples/github_login/snapshot.json",
  "screenshot_file": "samples/github_login/screenshot.png",
  "html_file": "samples/github_login/page.html",
  "elements": [
    {
      "id": 17,
      "description": "Sign in button at bottom of form",
      "bbox": { "x": 400, "y": 520, "w": 200, "h": 40 },
      "ground_truth": {
        "action_intent": "submit login credentials to authenticate",
        "reversibility": "reversible_on_failure",
        "risk_level": "safe_caution",
        "flow_role": "primary"
      },
      "notes": "Failed 3x may trigger captcha"
    }
  ]
}
```

## 4. 实验矩阵

### 4.1 输入组合(3 种)

| 组合 ID | 输入内容 | 估算 token | 目的 |
|---|---|---|---|
| **A** | 纯结构 snapshot(39 号文档定义的扁平清单) | ~500-1500 | 基线,最便宜 |
| **B** | snapshot + 截图(PNG base64) | ~2000-6000 | 测视觉增益 |
| **C** | snapshot + 对应元素周围 200 行 HTML | ~1500-3500 | 测静态结构增益 |

**为什么不测 "截图+HTML"**:那个组合成本最高,如果 B 和 C 分别有明显增益再组合;如果 B/C 都效果平平,组合起来也不会神奇。

### 4.2 Prompt 设计

统一 system prompt(三种输入都用):

```
You are a web UI analyst. For each interactive element given,
predict:
1. action_intent: one sentence describing what the user wants to achieve by interacting
2. reversibility: one of {reversible, semi_reversible, irreversible, conditional}
3. risk_level: one of {safe, safe_caution, destructive, external_effect}
4. flow_role: one of {primary, secondary, escape, navigation, cross_page_nav, utility}

For each, also provide confidence: 0.0-1.0.
Be honest about uncertainty — don't always say 0.9.

Output JSON only.
```

User prompt(组合 A 示例):

```
Page URL: https://github.com/login
Element ID: 17
Element description:
  tag=button, text="Sign in", visible=true, x=400, y=520
Other elements on page (for context):
  [15] textbox "Username or email" value=""
  [16] textbox "Password" value="" type=password
  [17] button "Sign in"   ← this one
  [18] link "Forgot password?" href=/password_reset
  [19] link "Create an account" href=/signup
```

组合 B 追加:

```
Screenshot attached.
```

组合 C 追加:

```
Surrounding HTML:
<form action="/session" method="post">
  <input type="text" name="login" autocomplete="username">
  <input type="password" name="password" autocomplete="current-password">
  <input type="submit" value="Sign in" class="btn btn-primary btn-block">
  ...
</form>
```

### 4.3 LLM 选型

**至少测 2 个模型**:
- Claude Opus 4.7(当前强模型,看上限)
- Claude Haiku(cheap 模型,看阶段 1 批量预处理的可行性)

**可选加测**:
- GPT-4.5 / GPT-4o(对比家族)
- DeepSeek V3(自建 provider 已接)

每个样本每个模型**独立调 3 次**,取多数结果,规避随机性。

### 4.4 总调用量估算

60 元素 × 3 输入组合 × 2 模型 × 3 次 = **1080 次 LLM 调用**

按平均 3000 token/次,合计约 300 万 tokens。Opus 档位费用 ~$45,Haiku 档位 ~$5。**总预算 < $50**。

一周完成的核心是**样本采集 + 人工标注**,LLM 调用本身一天就能跑完。

## 5. 指标定义

### 5.1 四维度各自的准确率

对每个输入组合 × 每个模型 × 每个维度,计算:

- **Exact Match Accuracy**:模型输出完全等于 ground truth 的比例
- **Semantic Match Accuracy**(针对 action_intent,用 Embedding 余弦相似度 > 0.7 认定语义等价)
- **Confidence Calibration**:模型自评的 confidence 和实际准确率的相关性

### 5.2 聚合指标

- **加权总准确率**:4 个维度各占 25%(可按场景调整)
- **性价比指数**:准确率 / 平均 token 消耗
- **稳定性**:同一元素 3 次调用输出一致的比例

### 5.3 按类别细分

- 按**页面类别**(登录/电商/后台/搜索/表单):哪类场景 LLM 最准?
- 按**元素类型**(button/input/link/checkbox):哪种元素判断最难?
- 按**风险级别**:破坏性动作的识别率(这个指标单独看)

## 6. Go / No-Go 决策阈值

实验结束后,依据数据做出决策:

### 6.1 Go 条件(继续按 40 号文档推进阶段 1)

满足**任一**即可:
- 组合 A(纯 snapshot)在简单场景(登录/搜索)下准确率 **≥ 75%**
- 组合 C(+HTML)在全场景准确率 **≥ 70%** 且成本增长 < 2×
- 即使组合 B(+截图)准确率 **≥ 85%**,说明多模态确实有效——可把阶段 3 提前

### 6.2 No-Go 条件(重新设计)

- 所有组合在所有场景准确率都 **< 60%**
- Confidence Calibration 极差(模型说 0.9 但错一半)——说明 LLM 在这个任务上**不可信**,阶段 1 预处理不可行,必须跳到阶段 2 模式库或阶段 3 多模态

### 6.3 有条件 Go(调整方向)

- 简单场景可 Go,复杂场景 No-Go → **缩小阶段 1 范围**到高频封闭场景
- 某些维度(如 risk_level)好,某些维度(如 action_intent)差 → **只做好维度**,差的维度交给阶段 2/3
- Opus 好但 Haiku 差 → **阶段 1 预处理不能用 cheap 模型**,成本结构变化,需要重新算账

## 8. 实施安排

### 8.1 时间表(1 周)

| 日期 | 工作 | 产出 |
|---|---|---|
| D1 | 10 个样本页面采集(HTML/截图/snapshot)+ 评估脚本搭建 | `samples/` 齐全 |
| D2 | 两人各自独立标注 30 个元素(每人) | `labels/` 初版 |
| D3 | 标注冲突对齐 + 标注完成 | `labels/` 终版 |
| D4 | LLM 实验脚本 + 小样本跑通(3 个元素 × 3 组合 × 2 模型) | Sanity check |
| D5 | 全量实验运行 + 结果收集 | `results/` 齐全 |
| D6 | 数据分析 + 失败案例深入 | 初步报告 |
| D7 | 写最终报告 + Go/No-Go 决策 | `report.md` |

### 8.2 所需资源

- **人力**:2 人(1 天标注工作量,其他可兼顾)
- **成本**:< $50 LLM API 费用
- **工具**:Python/Go 脚本(纯流程,无复杂依赖)
- **数据存储**:`sdk/docs/experiments/phase0/` 目录(纳入 git)

### 8.3 风险

- **样本偏见**:10 页不够覆盖所有场景 → **接受**,这是"可行性探针"不是"完备评估"
- **标注主观**:两人可能有共同盲区 → **接受**,记录为已知偏差
- **LLM 版本变化**:Opus 升级后结果可能不同 → **每季度重跑一次**

## 9. 为什么这个实验必须做

### 9.1 不做实验的代价

如果直接进阶段 1 开发,投入 3-5 周,然后发现:
- LLM 的语义标注准确率只有 50% — 整个阶段 1 白做
- 或发现纯结构足够,不需要 LLM 预处理 — 省了一大笔 token 但已经烧了工时
- 或发现某些维度 LLM 根本做不对 — 需要改架构

任何一个都意味着 **3-5 周工时报废 + 一次重新设计**。

### 9.2 做实验的价值

1 周实验 + < $50 成本,能得到:
- 对 LLM 语用推断能力的**定量认识**(不再拍脑袋估计)
- 阶段 1 的**实际可行范围**(哪些维度做,哪些放弃)
- 输入组合的**性价比曲线**(指导成本控制)
- 失败案例库(阶段 2 模式库的早期训练数据)

**这个实验是整个 40 号架构性价比最高的一步**。

---

## 10. 总结

**实验做完才谈写代码**。40 号文档规划很漂亮,但所有阶段 1 的"批量预处理"都建立在一个**未验证的假设**上——"LLM 在当前水平下能从 snapshot/HTML/screenshot 推断 L6-L8 语用"。

如果假设成立:阶段 1 顺利推进,预处理能拿到高质量语义标注。
如果假设不成立:阶段 1 产生低质量噪声,Agent 更容易被误导,比不做还糟。

**所以阶段 0 不是"可选步骤",是 Go/No-Go 的前置条件**。

先做实验,再写代码。
